# Cycle 5 Handoff — Scheduling & Calendar

**Status:** Cycle 5 **COMPLETE — all 10 steps done.** Step 1 lands `015_sch_bell_schedules_and_rooms.sql` with 3 base tables (sch_bell_schedules, sch_periods, sch_rooms) for the foundation of the M22 Academic Scheduling module. Step 2 lands `016_sch_timetable_and_bookings.sql` with 3 base tables (sch_timetable_slots with two `EXCLUDE USING gist` constraints catching teacher and room double-booking against overlapping daterange windows, sch_room_bookings, sch_room_change_requests) plus the `btree_gist` extension. Step 3 lands `017_sch_calendar_and_coverage.sql` with 4 base tables (sch_calendar_events with a multi-column time-consistency CHECK keeping all_day in lockstep with start_time/end_time, sch_calendar_day_overrides with UNIQUE(school_id, override_date), sch_coverage_requests with a 3-branch assignment_chk and UNIQUE(timetable_slot_id, coverage_date), sch_substitution_timetable with UNIQUE(original_slot_id, effective_date) and ON DELETE CASCADE on coverage_request_id). Step 7 lands the first scheduling web surface — a `Schedule` launchpad tile gated on `sch-001:read`, 4 admin routes (`/schedule/bell-schedules` + `[id]` period editor, `/schedule/timetable` week-view grid, `/schedule/rooms`, `/schedule/room-bookings`), 15 React Query hooks in `use-scheduling.ts`, and a `scheduling-format.ts` helpers module. Step 8 lands the calendar / coverage UI — a second `Calendar` launchpad tile gated on `sch-003:read`, 3 routes (`/calendar` month-view with event-type colour coding + admin draft toggle + draft-or-publish create flow, `/schedule/coverage` day-of board grouped Open/Assigned/Cancelled with admin Assign + Cancel modals, `/schedule/coverage/history` filterable past-and-upcoming reporting), and 15 more React Query hooks (calendar events, day overrides, coverage, substitutions). Cycle 5 is the second cycle of Phase 3 (Expand). It connects Cycle 1's classes to a real timetable, Cycle 4's leave events to a substitute-coverage workflow, and ships a school calendar that every persona can read. Cycles 0–4 are COMPLETE; see `HANDOFF-CYCLE1.md`, `HANDOFF-CYCLE2.md`, `HANDOFF-CYCLE3.md`, and `HANDOFF-CYCLE4.md` for the foundation this cycle builds on.
**Branch:** `main`

## What this cycle adds on top of Cycles 0–4

### Out-of-scope decisions for Step 1

- **No `rotation_day` / `cycle_day` column on `sch_periods`.** The ERD includes columns for A/B day rotation schedules (ADR-053 advanced scheduling); the plan defers `sch_rotation_cycles` + `sch_rotation_calendar` to Cycle 5b. `day_of_week` is nullable so rotation-driven schedules can leave it blank, but the rotation linkage itself is not modelled this cycle.
- **`fac_space_id` on `sch_rooms` is DISPLAY-ONLY.** The future M52 Facilities module will own `fac_spaces`; Cycle 5 doesn't ship that table. Keeping a nullable UUID column on rooms now means later facilities work is purely additive — no DDL change to `sch_rooms` once `fac_spaces` exists.
- **No `is_default` enforcement that exactly-one bell schedule is default.** The partial UNIQUE INDEX rejects two simultaneous defaults, but a school with zero defaults is allowed (e.g. a freshly-provisioned tenant before the seed runs). Step 5's `BellScheduleService.setDefault` will own the "atomically flip default" pattern (clear all, then set the one).
- **No `EXCLUSION` constraints on `sch_periods` for time-range overlap.** The UNIQUE on `(bell_schedule_id, COALESCE(day_of_week,-1), start_time)` rejects two periods that start at the same minute, but two periods that overlap (e.g. 09:00–10:00 and 09:30–10:30) are allowed at the schema layer. The plan deliberately keeps periods overlap-free at the application layer (the bell schedule editor in Step 7 catches it) and reserves `EXCLUSION` constraints for `sch_timetable_slots` in Step 2 where the conflict semantics are dramatically more important.
- **`day_of_week` is `SMALLINT`, not `TEXT`.** The plan calls out 0=Monday..6=Sunday. Storing as integer is more compact and lets the period-listing query use a simple `ORDER BY day_of_week, start_time`. The COALESCE(-1) in the UNIQUE index handles the rotation case where `day_of_week IS NULL`.
- **`has_projector` and `has_av` are the only amenity flags.** The plan calls out these two; richer room metadata (smart board, accessibility, climate control) can ship as additive columns later without breaking anything. Schools that need richer metadata today can put it in `notes` (added in a future iteration) or model it through a separate `sch_room_amenities` join table.
- **No `building` / `floor` validation.** Both are free TEXT — schools' building taxonomies vary too much to constrain at the schema layer.
- **No partitioning.** Volume bounded by (schools × ~10 schedules × ~50 periods) and (schools × ~50 rooms). Far below the partitioning threshold even at multi-school scale.
- **No seed yet.** Step 4 owns the bell-schedule + rooms seed (8-period Standard Day + Early Dismissal variant + 10 rooms).
- **CHECK strings still cannot contain `;`.** Carries forward from every prior cycle. Spot-checked all CHECK predicates and `COMMENT ON COLUMN` strings in 015 — none contain `;`. The block-comment header was reviewed for the splitter trap.

### EXCLUSION constraints — btree_gist

The `btree_gist` extension is created at the top of the migration with `CREATE EXTENSION IF NOT EXISTS btree_gist`. The extension lives at database scope, so re-provisioning a tenant on the same database re-uses the existing install — idempotent. Two `EXCLUDE USING gist` constraints on `sch_timetable_slots`:

```sql
CONSTRAINT sch_timetable_slots_teacher_no_overlap EXCLUDE USING gist (
    teacher_id WITH =,
    period_id WITH =,
    daterange(effective_from, COALESCE(effective_to, 'infinity'::date), '[)') WITH &&
)
CONSTRAINT sch_timetable_slots_room_no_overlap EXCLUDE USING gist (
    room_id WITH =,
    period_id WITH =,
    daterange(effective_from, COALESCE(effective_to, 'infinity'::date), '[)') WITH &&
)
```

The half-open `[)` daterange means a slot ending on a date and another slot starting on the same date do **not** overlap — the natural "this contract closes 2027-08-31, the new contract starts 2027-09-01" pattern. Open-ended assignments coalesce `effective_to=NULL → 'infinity'::date` so they block every future date for the same teacher/room + period.

`teacher_id` is nullable, and PostgreSQL's `=` operator class treats NULL as not-equal-to-anything, so two TBD-teacher slots (both `teacher_id = NULL`) on the same period and overlapping date range do **not** conflict. This is the right behaviour: TBD-teacher is "to be determined later", not "not staffed". The room EXCLUSION still blocks two TBD-teacher slots in the same room — which is what we want.

Constraint violations raise SQLSTATE 23P01. Step 5's `TimetableService.createSlot` catches this and returns `409 Conflict` with a message naming the conflicting actor (teacher James Rivera / Room 101) — the schema, not the application, is the authoritative conflict gate.

### Out-of-scope decisions for Step 2

- **No DB-enforced uniqueness against `sch_room_bookings` and `sch_timetable_slots` together.** A confirmed booking that overlaps the timetable in time + room is allowed at the schema layer — the conflict semantics differ (one is class-period-bounded, the other is wall-clock-bounded). The Step 5 `RoomBookingService.checkConflict` does the cross-table check at the application layer before flipping `status='CONFIRMED'`. Pushing this down to a DB constraint would require materialising every (room, day-of-week, period start_at, period end_at) tuple from the active timetable into a queryable form, which is more complexity than the conflict matters.
- **No `effective_to NOT NULL` for the EXCLUSION** — the daterange COALESCE(NULL, 'infinity') is the load-bearing trick. Without it, two open-ended slots wouldn't compare correctly and the constraint would silently miss the most common case (no end date).
- **No EXCLUSION on `(class_id, period_id, daterange)`.** The plan calls for UNIQUE(class_id, period_id, effective_from) as the class-side dedup, not an EXCLUSION. A class can hold non-overlapping slots for the same period (e.g. teacher rotation) — the UNIQUE just rejects exact duplicates. The teacher and room EXCLUSIONs catch the realistic conflict cases; the class side is admin-controlled (one row per class per period at any given instant is an application invariant, not a schema one).
- **No cascade on any FK from `sch_timetable_slots`.** Deleting a class, period, room, or teacher should fail loudly when a slot still references them — Step 5's services own the orderly close-out (`effective_to = today` → archive). This matches the same "fail loudly" stance as Cycle 4's `hr_leave_balances.academic_year_id`.
- **`sch_room_bookings` does not reference `sch_timetable_slots`.** Bookings are timetable-independent in the schema; the conflict check is application-side. Modelling a soft FK from a booking to a slot would imply an orderly relationship that doesn't exist (a booking can land on a holiday when no slot is active).
- **`sch_room_change_requests.requested_room_id` is NULLABLE.** NULL means "any available room — let the admin pick at approval". The application-side approval flow inspects the timetable + bookings to recommend a room before flipping status to APPROVED.
- **No `sch_room_bookings.is_recurring` column.** Recurrence is reserved for a future iteration alongside the calendar event recurrence model. For Cycle 5, every booking is a single instance; recurring use cases are surfaced through the school calendar (Step 3's `sch_calendar_events`).
- **`AUTO_APPROVED` is a status value but no auto-approval logic ships in Cycle 5.** The plan reserves it for a future "if the requested room is empty during the requested period, auto-approve" sweep. The Step 5 service exposes manual APPROVED/REJECTED only; AUTO_APPROVED is forward-compat.
- **No partitioning.** Volume bounded by (schools × classes × periods × academic-year history) — well below partitioning threshold.
- **No seed yet.** Step 4 owns the timetable seed (Rivera's 6 classes slotted into Periods 1–6 M-F).
- **Splitter `;`-in-string trap not tripped this time.** Spot-checked all CHECK predicates, default expressions, and COMMENT strings in 016 — none contain `;`. The only `;` characters in the file are statement terminators (every line that ends with `;`).

### Out-of-scope decisions for Step 3

- **No EXCLUSION constraint on substitute double-booking.** A substitute could in theory be assigned to two overlapping slots on the same date. Modelling this as an EXCLUSION would require joining through `sch_periods` to materialise time ranges (substitution rows carry only `effective_date`, not start/end times), which is messier than the conflict deserves. The Step 6 `SubstitutionService.assign` does the cross-table check at the application layer before INSERTing.
- **`leave_request_id` is soft (intra-tenant).** The plan calls it out as soft; the live CoverageConsumer carries the leave id straight off the `hr.leave.coverage_needed` envelope payload, not from a synchronous DB lookup. Keeping the FK soft lets the seed in Step 4 plant a representative coverage row even if the matching leave request is being seeded out of order.
- **CANCELLED is unconstrained in the assignment CHECK.** A coverage can be cancelled before assignment (substitute NULL) — e.g. the leave is rescinded — or after assignment (substitute populated for audit) — e.g. the sub falls ill. Both shapes are legitimate, so the CHECK accepts either.
- **`COVERED` is a terminal post-event status, not a separate transition path.** The plan reserves it for a future "the day passed and the substitution actually happened" sweep. The Step 6 service exposes OPEN/ASSIGNED/CANCELLED transitions only; COVERED is forward-compat. The `assignment_chk` accepts COVERED with the same NOT NULL constraints as ASSIGNED.
- **No partitioning.** Volume bounded by (schools × ~200 events/year + ~40 day overrides/year + ~50 coverage requests/year + ~50 substitution rows/year) — well below partitioning threshold.
- **No seed yet.** Step 4 owns the calendar + coverage seed (5 events, 1 day override, 1 coverage request linked to Rivera's seeded sick leave with VP Park as substitute, 1 substitution row).
- **Splitter `;`-in-string trap not tripped this time.** Spot-checked all CHECK predicates, default expressions, and COMMENT strings in 017 — none contain `;`. The single `--` mid-sentence in the block-comment header sits safely inside `/* */` (the splitter only filters statement-leading `--` after trim).

### Idempotency design

The seed gates on `SELECT count(*) FROM sch_bell_schedules` — non-zero short-circuits the whole script with an "already populated — skipping" log. ### Conflict translation — the load-bearing trick

`TimetableService.translateConflict(e, periodId, teacherId, roomId)` is the function that lifts the schema-layer EXCLUSION constraint into a meaningful HTTP error. When `INSERT` / `UPDATE` raises:

- **SQLSTATE 23P01 with `sch_timetable_slots_teacher_no_overlap`** → look up the teacher's name (`hr_employees → iam_person`) and the period's name → `409 Conflict — "Teacher {firstName lastName} is already scheduled for {Period N} during the requested date range"`.
- **SQLSTATE 23P01 with `sch_timetable_slots_room_no_overlap`** → look up the room's name → `409 Conflict — "Room {name} is already scheduled for {Period N} during the requested date range"`.
- **SQLSTATE 23503** (FK violation — bogus class_id / period_id / teacher_id / room_id) → `400 Bad Request — "One of class_id / period_id / teacher_id / room_id does not exist in this tenant"`.
- **SQLSTATE 23505** with `sch_timetable_slots_class_period_from_uq` → `409 Conflict — "A timetable slot already exists for this class + period + start date"`.
- **Anything else** → re-throw the original error.

The look-up reads run on the failure path only — happy-path inserts pay no extra round trip.

### Row-level authorisation pattern

All five services follow the row-level auth pattern from REVIEW-CYCLE1 and Cycle 4:

- Reads use `actor.isSchoolAdmin` for admin-bypass and `actor.employeeId` for own-row scope.
- Writes that mutate per-employee state (`POST /room-bookings`, `POST /room-change-requests`) require `actor.employeeId` to be non-null — non-staff personas (parent, student, the synthetic Platform Admin without an `hr_employees` row) are forbidden at the service layer.
- Admin-only writes (`POST /bell-schedules`, `POST /timetable/slots`, etc.) check `actor.isSchoolAdmin` directly. The endpoint permission gate (`@RequirePermission`) handles the persona-level check; the service-layer `actor.isSchoolAdmin` handles the tenant-scope-chain admin check (REVIEW-CYCLE1 fix).
- `RoomChangeRequestService.list` filters non-admin reads to `requested_by = actor.employeeId` server-side so a teacher can only see their own change requests.

### School ID resolution

Every service uses `getCurrentTenant().schoolId` (populated by `TenantResolverMiddleware`) for the school-scoped fields. This is more direct than the `school_config` lookup HrModule uses and works the same way. The `school_config` fallback is only needed when seeding before any tenant identity exists; runtime always has the school id in context.

### Kafka emit

`TimetableService.create` / `update` / `delete` each fire `sch.timetable.updated` via `KafkaProducerService.emit({sourceModule:'scheduling', topic:'sch.timetable.updated', key: slotId, payload:{slotId, action: 'created'|'updated'|'deleted'}})`. The producer wraps the payload in the ADR-057 envelope, env-prefixes the topic to `dev.sch.timetable.updated`, and sets the legacy transport headers. There is no consumer for this topic in Cycle 5 Step 5 — it's published for future cycles (most likely the calendar / coverage path in Step 6 for cache invalidation).

### CoverageConsumer (1 Kafka consumer)

Subscribes to `prefixedTopic('hr.leave.coverage_needed')` (i.e. `dev.hr.leave.coverage_needed` in dev) under group `coverage-consumer`. Reuses the `unwrapEnvelope` + `processWithIdempotency` helpers from `apps/api/src/notifications/consumers/notification-consumer-base.ts` — same pattern as Cycle 3's notification consumers (claim-after-success, REVIEW-CYCLE2 BLOCKING 2).

Per inbound event:

1. Read `requestId`, `employeeId`, `startDate`, `endDate`, `affectedClasses[]` from the envelope payload.
2. Drop the message if any required field is missing or `affectedClasses` is empty.
3. `processWithIdempotency` does a read-only `IdempotencyService.isClaimed` check against `platform.platform_event_consumer_idempotency` keyed on `(consumer_group, event_id)`. The deterministic event_id from REVIEW-CYCLE4 MAJOR 3 means a Kafka redelivery of the same `hr.leave.approved` republishes with the same id — already-claimed → drop.
4. Inside `runWithTenantContextAsync`, enumerate every weekday (Mon–Fri) date in `[startDate, endDate]` via `enumerateWeekdayDates(startDate, endDate)`.
5. Inside `executeInTenantTransaction`, load active `sch_timetable_slots` for the supplied class ids whose date range covers `[startDate, endDate]`, joined through `sch_periods` to read `day_of_week`.
6. For each `(slot, date)` pair where the slot's period `day_of_week` is `NULL` or matches the date's ISO weekday (`isoWeekdayIndex`), `INSERT` one `sch_coverage_requests` row with status=OPEN. Each INSERT is wrapped in a try/catch — if SQLSTATE 23505 fires on `sch_coverage_requests_slot_date_uq`, swallow it (already created on a previous run; the schema-side UNIQUE is the belt-and-braces dedup).
7. After the tx commits, if any new rows landed, emit `sch.coverage.needed` once with `{leaveRequestId, employeeId, startDate, endDate, createdCoverageIds[]}` for the Step 8 admin UI.
8. `processWithIdempotency` calls `IdempotencyService.claim(consumerGroup, eventId, topic)` only after the `process` callback returns successfully. A transient DB failure mid-run leaves the event-id unclaimed → next Kafka redelivery rebuilds the work; the schema UNIQUE still catches duplicate rows.

### CalendarService.resolveDay — the resolution algorithm

The `GET /calendar/day/:date` endpoint is what the Step 8 calendar UI hits when rendering a single day. The resolution order matches the schema's COMMENT documentation on `sch_calendar_day_overrides.is_school_day`:

1. **Override wins outright.** `sch_calendar_day_overrides` for `(school_id, override_date)`. If `is_school_day=false`, the day is closed (snow day, emergency closure) — `resolvedFrom='OVERRIDE'`, `isSchoolDay=false`, `bellScheduleId=null`. If `is_school_day=true`, the override's `bell_schedule_id` is the effective schedule.
2. **First published event with `bell_schedule_id`.** If no override, look at `sch_calendar_events` rows whose date range covers the date AND that are `is_published=true` AND that carry a non-null `bell_schedule_id`. The first such row's schedule wins — `resolvedFrom='EVENT'`. (Events without `bell_schedule_id` describe the _contents_ of an otherwise normal day — `affects_attendance=true` flips off attendance pre-population — but they don't override the schedule.)
3. **School default.** If neither, fall back to the school's `is_default=true` `sch_bell_schedule` — `resolvedFrom='DEFAULT'`.
4. **None.** No default schedule exists for this school — `resolvedFrom='NONE'`, `bellScheduleId=null`. Treated as `isSchoolDay=true` (no schedule doesn't mean "closed", it means "not configured").

The endpoint also returns `eventIds[]` — every published event overlapping the date — so the calendar UI can render them as additional context (e.g. "Spring Break HOLIDAY" pill on a day where the schedule is still the default).

### Out-of-scope decisions for Step 6

- **No `sch.coverage.assigned` notification consumer yet.** The emit shape is ready (full join shape inline so a downstream consumer doesn't need DB lookups), but the actual fan-out to the substitute via `NotificationQueueService` lands in a future patch — most likely Step 8 alongside the coverage board UI. For Step 6 the emit is fire-and-forget; `NotificationDeliveryWorker` is the substrate that's ready to be wired up.
- **Day overrides do not currently override an active calendar event.** The resolution endpoint reads override first which means an override always wins; that's the documented behaviour. There's no "merge override + event" mode — that would need a richer composition rule that the plan doesn't call for.
- **`COVERED` status is not exposed via any write path.** The schema accepts COVERED with the same NOT NULL `assigned_substitute_id` + `assigned_at` constraints as ASSIGNED, but Step 6 only ships OPEN / ASSIGNED / CANCELLED transitions. A future "post-event sweep" worker could flip ASSIGNED → COVERED after the date passes.
- **No EXCLUSION on substitute double-booking.** A substitute could in theory be assigned to two overlapping `sch_substitution_timetable` rows on the same date in different periods. Modelling this as an EXCLUSION would require materialising period clock-times on the substitution row, which is out of scope. Step 8's coverage board UI surfaces a warning when an admin tries to assign a substitute who's already covering another period that day.
- **Seed coverage doesn't trigger the consumer.** The seed in Step 4 plants 1 ASSIGNED row directly via SQL; it doesn't go through the consumer. That's intentional — `seed-scheduling.ts` runs once per tenant before the API is up, so the consumer can't fire. The end-to-end test exercises the consumer chain via a _new_ leave submitted at runtime.
- **No `is_school_day` reads on the attendance pipeline yet.** Cycle 1's attendance pre-population doesn't query `sch_calendar_day_overrides` to skip closure days. The schema is ready (`affects_attendance` flag on events; `is_school_day` flag on overrides) but the integration lands in a future cross-cycle hardening sweep, not Step 6.
- **`@Public` on `/calendar/day/:date`?** No — the resolution endpoint stays gated on `sch-003:read` because schedule data is school-confidential. The student / parent persona reads through the same endpoint as admins; only the gate level differs.
- **Splitter / SQL trap considerations.** None — Step 6 ships no SQL migrations; everything is parameterised raw SQL inside services.

### Backend addition

`apps/api/src/scheduling/timetable.service.ts` gains `listForStudent(studentId, actor)`:

- Permission gate at the controller is `stu-001:read` — every relevant persona (teacher, parent, student, staff, admin) holds it from Cycle 1's seed.
- Row scope is enforced at the service layer via `StudentService.assertCanViewStudent(studentId, actor)`, the same helper Cycle 1 uses for `/students/:id/...` reads. Decision matrix:
  - Admin → all students.
  - Parent → only students linked via `sis_student_guardians` to a `sis_guardians` row whose `person_id` matches the parent's `actor.personId`.
  - Student → only the row whose `platform_students.person_id` matches `actor.personId`.
  - Teacher → only students with an ACTIVE `sis_enrollments` row in a class the teacher is on (`sis_class_teachers.teacher_employee_id = actor.employeeId`).
  - Anyone else → 404.
- Query: `SELECT_SLOT_BASE WHERE s.class_id IN (SELECT class_id FROM sis_enrollments WHERE student_id = $1::uuid AND status = 'ACTIVE') ORDER BY p.day_of_week NULLS FIRST, p.start_time, c.section_code`. Reuses the existing `SELECT_SLOT_BASE` join graph (sis_classes / sis_courses / sch_periods / sch_rooms / hr_employees / iam_person) so the response shape is identical to the other timetable endpoints.

`SchedulingModule` now imports `SisModule` to get the `StudentService` provider.

`apps/api/src/scheduling/timetable.controller.ts` adds:

```
@Get('student/:studentId')
@RequirePermission('stu-001:read')
async forStudent(...) { return this.timetable.listForStudent(studentId, actor); }
```
