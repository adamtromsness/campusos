Cycle 7: Tasks & Approval Workflows
Implementation Plan

Cross-cutting infrastructure that every module uses. A unified task system with auto-task rules driven by Kafka events, plus a configurable multi-step approval engine. Generalizes the hardcoded approval patterns from leave (Cycle 4), room changes (Cycle 5), and child-link requests (Cycle 6.1).
May 2026 · 10 Steps · ERD v11 · M1 Task Management (6 tables) + M2 Approval Workflows (7 tables)

Plan Overview

Cycle 7 spans two tightly-coupled ERD modules: M1 (Task Management) and M2 (Approval Workflows). Together they provide the "to-do list" and "submit → approve → execute" patterns that every other module consumes. The Task Worker (ADR-011) is the sole writer to tsk_tasks — domain modules emit Kafka events and auto-task rules translate them into actionable tasks. The Workflow Engine (ADR-012) is the sole writer to wsk_approval_requests and wsk_approval_steps — source modules submit via REST and listen on approval.request.resolved.

This cycle is architecturally significant because it retroactively connects Cycles 1–6. After it ships: a teacher publishing an assignment auto-creates a "Complete: Chapter 5 Homework" task on every enrolled student's to-do list. A leave request flows through a configurable multi-step approval chain. An absence request submitted by a parent routes to the right admin based on workflow templates. The existing hardcoded approve/reject patterns in LeaveService, RoomChangeRequestService, and ChildLinkRequestService become consumers of the generalized engine.

Vertical Slice Deliverable

Teacher publishes an assignment → Task Worker consumes cls.assignment.posted → auto-task rule creates a TODO task on every enrolled student's to-do list → student opens their "Tasks" app and sees "Complete: Chapter 5 Homework — due Fri 3pm" → student marks it DONE → task moves to completed. Separately: teacher submits a leave request → WorkflowEngineService creates a multi-step approval request (department head → principal) → department head approves step 1 → step 2 activates for principal → principal approves → approval.request.resolved fires → LeaveService consumes it and approves the leave → coverage_needed flows as before.

Auto-task rules for ASSIGNMENT_POSTED, LEAVE_APPROVED, ABSENCE_REQUEST_SUBMITTED, GRADE_POSTED. Student/parent "Tasks" app tile. Acknowledgement workflow. Functions SYS-001 (core).

Parallel approval steps (is_parallel=true — schema supports it, engine deferred). Escalation timeout worker (schema + escalation table ready, cron deferred). Step-by-Step Build Plan

01 Task Schema — Tasks, Archive, Auto-Rules
Schema

The M1 Task Management tables. tsk_tasks is RANGE-partitioned monthly (matching the msg_messages pattern from Cycle 3). The Task Worker (ADR-011) is the sole writer — no domain module writes tasks directly.

Tables (6)
  1. tsk_tasks — RANGE(created_at) monthly, 24-month window (2025-08 → 2027-08). Composite PK (id, created_at). school_id UUID, owner_id UUID FK(platform_users), title TEXT NOT NULL, description TEXT, source ENUM(MANUAL, AUTO, SYSTEM), source_ref_id UUID, priority ENUM(LOW, NORMAL, HIGH, URGENT), status ENUM(TODO, IN_PROGRESS, DONE, CANCELLED), due_at TIMESTAMPTZ, task_category ENUM(ACADEMIC, PERSONAL, ADMINISTRATIVE, ACKNOWLEDGEMENT), acknowledgement_id UUID FK(tsk_acknowledgements) nullable, created_for_id UUID FK(platform_users) nullable. UNIQUE(owner_id, source, source_ref_id) WHERE source != 'MANUAL' — auto-task deduplication. INDEX(owner_id, status, due_at) on active partitions.
  2. tsk_tasks_archive — RANGE(created_at) annual partition. Same schema as tsk_tasks. No FK back. Background archiver moves DONE/CANCELLED rows older than 30 days. INDEX(owner_id, completed_at DESC). Deferred: the archiver job itself; schema-only this cycle.
  3. tsk_auto_task_rules — school_id UUID, trigger_event_type TEXT NOT NULL (maps to Kafka topic via the Task Worker), target_role TEXT, title_template TEXT (supports {student_name}, {assignment_title}, {class_name} placeholders), description_template TEXT, priority ENUM, due_offset_hours INT, task_category ENUM, is_active BOOLEAN, is_system BOOLEAN DEFAULT true. UNIQUE(school_id, trigger_event_type) for system rules. Schools can deactivate but not delete system rules.
  4. tsk_auto_task_conditions — rule_id FK, field_path TEXT, operator ENUM(EQUALS, NOT_EQUALS, IN, NOT_IN, GT, LT, EXISTS), value JSONB. Optional conditions on the event payload that must all be true for the rule to fire. Example: only create a task for ASSIGNMENT_POSTED when payload.isPublished = true.
  5. tsk_auto_task_actions — rule_id FK, action_type ENUM(CREATE_TASK, CREATE_ACKNOWLEDGEMENT, SEND_NOTIFICATION), action_config JSONB. Most rules have one CREATE_TASK action; acknowledgement rules have CREATE_TASK + CREATE_ACKNOWLEDGEMENT. Extensible for future actions.
  6. tsk_acknowledgements — school_id UUID, subject_id UUID FK(platform_users) (the person who must acknowledge), source_type ENUM(ANNOUNCEMENT, DISCIPLINE_RECORD, POLICY_DOCUMENT, SIGNED_FORM, CONSENT_REQUEST, CUSTOM), source_ref_id UUID, source_table TEXT, title TEXT NOT NULL, body_s3_key TEXT nullable, requires_dispute_option BOOLEAN DEFAULT false, status ENUM(PENDING, ACKNOWLEDGED, ACKNOWLEDGED_WITH_DISPUTE, EXPIRED), acknowledged_at TIMESTAMPTZ, dispute_reason TEXT, created_by UUID FK(platform_users), expires_at TIMESTAMPTZ. INDEX(subject_id, status). INDEX(source_type, source_ref_id). On INSERT: Task Worker creates a tsk_tasks row with task_category=ACKNOWLEDGEMENT.
Produces → 6 new tables + 24 monthly partitions on tsk_tasks + annual partitions on tsk_tasks_archive. Migration: 024_tsk_tasks_and_auto_rules.sql.
02 Workflow Schema — Templates, Requests, Steps
Schema

The M2 Approval Workflows tables. WorkflowEngineService (ADR-012) is the sole writer to wsk_approval_requests and wsk_approval_steps. Source modules submit via the REST API and listen on approval.request.resolved.

Tables (6)
  1. wsk_workflow_templates — school_id UUID, name TEXT NOT NULL, request_type TEXT NOT NULL (e.g. 'LEAVE_REQUEST', 'ABSENCE_REQUEST', 'CHILD_LINK_REQUEST', 'ROOM_CHANGE_REQUEST', 'CUSTOM'), description TEXT, is_active BOOLEAN. UNIQUE(school_id, request_type). Each school can have one active template per request type.
  2. wsk_workflow_steps — template_id FK, step_order INT, approver_type ENUM(SPECIFIC_USER, ROLE, MANAGER, DEPARTMENT_HEAD), approver_ref TEXT (user ID for SPECIFIC_USER, role name for ROLE, null for MANAGER/DEPARTMENT_HEAD — resolved dynamically), is_parallel BOOLEAN DEFAULT false (deferred — always sequential this cycle), timeout_hours INT nullable, escalation_target_id UUID FK(platform_users) nullable. INDEX(template_id, step_order).
  3. wsk_approval_requests — school_id UUID, template_id UUID FK, requester_id UUID FK(platform_users), request_type TEXT, reference_id UUID (the domain entity being approved — e.g. hr_leave_requests.id), reference_table TEXT, status ENUM(PENDING, APPROVED, REJECTED, CANCELLED, WITHDRAWN), submitted_at TIMESTAMPTZ. INDEX(school_id, status, request_type) WHERE status NOT IN ('APPROVED','REJECTED','CANCELLED','WITHDRAWN'). INDEX(requester_id, created_at DESC).
  4. wsk_approval_steps — request_id FK, step_order INT, approver_id UUID FK(platform_users), status ENUM(AWAITING, APPROVED, REJECTED, SKIPPED), actioned_at TIMESTAMPTZ, comments TEXT. INDEX(approver_id, status) WHERE status='AWAITING' — approver's pending queue.
  5. wsk_approval_comments — request_id FK, author_id UUID FK, body TEXT NOT NULL, is_requester_visible BOOLEAN DEFAULT true, created_at TIMESTAMPTZ. INDEX(request_id, created_at).
  6. wsk_workflow_escalations — IMMUTABLE. request_id FK, step_id FK, original_approver_id UUID FK, escalated_to_id UUID FK, escalation_reason TEXT DEFAULT 'Approval step timed out', hours_overdue NUMERIC(5,1), escalated_at TIMESTAMPTZ, resolved_at TIMESTAMPTZ, resolved_by UUID FK. INDEX(escalated_to_id, resolved_at) WHERE resolved_at IS NULL. Schema-only this cycle; escalation timeout worker deferred.
Produces → 6 new tables. Migration: 025_wsk_approval_workflows.sql. Total Cycle 7 tables: 12.

04 Task Worker — Kafka Consumer + Auto-Task Engine
Code

The Task Worker (ADR-011) is the sole writer to tsk_tasks. It subscribes to every Kafka topic that has an active auto-task rule and translates events into tasks. No domain module writes tasks directly.

Architecture
  1. TaskWorker Kafka consumer — on startup, queries tsk_auto_task_rules across all active tenants and subscribes to the union of trigger_event_type topics (env-prefixed via prefixedTopic). Consumer group: task-worker. Uses the standard unwrapEnvelope + processWithIdempotency pattern.
  2. Rule evaluation — for each inbound event, the worker resolves the tenant, queries active rules matching trigger_event_type, evaluates tsk_auto_task_conditions against the event payload (all conditions must match), then for each passing rule executes the tsk_auto_task_actions.
  3. Task creation — CREATE_TASK action resolves the target owner(s): for ASSIGNMENT_POSTED that's every enrolled student in the class; for LEAVE_APPROVED that's school admins; for ABSENCE_REQUEST_SUBMITTED that's school admins. Resolves template placeholders ({student_name}, {assignment_title}, etc.) from the event payload. INSERT into tsk_tasks with source='AUTO' and source_ref_id=event.payload.referenceId. Deduplication: UNIQUE(owner_id, source, source_ref_id) + Redis SET NX on tsk:auto:{owner_id}:{source_ref_id}.
  4. Acknowledgement creation — CREATE_ACKNOWLEDGEMENT action inserts a tsk_acknowledgements row, then the companion CREATE_TASK action creates the task with task_category=ACKNOWLEDGEMENT and acknowledgement_id linking back.
  5. Kafka events emitted: task.created on every task INSERT. task.completed when status transitions to DONE. student.acknowledgement.completed when an acknowledgement is ACKNOWLEDGED or ACKNOWLEDGED_WITH_DISPUTE.
Produces → TaskWorker Kafka consumer (sole writer to tsk_tasks). Subscribes to all active auto-task trigger topics. Emits task.created, task.completed, student.acknowledgement.completed.
05 Task NestJS Module — CRUD + Acknowledgements
Code

The request-path API for tasks. The Task Worker creates auto-tasks; this module lets users create manual tasks, manage status, and complete acknowledgements.

Services & Endpoints
  1. TaskService — GET /tasks (own tasks, filterable by status/category/priority/due date), GET /tasks/:id, POST /tasks (manual task creation — source=MANUAL), PATCH /tasks/:id (status transitions: TODO → IN_PROGRESS → DONE, or CANCELLED at any point), GET /tasks/assigned (tasks created_for the caller by others). All @RequirePermission('sys-001:read') or 'sys-001:write'.
  2. AcknowledgementService — GET /acknowledgements (own pending acknowledgements), POST /acknowledgements/:id/acknowledge (marks ACKNOWLEDGED, transitions the linked task to DONE, emits student.acknowledgement.completed), POST /acknowledgements/:id/dispute (requires dispute_reason, marks ACKNOWLEDGED_WITH_DISPUTE). @RequirePermission('sys-001:write').
  3. Row-level auth: users see only their own tasks (owner_id or created_for_id = actor.accountId). Admins see all tasks in the tenant. Teachers can see tasks they created for students (created_for_id set, owner_id = teacher's accountId).
Produces → TaskModule with TaskService, AcknowledgementService. ~8 endpoints.
06 Workflow Engine — Multi-Step Approval
Code

The WorkflowEngineService (ADR-012) — the sole writer to wsk_approval_requests and wsk_approval_steps. Source modules submit approval requests via REST; the engine handles step sequencing, approver resolution, and resolution notification.

Services & Endpoints
  1. WorkflowEngineService — submit(requestType, referenceId, referenceTable, requesterId): looks up the active wsk_workflow_templates for the request type, creates a wsk_approval_requests row with status=PENDING, resolves the Step 1 approver (by approver_type: SPECIFIC_USER = direct, ROLE = query IAM for role holders, MANAGER = requester's supervisor from hr_employees, DEPARTMENT_HEAD = department head from sis_departments/hr_positions), creates the first wsk_approval_steps row with status=AWAITING. Emits approval.step.awaiting (notification consumer creates a task for the approver).
  2. advanceStep(requestId, stepId, decision, comments, actorId) — SELECT FOR UPDATE on the step row. Validates step.status=AWAITING and step.approver_id=actorId (or admin override). On APPROVED: advances to next step or resolves the request. On REJECTED: resolves the request as REJECTED, skips remaining steps. Emits approval.request.resolved with the final status + reference_id so the source module can act.
  3. Approver resolution — resolves dynamically per step: ROLE queries iam_effective_access_cache for accounts holding the role in the school's scope chain. MANAGER queries hr_employees for the requester's supervisor (deferred — falls back to school admin if no supervisor set). DEPARTMENT_HEAD queries the employee's department head (deferred — falls back to school admin).
  4. API endpoints: POST /approvals (source module submits — or internal service call), GET /approvals (admin: all pending, requester: own), GET /approvals/:id (with step history + comments), POST /approvals/:id/steps/:stepId/approve, POST /approvals/:id/steps/:stepId/reject, POST /approvals/:id/comments, POST /approvals/:id/withdraw (requester cancels). All @RequirePermission('sys-001:read') or 'sys-001:write'.
Resolution Contract

When all steps are APPROVED, the request status flips to APPROVED and a Kafka event approval.request.resolved fires with payload { requestType, referenceId, referenceTable, status: 'APPROVED', requesterId }. On any step REJECTED, the request immediately resolves as REJECTED. Source modules consume this event to execute the approved action (e.g. LeaveService approves the leave, ChildLinkRequestService activates the link).

Produces → WorkflowEngineService with approval submission, step advancement, resolution. ~7 endpoints + 2 Kafka events (approval.step.awaiting, approval.request.resolved).
07 Leave Approval Migration to Workflow Engine
Code

Migrate the Cycle 4 leave approval from its hardcoded approve/reject pattern to the generalized workflow engine. This is the proof that the engine works for real domain workflows.

Changes
  1. LeaveService.submit() — after creating the hr_leave_requests row (status=PENDING, pending balance incremented), calls WorkflowEngineService.submit('LEAVE_REQUEST', leaveRequestId, 'hr_leave_requests', actor.accountId). The workflow template defines the approval chain (department head → principal per the seed).
  2. LeaveApprovalConsumer — new Kafka consumer on approval.request.resolved, filtered by requestType='LEAVE_REQUEST'. On APPROVED: calls the existing LeaveService.approveInternal(referenceId) which does the balance mutation + status flip + coverage_needed emit. On REJECTED: calls LeaveService.rejectInternal(referenceId).
  3. Backward compatibility: the existing PATCH /leave-requests/:id/approve and /reject endpoints stay in place for admin override (bypassing the workflow). They're marked as "direct admin action" in the audit trail. Schools without a LEAVE_REQUEST workflow template fall back to the direct pattern.
  4. Remove hardcoded routing: the previous LeaveNotificationConsumer that notified the requester on approve/reject is replaced by the workflow engine's approval.request.resolved notification (the engine emits a notification to the requester automatically).
Produces → Leave approval flows through the workflow engine. Multi-step approval chain configurable per school. Direct admin override preserved.

10 Vertical Slice Integration Test
Test

End-to-end proof that the task and approval engines work across the existing modules.

Test Script
  1. Auto-task from assignment: Teacher publishes an assignment in Period 1 Algebra. Verify cls.assignment.posted (or equivalent) event fires. Verify TaskWorker creates a TODO task on Maya's task list with title matching the template. Verify task.created event fires.
  2. Student completes task: Maya opens Tasks, sees the assignment task. Marks it DONE. Verify status transitions correctly. Verify task.completed event fires.
  3. Manual task creation: Teacher creates a task for Maya: "Prepare for oral presentation". Verify it appears in Maya's task list with created_for_id set.
  4. Leave through workflow engine: Rivera submits a leave request. Verify WorkflowEngineService creates a wsk_approval_requests row with 2 steps. Verify Step 1 approver (department head — resolved dynamically) receives an AWAITING step + notification.
  5. Step 1 approval: Department head approves Step 1. Verify step status → APPROVED. Verify Step 2 activates for the principal.
  6. Step 2 approval: Principal approves Step 2. Verify request status → APPROVED. Verify approval.request.resolved fires. Verify LeaveApprovalConsumer processes it. Verify leave request status → APPROVED. Verify balance updates. Verify coverage_needed fires.
  7. Rejection test: Submit another leave request. Step 1 approver rejects. Verify request immediately REJECTED (Step 2 is SKIPPED). Verify leave request stays PENDING (pending balance not consumed).
  8. Withdrawal: Submit a leave request. Requester withdraws before any step is actioned. Verify request WITHDRAWN. Verify leave request reverts to CANCELLED.
  9. Acknowledgement: Admin creates an acknowledgement for Maya (source_type=POLICY_DOCUMENT). Verify TaskWorker creates an ACKNOWLEDGEMENT task. Maya acknowledges. Verify student.acknowledgement.completed event fires.
  10. Permission denials: Student cannot approve workflow steps. Parent cannot see other users' tasks. Teacher cannot configure workflows.
Produces → Verified task + approval workflow vertical slice. Auto-task creation from existing events. Multi-step approval with leave integration.
