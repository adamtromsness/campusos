Development & Deployment Plan

The complete build pipeline: Claude develops, CI/CD executes, ChatGPT reviews, Human accepts
March 2026 · ERD v10 · 4 Environments · 29 Sections · Peer-Reviewed

About This Document

This document defines the complete development and deployment pipeline for CampusOS. Claude performs all development and unit testing. ChatGPT performs code review and architectural validation. GitHub Actions CI/CD handles execution and testing. A human performs customer acceptance testing. Production deployment follows zero-downtime procedures.

Four environments: DEV, SIT, CAT, PROD. Code flows through Git branches: dev → sit → cat → main. CAT always mirrors PROD except during active testing.

Technology stack: TypeScript full-stack with NestJS (backend), Next.js (frontend), Prisma (ORM), PostgreSQL, Kafka, Redis, deployed on AWS with infrastructure provisioned via Terraform.

29 sections covering: pipeline design, technology stack, environments, Git strategy, development workflow, review process, acceptance testing, deployment, database strategy, infrastructure, handoff format, rollback, Wave 1 build sequence, security, monitoring, CI/CD execution, contract testing, tenant isolation, rate limiting, feature flags, disaster recovery, E2E testing, Kafka scaling, Prisma scaling, IAM cache, observability, cost governance, parallel development, and API gateway.

1. Development Pipeline Overview

The CampusOS development pipeline uses a two-AI, one-human quality gate model. Each stage has a specific owner, a specific purpose, and specific exit criteria. Nothing advances without its gate passing.

Pipeline Stages

  • DEV — Claude develops and unit tests. Claude writes all application code, database migrations, unit tests, and API implementations. Code is committed to the dev branch. Unit tests must pass with >90% coverage before promotion. Claude produces a structured handoff package for each development cycle.
  • SIT — CI/CD executes, ChatGPT reviews. GitHub Actions CI automatically builds, migrates, and runs the full test suite (unit + integration + contract tests) in ephemeral Docker environments. ChatGPT receives the handoff package and CI results, performs static code review against the frozen architecture documents, validates architectural compliance, reviews test design, and analyses security. ChatGPT does NOT execute code — CI/CD does.
  • CAT — Human acceptance tests. Claude generates step-by-step test scripts tied to the user journey documents. A human follows the scripts in the CAT environment, verifying that every workflow works as described in the journeys. The human signs off or rejects with specific failure notes. CAT always mirrors PROD except during active testing.
  • PROD — Zero-downtime deployment. After CAT sign-off, code promotes to production via the main branch. Database migrations run first, then the application deploys with rolling updates. Feature flags control visibility of new functionality.

Quality Gate Summary

  • DEV → SIT gate: All unit tests pass. Coverage >90%. Migrations apply cleanly. Handoff package complete. Claude self-certifies against the ERD.
  • SIT → CAT gate: CI pipeline green (all unit, integration, and contract tests pass). ChatGPT code review passes. No architectural violations found. No security concerns flagged. ChatGPT produces written approval.
  • CAT → PROD gate: All manual test scripts pass. Human reviewer signs off. No open defects at P1 or P2 severity. CAT environment verified stable for 24 hours after final test.

Key Principle

This is not a sprint-based cadence. Development cycles complete when they are complete — Claude finishes a module or feature set, packages it, and hands it off. The handoff to ChatGPT happens when the work is ready, not on a fixed schedule. The human CAT review happens when SIT passes, not on a calendar.

2. Technology Stack

The technology stack is selected for three criteria: development velocity (Claude must be highly productive), architectural fit (the modular monolith with 36 modules must be cleanly expressible), and operational maturity (production-grade tooling for a K-12 SaaS platform).

Backend: NestJS (TypeScript)

NestJS is the backend framework. It provides native module boundaries that map directly to the 36 ERD modules — each CampusOS module (M20 SIS, M21 Classroom, etc.) becomes a NestJS module with its own controllers, services, and repository layer. The dependency injection system enforces module isolation at the code level.

  • Language: TypeScript 5.x. Strict mode. Full type safety across the stack.
  • Runtime: Node.js 20 LTS.
  • Framework: NestJS 10.x. Modular architecture. Decorator-based routing and guards.
  • API: REST with OpenAPI/Swagger auto-generated documentation. JSON request/response.
  • Auth: Passport.js with SAML 2.0 and OIDC strategies. JWT access tokens (15-minute expiry). HttpOnly refresh token cookies.
  • Permission checks: Custom @RequirePermission('att-001:write') decorator that reads from the IAM effective access cache. Checks permission codes, never role names.

Frontend: Next.js (TypeScript)

  • Framework: Next.js 14.x with App Router.
  • Styling: Tailwind CSS. Custom components built to the CampusOS design system.
  • State management: React Query (TanStack Query) for server state. Zustand for client state.
  • Forms: React Hook Form with Zod validation. Schema shared with backend.

Database: PostgreSQL 16

  • ORM: Prisma. Schema-first with migration generation. Type-safe query builder. Multi-schema support for tenant isolation.
  • Migrations: Prisma Migrate for schema changes. Custom scripts for partitions.
  • Connection pooling: PgBouncer in TRANSACTION mode per ADR-044.

Message Broker: Apache Kafka

  • Managed: Amazon MSK.
  • Client: KafkaJS.
  • Schema registry: Confluent Schema Registry. Avro schemas. BACKWARD_TRANSITIVE compatibility mode.
  • Topic naming: {env}.{domain}.{entity}.{verb} per ADR-057.

Cache: Redis

  • Managed: Amazon ElastiCache for Redis.
  • Client: ioredis.
  • Uses: IAM permission cache (5-min TTL + event-driven invalidation), tenant routing (500ms TTL), rate limiting (sliding window), feature flags (30s TTL).

Extracted Services

  • Video Processing: FFmpeg on ECS with GPU. S3 for storage.
  • AI Inference: AWS Bedrock behind the AI Gateway (ADR-004).
  • Notification / Push: Amazon SNS + SES. WebSocket via Socket.io.
  • Scheduling Solver: OR-Tools (Python) as a stateless container.
  • Transport Dispatch: Dedicated NestJS service with WebSocket.
  • Emergency Alert: Isolated service with local fallback state.

Development Tooling

  • Monorepo: pnpm + Turborepo.
  • Testing: Vitest (unit), Supertest (API integration), Pact (contract), Playwright (E2E).
  • Linting: ESLint + Prettier.
  • CI/CD: GitHub Actions. ECR for container images.

3. Environment Architecture

Four environments, each with its own AWS account (or VPC isolation), database cluster, Kafka cluster, and Redis instance. Complete isolation — DEV cannot accidentally connect to PROD data.

DEV — Development

  • Purpose: Active development. Claude writes and tests here.
  • Database: Single PostgreSQL instance. Rebuilt frequently. Synthetic seed data.
  • Kafka: Single-broker. Topics auto-created.
  • Scale: Minimal. One tenant schema. No real student data ever.
  • Deploy: Direct from dev branch on push.

SIT — System Integration Testing

  • Purpose: CI/CD executes automated tests. ChatGPT performs static review.
  • Database: Single PostgreSQL instance. 2–3 tenant schemas to test multi-tenancy.
  • Kafka: Single-broker. Full topic set.
  • Data: Richer than DEV — includes cross-module scenarios.
  • Deploy: Automated from sit branch after DEV gate passes. Ephemeral Docker environments for CI test runs.

CAT — Customer Acceptance Testing

  • Purpose: Human acceptance testing. Manual test scripts executed here.
  • Database: PostgreSQL cluster matching PROD topology. Multi-tenant.
  • Kafka: 3-broker cluster matching PROD.
  • Redis: Clustered matching PROD.
  • Scale: PROD-like. Same instance sizes. Same configuration.
  • Data: Initially: rich synthetic data. Post-launch: scheduled PROD copy-down with PII masking.
  • Key rule: CAT always mirrors PROD unless actively testing a new release.

PROD — Production

  • Database: PostgreSQL cluster. Multi-AZ. Automated backups. Read replicas for analytics.
  • Kafka: 3-broker MSK cluster. Multi-AZ.
  • Redis: ElastiCache cluster. Multi-AZ. AOF persistence.
  • Scale: Production. Auto-scaling groups. AWS API Gateway in front.
  • Data: Real. Protected. Encrypted at rest and in transit.
  • Deploy: From main branch after CAT sign-off. Zero-downtime rolling deployment.

4. Git Branching Strategy

Four long-lived branches map directly to the four environments. Feature branches are short-lived and merge into dev.

Branch Structure

  • main — production code. Only receives merges from cat. Tagged with semantic versions. Protected.
  • cat — customer acceptance testing. Only receives merges from sit. Protected.
  • sit — system integration testing. Only receives merges from dev. Protected.
  • dev — active development. Receives merges from feature branches.
  • feature/* — short-lived feature branches off dev. Named feature/{module}-{description}.

Promotion Flow

Code flows in one direction only: dev → sit → cat → main. No cherry-picking between environments.

  • dev → sit: Merge request with handoff package. CI auto-triggers. ChatGPT reviews.
  • sit → cat: ChatGPT approval + CI green. ChatGPT produces written verdict.
  • cat → main: Human sign-off. All CAT test scripts pass. Release notes generated.

Hotfix Procedure

  • Create hotfix/* branch from main.
  • Claude fixes and writes a targeted test.
  • Hotfix merges to main AND back-merges to cat, sit, dev.
  • ChatGPT reviews as a priority item.
  • Human verifies in PROD.
  • This is the ONLY exception to the one-direction flow.

5. Development Workflow — Claude

Claude is the sole developer for Wave 1. Every line of application code, every migration, every unit test, and every API endpoint is written by Claude.

Development Cycle

  • Step 1: Select the next module based on the Wave 1 build sequence (Section 13) and dependency order.
  • Step 2: Write database migrations. Prisma schema definitions. Verified in DEV.
  • Step 3: Write the module code. NestJS module: entities, repository, service, controller, events, guards.
  • Step 4: Write unit tests. Coverage >90%, 100% on critical paths.
  • Step 5: Run and verify. All tests pass. APIs respond. Events emit and consume.
  • Step 6: Build the handoff package (Section 11).
  • Step 7: Commit and push. Merge request to sit.

What Claude Produces Per Cycle

  • Database migration files (Prisma schema + SQL for partitions/constraints)
  • NestJS module with full CRUD + business logic
  • REST API endpoints with OpenAPI documentation
  • Kafka event producers and consumers
  • Permission guard decorators mapped to function codes
  • Unit tests with >90% coverage
  • Seed data scripts
  • Handoff package with compliance checklist

Naming Conventions

  • Modules: src/modules/{prefix}/ (e.g. src/modules/sis/, src/modules/cls/)
  • Files: {entity}.entity.ts, .repository.ts, .service.ts, .controller.ts
  • Tests: {entity}.service.spec.ts, {entity}.controller.spec.ts
  • Migrations: prisma/migrations/{timestamp}_{description}/migration.sql
  • Events: src/modules/{prefix}/events/{event-name}.event.ts
  • Guards: src/modules/{prefix}/guards/{function-code}.guard.ts

6. Code Review & Integration Testing — ChatGPT

ChatGPT receives the handoff package and CI/CD results. ChatGPT’s role is static review and architectural validation — the CI/CD pipeline handles execution.

What ChatGPT Receives

  • The complete handoff package (Section 11).
  • CI/CD test results (unit, integration, contract tests — all automated).
  • The frozen architecture documents as reference: ERD v10, Architecture Review, Function Library, Function-ERD Mapping.
  • A specific review prompt telling ChatGPT exactly what to verify.

What ChatGPT Does

  • Code review: reads every file and verifies: table names match ERD v10, permission codes match Function-ERD Mapping, module boundaries respected, event contracts match Kafka Topic Register, FK patterns follow ADR-055.
  • Architecture compliance: verifies Cross-Module Interaction Contract (5 rules), Finance Boundary, Procurement Scope.
  • Security analysis: verifies permission guards on every endpoint, sensitive data access-restricted, no injection vectors.
  • Test design review: evaluates whether tests cover critical paths and edge cases.

What ChatGPT Produces

  • Verdict: PASS or FAIL.
  • Findings: each with severity (P1–P3), file/line reference, ADR/ERD reference, recommended fix.
  • Approval or rejection: PASS = approve merge to cat. FAIL = Claude fixes and resubmits.

SIT Exit Criteria

  • CI pipeline green: all unit, integration, and contract tests pass.
  • Coverage >90% line coverage, 100% on critical paths.
  • ChatGPT static review: PASS with zero P1 and zero unacknowledged P2.

7. Customer Acceptance Testing — Human

The human reviewer is the final quality gate. Claude generates manual test scripts tied to the user journey documents.

CAT Script Structure

  • Precondition: what data/state must exist before the test.
  • Steps: specific actions with exact UI instructions.
  • Expected result: what the system should show after each action.
  • Pass/Fail: the human marks each step.

CAT Exit Criteria

  • All test scripts pass.
  • No open P1 or P2 defects.
  • Human reviewer signs off with date and name.
  • CAT stable for 24 hours after final test.

Defect Handling

  • Document the failure: which step, what happened, what was expected.
  • Roll back CAT to previous stable version (which matches PROD).
  • Claude fixes in DEV. Fix goes through full pipeline: DEV → SIT → CAT.
  • No shortcutting. Same code, same gates.

CAT = PROD Mirror

When no testing is active, CAT runs the same application version as PROD. Same schema. Same configuration. Same feature flags. After a release promotes to PROD, CAT is updated to match within 24 hours.

8. Production Deployment

Deployment Procedure

  • Step 1: Merge cat → main. Tag with semantic version (v1.0.0).
  • Step 2: Database migrations run against PROD. Backward-compatible — current app continues running.
  • Step 3: Rolling application deployment. New instances start, old drain, then shut down. Zero downtime.
  • Step 4: Health checks. Automatic rollback if checks fail.
  • Step 5: Feature flags control visibility. New features may be deployed but hidden.
  • Step 6: Post-deployment smoke tests.
  • Step 7: Update CAT to match PROD within 24 hours.

9. Database Strategy

Initial Seed Data

  • Platform schema: organisations, schools, grading scales, feature flags, IAM seed (roles, permissions, all 378 permission codes), scope types, identity providers.
  • Tenant schema: 50–100 students, 20–30 staff, guardians, classes, schedules, attendance, grades, behaviour, health (synthetic), cross-module relationships.
  • Persona accounts: one user per persona type with appropriate role assignments so every journey can be tested.

PROD Copy-Down

  • Frequency: weekly or on-demand before a testing cycle.
  • Process: pg_dump from PROD read replica → restore to CAT/SIT → data masking scripts.
  • Data masking (mandatory): student names randomised, emails → @example.com, phones zeroed, guardian names randomised, health details replaced with synthetic, financial accounts randomised. Referential integrity preserved (same person_id links to same randomised name).
  • NOT masked: UUIDs, structural data (school names, role assignments, class structures, schedule configs), aggregate counts.
  • Verification: validation script confirms no real PII remains.

Migration Management

  • All migrations forward-only. No DROP TABLE, no DROP COLUMN.
  • Column renames: add → backfill → swap pattern.
  • Partitions managed in migration scripts.
  • Each migration tested in DEV, verified in SIT, confirmed in CAT.

12. Rollback Strategy

CAT Rollback

  • Stop test cycle. Document failures.
  • Roll back CAT application to previous stable (PROD-mirror) version.
  • No database rollback needed — backward-compatible migrations mean the PROD-version app works on the current schema.
  • Claude fixes in DEV. Fix goes through full pipeline.

PROD Rollback

  • Application rollback: revert to previous ECS container image. One-command operation.
  • Database: previous app version works on current schema (backward-compatible). No DB rollback.
  • If database rollback required (extremely rare): restore from pre-deployment snapshot.
  • Hotfix procedure (Section 4) applies.

Key Principle

Backward-compatible migrations are the foundation. If the previous app version runs on the current schema, rollback is a container swap — fast, safe, data-preserving. This is why DROP COLUMN and DROP TABLE are forbidden.

16. SIT Execution Model — CI/CD, Not AI Execution

Peer review finding 2.1 — corrected. The original plan stated that ChatGPT executes code in its own environment. That was architecturally incorrect. AI cannot reliably serve as an execution environment — there is no deterministic runtime, no system-level validation, and no true integration environment. This section replaces that model.

Corrected SIT Architecture

SIT execution is performed by automated CI/CD infrastructure, not by ChatGPT. The pipeline is:

  • GitHub Actions CI runner is the execution layer. When code is promoted from dev to sit, a CI workflow automatically: checks out the code, spins up ephemeral Docker containers (PostgreSQL, Kafka, Redis), applies migrations, runs the full unit test suite, runs integration tests, runs contract tests, and produces a test report.
  • Ephemeral preview environments are created per merge request using Docker Compose (or AWS ECS task definitions for heavier scenarios). The environment lives for the duration of the review, then is torn down. No persistent SIT infrastructure is required for execution — only for ChatGPT’s static review.
  • ChatGPT’s corrected role in SIT is: (1) static code review — reading the source code and verifying it against the frozen architecture documents; (2) architectural compliance validation — checking that table names match ERD v10, permission codes match Function-ERD Mapping, module boundaries are respected, and event contracts match the Kafka Topic Register; (3) test design review — evaluating whether the test suite covers the critical paths and edge cases; (4) security analysis — identifying permission guard gaps, PII exposure risks, and injection vectors. ChatGPT does NOT execute code.

SIT Workflow (Revised)

  • Step 1: Claude pushes to dev, creates merge request to sit with the handoff package.
  • Step 2: GitHub Actions CI automatically triggers: build → migrate → unit tests → integration tests → contract tests → coverage report.
  • Step 3: CI results are attached to the merge request.
  • Step 4: ChatGPT receives the merge request + CI results + handoff package. ChatGPT performs static review and architectural compliance checks.
  • Step 5: ChatGPT produces a written review: PASS (approve merge to sit) or FAIL (specific findings for Claude to fix).
  • Step 6: On PASS + CI green, code merges to sit and deploys to the persistent SIT environment for exploratory review.

SIT Exit Criteria (Revised)

  • CI pipeline green: all unit, integration, and contract tests pass.
  • Coverage >90% line coverage, 100% on critical paths.
  • ChatGPT static review: PASS with zero P1 and zero unacknowledged P2 findings.
  • No architectural violations detected.

17. Contract Testing Strategy

Peer review finding 2.2 — addressed. Cross-module interactions via Kafka events and REST APIs must have enforceable contracts. Without contract testing, integration breaks will be frequent and silent.

Kafka Contract Testing

  • Schema Registry: Confluent Schema Registry deployed alongside MSK in all environments. All Kafka event payloads are defined as Avro schemas registered in the schema registry.
  • Compatibility mode: BACKWARD_TRANSITIVE. A new schema version must be backward-compatible with all previous versions. Consumers using older schema versions must be able to read messages produced with newer schemas. This is enforced at the registry level — a non-compatible schema change is rejected at registration time, before any code deploys.
  • Contract tests per topic: every Kafka topic has a contract test that verifies: (1) the producer emits a message that validates against the registered schema; (2) every registered consumer can deserialise the message; (3) required fields are present; (4) the payload matches the event contract documented in the Kafka Topic Register.
  • Schema evolution rules: adding optional fields = minor version (backward-compatible). Removing fields, changing types, or renaming = major version (requires new topic + consumer migration plan). No in-place breaking changes to any published schema.
  • Dead letter queue policy: every consumer has a DLQ topic ({original-topic}.dlq). Messages that fail deserialization or processing after 3 retries go to the DLQ. DLQ depth is monitored — any non-zero DLQ depth triggers a P2 alert.

API Contract Testing

  • Tooling: Pact (consumer-driven contract testing). Each frontend or service that consumes an API endpoint publishes a Pact contract defining the expected request and response shapes.
  • Provider verification: the CI pipeline runs Pact provider verification tests on every backend build. If a code change would break an existing consumer contract, the build fails.
  • API versioning: URL-path versioning (/api/v1/, /api/v2/). New versions are additive — v1 endpoints continue to work after v2 is released. Deprecation: v1 endpoints log a deprecation warning for 2 release cycles before removal.
  • OpenAPI spec as contract source: the generated OpenAPI spec is checked into Git. CI verifies that the running API matches the committed spec — any divergence fails the build. This prevents undocumented API changes.

18. Multi-Tenant Isolation Enforcement

Peer review finding 2.3 — addressed. Schema-per-tenant provides database-level isolation, but a single application bug could route a query to the wrong schema. Runtime enforcement is required at every layer.

Layer 1 — Request-Level Tenant Resolution

  • Every HTTP request must resolve to a tenant before any business logic executes.
  • Tenant resolution middleware (NestJS global interceptor) extracts tenant context from: (1) the JWT’s tenant_id claim, (2) the request hostname (subdomain routing via Route 53), or (3) an explicit X-Tenant-ID header (for platform-level service-to-service calls only, validated against the caller’s service principal scope).
  • If tenant resolution fails, the request is rejected with 400 Bad Request. No fallback to a default tenant. No anonymous queries.
  • The resolved tenant_id is stored in a request-scoped AsyncLocalStorage context that is available to every layer of the application without passing it as a parameter.

Layer 2 — ORM-Level Schema Enforcement

  • Prisma middleware intercepts every database query and sets the PostgreSQL search_path to the resolved tenant’s schema before execution.
  • The middleware reads the tenant_id from AsyncLocalStorage. If no tenant context exists (programming error), the query is rejected with an internal error — never executed against a default or public schema.
  • Platform schema queries (platform_, iam_ tables) explicitly set search_path to the platform schema. These are the only queries permitted to read cross-tenant data.
  • Every query log entry includes the resolved tenant_id for audit trail and debugging.

Layer 3 — Cross-Tenant Access Prevention

  • Integration tests explicitly verify that a request authenticated as Tenant A cannot read, write, or receive data belonging to Tenant B.
  • A dedicated cross-tenant isolation test suite runs in CI on every build: it creates two tenant schemas, inserts data in each, and verifies that queries scoped to Tenant A return zero rows from Tenant B.
  • The CI pipeline fails if any cross-tenant data leakage is detected.
  • Production monitoring: a weekly audit job compares query logs against tenant boundaries to detect any anomalous cross-tenant access patterns.

Layer 4 — PostgreSQL Row-Level Security (Wave 2+)

As an additional defence-in-depth measure, PostgreSQL Row-Level Security (RLS) policies will be added to high-sensitivity tables (hlth_, svc_, pay_, fin_) in Wave 2. RLS acts as a database-level backstop — even if the application middleware fails, the database itself enforces that a connection authenticated to Tenant A’s schema cannot access Tenant B’s rows. This is defence-in-depth, not the primary enforcement mechanism.

19. Rate Limiting & Abuse Protection

Peer review finding 2.4 — addressed. Authentication and authorisation are necessary but not sufficient. Without rate limiting, a compromised or malfunctioning client can exhaust resources for all tenants.

API Rate Limiting

  • Implementation: Redis-backed sliding window rate limiter (NestJS @Throttle decorator backed by ioredis).
  • Per-user limits: 100 requests/minute for standard API endpoints. 10 requests/minute for expensive operations (report generation, bulk exports, AI study tool). 1,000 requests/minute for high-frequency endpoints (real-time attendance check-in, POS transactions).
  • Per-tenant limits: aggregate rate limit across all users in a tenant. Calculated based on tenant tier (platform_tenant_quotas.api_rate_limit). Default: 5,000 requests/minute for standard tenants. Configurable per tenant.
  • Response on limit: HTTP 429 Too Many Requests with Retry-After header. Logged in platform_tenant_usage_metrics for noisy-neighbor detection (ADR-045).

WebSocket Protection

  • WebSocket connections require authenticated JWT. No anonymous WebSocket connections.
  • Per-user connection limit: 5 concurrent WebSocket connections. Additional connections are rejected.
  • Per-tenant connection pool: configurable maximum concurrent WebSocket connections per tenant.
  • Message rate limit: 30 messages/second per connection. Exceeding the limit triggers a 60-second cooldown.
  • Transport Dispatch (driver WebSocket) and Emergency Alert have dedicated connection pools that are isolated from the general WebSocket pool.

DDoS Protection

  • AWS WAF (Web Application Firewall) in front of the ALB. Managed rule sets for common attack patterns (SQL injection, XSS, known bad IPs).
  • AWS Shield Standard (included with ALB) for volumetric DDoS protection.
  • CloudFront for static asset serving — absorbs traffic spikes at the edge.
  • Geographic restriction: configurable per-tenant IP allowlisting for admin-level access.

Abuse Detection

  • platform_tenant_usage_metrics (hourly) feeds the noisy-neighbor detection system (ADR-045).
  • Automated escalation: 2× rolling median → throttle. 5× → isolate. 10× → alert + manual review.
  • Credential stuffing detection: 5 failed login attempts within 5 minutes → account lockout (iam_auth_event monitoring).

20. Feature Flag System Design

Peer review finding 2.5 — addressed. Feature flags are a critical deployment safety mechanism. They need defined scope, rollout strategy, and kill-switch behaviour.

Flag Scope Hierarchy

  • PLATFORM: applies to all tenants globally. Controlled by platform administrators only. Used for: enabling entirely new modules, platform-wide maintenance mode, emergency feature disablement.
  • DISTRICT: applies to all schools within an organisation. Controlled by district administrators. Used for: district-level policy features (e.g. content moderation strictness, procurement approval chains).
  • BUILDING: applies to a single school. Controlled by building administrators. Used for: school-specific features (e.g. achievements system on/off, hall pass limits, school store availability, device selection).
  • ROLE: applies to users with a specific role. Used for: beta features visible only to administrators before general rollout.
  • USER: applies to a specific user. Used for: individual beta testers, debugging specific user issues.
  • Resolution order: most specific wins. A user-level flag overrides a role-level flag, which overrides a building-level flag, and so on. If no flag is set at any level, the platform default applies.

Rollout Strategy

  • Gradual rollout: percentage-based. A flag can be set to 10% → 25% → 50% → 100% over successive deployments. The percentage is applied consistently per user (hash of user_id % 100) so the same user always sees the same state.
  • Canary releases: deploy to a single building first, monitor for 24 hours, then expand. The feature flag makes this a configuration change, not a code deployment.
  • A/B testing: two variants of a feature can be flagged independently. Analytics events include the variant ID for comparison.

Kill-Switch Behaviour

  • Every feature flag has a kill-switch: setting the flag to OFF at PLATFORM level immediately disables the feature for all tenants, regardless of lower-level overrides.
  • Kill-switch latency: <30 seconds. Feature flag state is cached in Redis with a 30-second TTL. The kill-switch writes to Redis directly (bypassing the database) for immediate propagation.
  • Kill-switch does NOT delete data. If a feature is killed, the data created while it was active is preserved. Reactivating the flag restores access to the data.

Implementation

school_feature_flags (existing table in ERD v10) is extended with: scope ENUM(PLATFORM, DISTRICT, BUILDING, ROLE, USER), scope_entity_id UUID (the org/school/role/user the flag applies to), rollout_percentage INT (0–100), and variant TEXT. The FeatureFlagService reads the effective flag state from Redis (30s TTL) with database fallback.

21. Disaster Recovery Strategy

Peer review finding 2.6 — addressed. Backups and rollback are necessary but not sufficient for a K-12 SaaS platform where downtime directly impacts school operations.

Recovery Targets

  • RTO (Recovery Time Objective): 1 hour for full platform restoration. 15 minutes for the Emergency Alert Service (which has its own minimal-dependency execution path per Architecture Review Section 16.6).
  • RPO (Recovery Point Objective): 5 minutes for database (continuous WAL archiving). 0 for Kafka (replicated across brokers). 30 seconds for Redis (AOF persistence with 1-second fsync).

Database Recovery

  • Automated backups: RDS automated daily snapshots + continuous WAL archiving to S3. Retention: 35 days.
  • Point-in-time recovery: restore to any second within the retention window. Tested monthly: restore a snapshot to a temporary instance and verify data integrity.
  • Cross-region backup: automated snapshot replication to a secondary AWS region (us-west-2 if primary is us-east-1). Supports regional failover.
  • Backup validation: weekly automated job restores the latest snapshot to a temporary instance, runs a schema validation script, and confirms row counts match expected ranges. Alert if validation fails.

Kafka Recovery

  • MSK replication factor = 3. Data survives the loss of 2 brokers.
  • Topic retention: 7 days for operational topics. 30 days for audit topics. Infinite for DLQ topics.
  • Recovery procedure: if a broker fails, MSK automatically replaces it and replicates data. If the entire cluster fails, consumer offsets are stored in a separate coordination topic that recovers with the cluster.
  • For catastrophic cluster loss: consumers replay from the earliest available offset. Idempotent consumers (platform_event_consumer_idempotency) prevent duplicate processing.

Redis Recovery

  • ElastiCache Redis with AOF persistence (appendfsync everysec). Multi-AZ replica for automatic failover.
  • If Redis is lost entirely: the application falls back to database reads. IAM cache misses are computed from iam_role_assignment → roles → role_permissions. Tenant routing falls back to platform_tenant_routing table. Performance degrades but the system remains operational.
  • Redis is a cache, not a source of truth. Full Redis loss = performance degradation, not data loss.

Regional Failover (Wave 4+)

Full multi-region active-passive architecture is planned for Wave 4+ per Architecture Review Section 21. The current single-region deployment uses Multi-AZ for all services, which protects against availability zone failure but not regional failure. For Wave 1–3, the regional failover plan is: restore from cross-region backups in the secondary region. Estimated RTO for regional failover: 4–6 hours (acceptable for Wave 1–3 scale).

Disaster Recovery Runbook

A DR runbook will be produced alongside the infrastructure provisioning. It will include: step-by-step procedures for each failure scenario (single instance, AZ failure, regional failure), responsible parties, communication templates, and post-incident review requirements. The runbook is tested quarterly via a tabletop exercise.

22. Automated End-to-End Testing Strategy

Peer review finding 2.7 — addressed. Manual CAT is the initial approach, but without automated E2E testing, CAT becomes a slow manual bottleneck as the system grows.

Phased Approach

  • Wave 1: manual CAT scripts (as defined in Section 7). Claude generates step-by-step test scripts tied to journey documents. Human executes manually.
  • Wave 2: automated E2E tests using Playwright. Claude converts the highest-value CAT scripts into automated Playwright tests that run against the CAT environment. Manual testing continues for new features; automated tests cover regression.
  • Wave 3+: full automated regression suite. Every CAT script that has been manually verified at least once is converted to an automated Playwright test. Manual testing is reserved for new features and exploratory testing only.

Playwright Test Architecture

  • Journey-based test suites: one test file per persona journey section. Example: tests/journeys/teacher/attendance-taking.spec.ts exercises the teacher’s attendance workflow.
  • Synthetic test users: pre-seeded accounts per persona type with known credentials. The test framework logs in as the appropriate persona before executing the journey steps.
  • Data isolation: E2E tests run in a dedicated tenant schema. Test data is reset before each suite run.
  • CI integration: Playwright tests run in the CI pipeline on the cat branch. They are NOT a gate for sit → cat promotion (that remains ChatGPT’s static review + CI unit/integration tests). They ARE a gate for cat → main (production) promotion.
  • Flake management: tests that fail intermittently are quarantined (marked @flaky) and investigated. Quarantined tests do not block the pipeline but are reported.

Synthetic User Flows

The following critical paths are automated first (highest risk, highest frequency):

  • Teacher attendance cycle: login → take attendance → mark tardy → verify parent notification
  • Student assignment cycle: login → view assignment → submit → verify submission recorded
  • Parent billing cycle: login → view balance → make payment → verify receipt
  • Admin enrolment cycle: create application → approve → generate offer → accept → verify student created
  • Procurement cycle: submit requisition → approve → create PO → receive → distribute

23. Kafka Scaling & Partition Strategy

Peer review finding 3.1 — addressed.

Partition Strategy

  • Default: 6 partitions per topic. Sufficient for Wave 1–2 throughput. Expandable without downtime.
  • High-volume topics (sis.attendance.recorded, cls.grade.posted, msg.notification.queued): 12 partitions from day one.
  • Partition key: tenant_id for all topics. This ensures all events for a single tenant are ordered within a partition (important for attendance, grading, and financial events). Cross-tenant ordering is not required.
  • Consumer group scaling: one consumer instance per partition. Auto-scaling consumer groups based on consumer lag metrics. If lag exceeds 1,000 messages for >5 minutes, scale up consumers.

Idempotent Consumers

All consumers are idempotent per ADR-057. The platform_event_consumer_idempotency table records (consumer_name, event_id, processed_at). Before processing, the consumer checks for the event_id. If present, the event is skipped. This makes replay safe — consumers can be restarted from any offset without duplicate side effects.

Dead Letter Queue Policy

  • Every topic has a corresponding .dlq topic.
  • Messages fail to DLQ after 3 processing attempts with exponential backoff (1s, 5s, 30s).
  • DLQ depth is a monitored metric. Non-zero DLQ depth = P2 alert.
  • DLQ replay: a manual CLI tool allows operators to replay DLQ messages back to the original topic after the root cause is fixed.
  • DLQ retention: infinite (messages are never auto-deleted from DLQ).

24. Prisma Multi-Schema Scaling Strategy

Peer review finding 3.2 — addressed. Prisma with schema-per-tenant can degrade at high tenant counts. The mitigation is a tiered approach.

Tiered Tenant Hosting

  • Tier 1 (Wave 1–3, <500 tenants): schema-per-tenant on a single RDS cluster. 80 active schemas per cluster (ADR-001, ADR-008), 20-schema migration buffer. At 80 schemas, add a second cluster.
  • Tier 2 (500–2,000 tenants): multiple RDS clusters with platform_tenant_routing directing each tenant to its cluster. Prisma client pool per cluster. Migration orchestrator applies migrations to all clusters in parallel.
  • Tier 3 (2,000+ tenants, if needed): evaluate hybrid model — smaller tenants share a schema with tenant_id column filtering (logical isolation), larger tenants retain dedicated schemas (physical isolation). This is an architectural change that requires validation at Wave 4+ scale.

Migration at Scale

  • Migration orchestrator: a platform-level service that reads all active tenant schemas from platform_tenant_routing and applies migrations in batches of 10 schemas per cluster.
  • Migration health: platform_migration_status table tracks per-tenant migration state (PENDING, RUNNING, COMPLETED, FAILED). Failed migrations are retried automatically. After 3 failures, the tenant is flagged for manual intervention.
  • Zero-downtime: migrations are backward-compatible (Section 12). The application continues running on the pre-migration schema while migrations apply. No maintenance window required.

25. IAM Cache Invalidation Strategy

Peer review finding 3.3 — addressed. A 5-minute TTL means permission changes take up to 5 minutes to propagate. For security-critical changes (role revocation, account suspension), this is too slow.

Dual Invalidation Model

  • TTL-based (background): Redis TTL of 5 minutes for routine cache refresh. This handles the steady-state case where permissions rarely change.
  • Event-driven (immediate): iam.role.assigned and iam.role.revoked Kafka events trigger immediate cache invalidation. The IAMCacheInvalidationWorker consumes these events and: (1) deletes the affected user’s entry from Redis; (2) updates iam_effective_access_cache in the database; (3) if the change is a REVOCATION or SUSPENSION, publishes a cache-bust event to all API instances via Redis Pub/Sub, forcing an immediate re-read on the next request for that user.
  • Latency targets: routine changes (new role grant) = <30 seconds via Kafka consumer. Security-critical changes (role revocation, account suspension, emergency access grant) = <5 seconds via Redis Pub/Sub direct invalidation.
  • Account suspension: iam.account.suspended event triggers immediate invalidation across ALL caches and ALL API instances. The auth middleware checks a Redis SUSPENDED_ACCOUNTS set on every request (O(1) lookup). Suspension is effective within 1 second.

26. Observability & Distributed Tracing

Peer review finding 4.2 — addressed. Metrics are necessary but not sufficient for debugging distributed flows. Trace-first observability is required.

Trace Propagation

  • Trace ID: a UUIDv7 trace_id is generated at the API gateway (or the first service to receive a request) and propagated through: (1) all HTTP headers (X-Trace-ID); (2) all Kafka event envelopes (correlation_id field per ADR-057); (3) all database query comments (/* trace_id=... */); (4) all log entries.
  • Span tracking: AWS X-Ray SDK (or OpenTelemetry) creates spans for: HTTP request handling, database queries, Kafka produce/consume, Redis operations, and external service calls (Stripe, AI Gateway, IdP).
  • Cross-service propagation: when the monolith calls an extracted service (Video Processing, Scheduling Solver, etc.), the trace_id is passed in the request header. The extracted service creates child spans under the same trace.

Structured Logging

  • All log entries are structured JSON with mandatory fields: timestamp, level, trace_id, tenant_id, user_id (if authenticated), module, message.
  • Sensitive data (PII) is never logged. Log entries reference entity IDs, not entity content.
  • Log levels: ERROR (P1 alert), WARN (monitored), INFO (operational), DEBUG (DEV/SIT only).
  • Log aggregation: CloudWatch Logs Insights for search and analysis. Retention: 90 days in CloudWatch, archived to S3 for 1 year.

Correlation ID End-to-End Example

A teacher takes attendance → POST /api/v1/attendance with X-Trace-ID: abc123 → the attendance service inserts a row into sis_attendance_records (query comment includes trace_id) → emits sis.attendance.recorded Kafka event with correlation_id: abc123 → the notification worker consumes the event, sends a parent notification, and logs the delivery with correlation_id: abc123 → the analytics worker consumes the same event and updates rpt_daily_attendance_summary with correlation_id: abc123. Every step of this flow is searchable by trace ID abc123.

27. Cost Governance

Peer review finding 4.3 — addressed. The AWS stack (MSK, RDS, ECS, ElastiCache, S3, CloudFront) will become expensive at scale. Cost governance must be built in from day one.

Environment Cost Control

  • DEV and SIT: minimal instance sizes. Auto-shutdown outside working hours (ECS desired count = 0 from 8pm to 6am). RDS single instance (no Multi-AZ). Estimated: $200–400/month per environment.
  • CAT: PROD-like sizing but with Reserved Instances or Savings Plans. Can be scaled down when not actively testing. Estimated: $800–1,200/month.
  • PROD: right-sized based on actual utilisation data. Auto-scaling with scale-in during off-hours (school SaaS has predictable usage patterns: heavy 7am–4pm, light evenings and weekends). Estimated: $2,000–5,000/month initially, scaling with tenant count.

Tenant-Level Cost Attribution

  • platform_tenant_usage_metrics (hourly) tracks per-tenant: API requests, database query count, Kafka messages produced/consumed, S3 storage, and AI token usage.
  • Monthly cost attribution report: allocates infrastructure cost to tenants based on usage proportions. This supports: (1) pricing model validation (are large tenants subsidised by small ones?); (2) noisy-neighbor identification; (3) capacity planning.

Cost Alerts

  • AWS Budgets alert at 80% and 100% of monthly budget per environment.
  • Per-service cost tracking: if any single service (RDS, MSK, ECS) exceeds 150% of its expected monthly cost, a P3 alert is triggered for investigation.
  • S3 lifecycle policies: move infrequently accessed objects to S3 Infrequent Access after 90 days, to S3 Glacier after 365 days.

28. Parallel Development & Ownership Model

Peer review finding 4.1 — addressed. Claude is the sole developer for Wave 1. This is a practical constraint (not a design choice) and will not scale. This section defines how the development model evolves.

Wave 1 — Single Developer (Claude)

Claude develops sequentially, module by module. This works because: (1) the build sequence is dependency-ordered — each cycle depends on the previous; (2) there are no merge conflicts because there is one developer; (3) the architecture is sufficiently specified that a single developer can maintain coherence across all modules. The risk is throughput — Wave 1 will take longer than a parallel team would need.

Wave 2+ — Parallel Development Model

When additional developers (human or AI) are added, the module-to-service-domain mapping (Architecture Review Section 18) defines ownership boundaries:

  • Each service domain is owned by one developer or team.
  • Cross-domain dependencies are managed through the contract testing strategy (Section 17) — Kafka schemas and API Pact contracts prevent one domain’s changes from breaking another.
  • Merge conflicts are limited to shared infrastructure (M0 Platform Core, shared types, CI configuration). These are resolved by the platform owner.
  • The handoff package format (Section 11 of the original plan) works regardless of whether the developer is Claude, another AI, or a human — it’s a structured deliverable, not a conversation.

Ownership Register

When parallel development begins, each service domain will have a named owner. The owner is responsible for: code quality within the domain, migration correctness, contract maintenance, and handoff package production. The ownership register will be maintained as a living document alongside the architecture baseline.

29. API Gateway Layer

All external traffic routes through AWS API Gateway before reaching any CampusOS service. The gateway is the front-door control plane for auth enforcement, rate limiting, request validation, and traffic shaping.

Architecture

  • Service: AWS API Gateway (HTTP API) in front of the ALB. No direct ALB access from the internet.
  • Upgrade path: Kong Gateway on ECS if more control needed (Wave 3+).
  • Internal traffic: service-to-service within VPC bypasses the gateway. Authenticated via service principal JWTs.

Gateway Responsibilities

  • Authentication: validates JWT signature and expiry. Invalid tokens rejected at the gateway — never reach the application.
  • Rate limiting: per-user and per-tenant limits (Section 19) via usage plans. 429 returned before reaching ECS.
  • Request validation: body schemas validated against OpenAPI spec for critical endpoints.
  • Traffic shaping: burst limits (500 req/s burst, 100 req/s sustained per tenant).
  • CORS: per-environment. DEV allows localhost; PROD allows tenant subdomain only.
  • Access logging: every request logged with tenant_id, user_id, endpoint, status, latency.

Routing

  • /api/v1/* → main application (ECS ALB)
  • /api/v1/video/* → Video Processing Service
  • /api/v1/dispatch/* → Transport Dispatch Service (WebSocket upgrade)
  • /api/v1/emergency/* → Emergency Alert Service (isolated, no shared dependencies)
  • /ws/* → WebSocket connections (authenticated on $connect)
  • /health → health check (no auth, Route 53 target)

WAF Integration

AWS WAF attached to the API Gateway (not the ALB). All traffic passes through WAF before reaching any service. Managed rule sets: AWSManagedRulesCommonRuleSet (XSS, SQLi), AWSManagedRulesKnownBadInputsRuleSet, and a custom rule blocking tenant_id in query parameters (tenant resolution must come from JWT, never client-supplied).
