April 2026 · 11 Steps · ERD v11 · M20 SIS Core

Plan Overview

Cycle 1 is the first domain module. Where Cycle 0 built the platform (identity, IAM, tenancy, auth), Cycle 1 builds the first feature a school actually uses. It establishes the pattern that every subsequent cycle follows: tenant schema migration, NestJS module, API endpoints with permissions, Next.js UI pages, seed data, and an end-to-end vertical slice test.

Vertical Slice Deliverable

Teacher logs in → sees today's classes → clicks Period 1 Algebra → sees 25 students (all pre-populated as Present) → taps Maya Chen → marks Tardy at 8:15 → submits → Kafka event fires → David Chen (parent) sees the notification in his dashboard.

Cycle 1 is the Pattern-Setting Cycle

This cycle takes longer than subsequent ones because it establishes conventions that every future module reuses: how tenant migrations work alongside platform schema, how NestJS domain modules are structured, how API DTOs and validation work, how the UI layout shell and authenticated pages are built, and how Kafka events bridge between modules. Cycles 2–8 follow the same template.

Step-by-Step Build Plan

01 Academic Structure Schema
Schema

Create the tenant-scoped tables that define a school's academic structure: years, terms, departments, courses, classes, teacher assignments, and student enrollments. These are the building blocks that attendance, grading, scheduling, and everything else reference.

Tables (7)
  1. sis_academic_years — school_id, name, start_date, end_date, is_current. One row per school year.
  2. sis_terms — FK(academic_year), name, start_date, end_date, term_type (SEMESTER, QUARTER, TRIMESTER). Subdivisions of a year.
  3. sis_departments — school_id, name, head_employee_id. Academic departments (Math, English, Science).
  4. sis_courses — FK(department), code, name, credit_hours, grade_level. The course catalogue.
  5. sis_classes — FK(course), FK(academic_year), FK(term), section_number, room, capacity, schedule. A specific section of a course in a term.
  6. sis_class_teachers — FK(class), FK(hr_employees via person), is_primary_teacher. Teacher-to-class assignment.
  7. sis_enrollments — FK(student), FK(class), status (ACTIVE, DROPPED, TRANSFERRED), enrolled_at. Student-to-class membership.
Implementation

These are tenant-schema tables, so they go in the SQL tenant migration pipeline (not Prisma platform schema). Create packages/database/prisma/tenant/migrations/002_sis_academic_structure.sql. All existing tenant schemas (tenant_demo, tenant_test) get this migration applied on next provisioning run.

Produces → 7 new tables in every tenant schema. pnpm --filter @campusos/database provision --subdomain=demo applies cleanly.
02 Student & Family Schema
Schema

Create the tenant-scoped tables for student records, staff records, guardians, families, and emergency contacts. These tables link back to the platform schema via FK(iam_person) and FK(platform_students) for identity, and FK(platform_users) for auth context.

Tables (8)
  1. sis_students — FK(platform_students), school_id, student_number, grade_level, homeroom_class_id, enrollment_status. The school-scoped student record.
  2. sis_staff — FK(iam_person), FK(platform_users), school_id, staff_type (TEACHER, ADMINISTRATOR, SUPPORT, COUNSELLOR).
  3. sis_guardians — FK(iam_person), FK(platform_users nullable), school_id, relationship type, preferred contact method, FK(sis_families).
  4. sis_student_guardians — FK(student), FK(guardian), has_custody, is_emergency_contact, receives_reports, portal_access, portal_access_scope. The link that triggers IAM guardian access derivation.
  5. sis_families — family_name, address, FK(platform_families nullable). The household unit.
  6. sis_family_members — FK(family), FK(iam_person), person_type, relationship_to_family, is_primary_contact. Everyone in the family.
  7. sis_emergency_contacts — FK(student), name, relationship, phone, is_authorised_pickup.
  8. sis_student_notes — FK(student), FK(author hr_employee), note_text, note_type, is_parent_visible. Teacher pastoral observations.
Identity Contract (ADR-055)

sis_students references platform_students (portable identity) which references iam_person (universal identity). The student's name and date of birth live in iam_person — never duplicated in sis_students. The school-scoped record carries only school-specific data: student number, grade level, enrollment status.

Produces → 8 new tables in every tenant schema. Student ↔ guardian relationships ready for IAM access derivation.
Depends on → Step 1 (sis_classes for homeroom FK)
03 Attendance Schema
Schema

The core attendance tables. Attendance records are pre-populated at school day start with PRESENT status. Teachers then mark exceptions. Each record carries evidence source, confirmation status, and links to parent explanations and absence requests.

Tables (3)
  1. sis_attendance_records — FK(student), FK(class), date, period, status (PRESENT, ABSENT, TARDY, EARLY_DEPARTURE, EXCUSED), confirmation_status (PRE_POPULATED, CONFIRMED), evidence_source, marked_by FK(platform_users), parent_explanation, absence_request_id. The main attendance fact table.
  2. sis_attendance_evidence — FK(attendance_record), evidence_type (BUS_SCAN, DOOR_SCAN, PHOTO, NOTE), source_ref_id. Supporting evidence for automated or manual attendance.
  3. sis_absence_requests — FK(student), submitted_by FK(platform_users/guardian), date range, reason category, status (PENDING, APPROVED, REJECTED, AUTO_APPROVED), reviewed_by, supporting document. Parent-submitted absence notifications and advance requests.
Pre-Population Pattern

At school day start (or on first access), a background job or on-demand service creates one sis_attendance_records row per student per class period with status=PRESENT and confirmation_status=PRE_POPULATED. The teacher's action is to mark exceptions and confirm. Only CONFIRMED records count in official reports. This means the teacher's workflow is exception-based: scan the list, tap the 2–3 students who are tardy or absent, submit.

Produces → 3 new attendance tables. Total tenant schema now has ~23 tables (5 from Cycle 0 + 18 from Cycle 1).
Depends on → Steps 1–2 (students and classes must exist)

Produces → A demo school that looks and feels populated. The teacher sees real class rosters. The parent sees real attendance data.

05 SIS NestJS Module
Code

Build the SIS domain module in the NestJS API. This is the first domain module added to the modular monolith — it establishes the pattern for all subsequent modules.

Module Structure
  1. SisModule — NestJS module declaration. Imports TenantModule for scoped Prisma. Exports services for other modules to consume.
  2. StudentService — CRUD for students. Resolves identity from platform schema. Lists students by class, grade level, enrollment status.
  3. ClassService — CRUD for classes. Includes roster (enrolled students), assigned teachers, schedule. GET /classes/:id/roster is the key endpoint for attendance.
  4. FamilyService — Family and guardian management. Links guardians to students. Queries for "who are Maya's guardians?" and "what students does David have?"
  5. StudentController — REST endpoints: GET /students, GET /students/:id, POST /students, PATCH /students/:id. All protected with @RequirePermission('stu-001:read') or 'stu-001:write'.
  6. ClassController — GET /classes, GET /classes/:id, GET /classes/:id/roster, GET /classes/my (teacher's own classes).
  7. DTOs — Zod schemas in packages/shared for request/response validation. Shared between API and UI for type safety.
Tenant-Scoped Queries

Every query in SisModule uses TenantPrismaService.executeInTenantContext() so queries automatically target the correct tenant schema. The teacher at Lincoln Elementary only sees Lincoln Elementary's students.

Produces → SisModule with 3 services, 2 controllers, ~15 REST endpoints. Swagger documentation auto-generated. All endpoints permission-protected.
Depends on → Step 4 (seed data to test against), Cycle 0 (TenantModule, IamModule, AuthModule)
06 Attendance NestJS Module
Code

The attendance-specific logic. Handles pre-population, individual and batch marking, confirmation, and Kafka event emission. This module consumes SisModule services (student roster) and produces events that downstream modules will consume (notifications in Cycle 3).

Module Structure
  1. AttendanceModule — NestJS module. Imports SisModule for student/class data.
  2. AttendanceService — Core logic: pre-populate a class period, mark individual student, batch-mark a class, get attendance for a date/class, get student attendance history.
  3. AttendancePrePopulationService — Creates PRESENT records for all enrolled students at day start. Called on-demand when a teacher opens a class or via a scheduled job.
  4. AttendanceController — REST endpoints: GET /classes/:id/attendance/:date (roster with statuses), PATCH /attendance/:id (mark single student), POST /classes/:id/attendance/:date/batch (submit whole class), GET /students/:id/attendance (student history), POST /absence-requests (parent submits absence).
  5. Kafka Events — att.attendance.marked (individual mark), att.attendance.confirmed (class period submitted), att.student.marked_tardy (triggers parent notification), att.student.marked_absent (triggers absence follow-up). Events emitted but not consumed in Cycle 1 — Cycle 3 (Communications) adds the notification consumer.
Permission Mapping

ATT-001 (Attendance Taking): att-001:read to view, att-001:write to mark and submit. ATT-002 (Late Arrival): att-002:write to process late arrivals. ATT-003 (Early Dismissal): att-003:write to process early departures. All enforced via @RequirePermission.

Produces → AttendanceModule with 2 services, 1 controller, ~8 REST endpoints. Kafka event emission. Swagger docs.
Depends on → Step 5 (SisModule for class roster), Cycle 0 (Kafka connection for events)
07 UI Shell & Design System
UI

Before building any feature pages, we build the application shell that every page lives inside. This is the layout, navigation, authentication context, data fetching layer, and design system components that all subsequent UI work reuses.

Components
  1. AuthProvider — React context that manages the JWT access token, refresh flow, and current user state. Wraps the entire app. Redirects to login if token is expired/missing.
  2. ApiClient — Configured fetch wrapper (or Axios) that automatically attaches the Bearer token, X-Tenant-Subdomain header, and handles 401 refresh. Used by all React Query hooks.
  3. AppLayout — The authenticated layout shell: sidebar navigation, top bar with user avatar and school name, main content area. Responsive — sidebar collapses to a hamburger on mobile.
  4. Sidebar — Navigation grouped by persona: teacher sees classes, attendance, gradebook. Parent sees children, attendance, messages. Admin sees everything. Driven by the user's permissions.
  5. Design tokens — Tailwind config extended with CampusOS brand palette, typography (DM Sans + display font), spacing scale, component-level tokens (card radius, shadow levels, status colours for attendance).
  6. Shared components — PageHeader, DataTable, StatusBadge, EmptyState, LoadingSpinner, Avatar, Modal, Toast notifications. Built with Tailwind, no external component library.
  7. React Query provider — TanStack Query configured with default options (stale time, retry, error handling). All API calls go through React Query hooks for caching and loading states.
Produces → A polished, authenticated app shell. Every subsequent page is a new route inside this layout. Login works via the /auth/dev-login endpoint initially.
Depends on → Cycle 0 (AuthModule, Next.js app exists)
08 Teacher Dashboard
UI

The first real page a teacher sees after logging in. Shows today's class schedule, quick attendance stats, and navigation to each class's attendance view.

Page: /dashboard
  1. Today's Classes — a list/grid of the teacher's classes for today. Each card shows: period number, course name, section, room, number of students, and attendance status (not started / in progress / submitted).
  2. Quick Stats — total students across all classes, overall attendance rate today, number of tardies, number of absences.
  3. Click a class card — navigates to /classes/:id/attendance for that class and today's date.
  4. Recent Activity — last few attendance submissions, recent absence requests from parents.
API Calls

GET /api/v1/classes/my — returns the teacher's assigned classes with today's attendance summary. This is a React Query hook: useMyClasses().

Produces → The teacher dashboard page. A teacher logs in and immediately sees their classes and attendance status.
Depends on → Step 7 (UI shell), Steps 5–6 (API endpoints), Step 4 (seed data)
09 Attendance Taking UI
UI

The core attendance workflow page. The teacher sees their class roster and marks attendance for the period. This is the most important screen in Cycle 1 — it needs to be fast, intuitive, and satisfying to use.

Page: /classes/:id/attendance
  1. Header — class name, period, date (with date picker for viewing other days), teacher name, submission status.
  2. Student roster — list of all enrolled students. Each row shows: student name, avatar/initials, current status (colour-coded: green Present, amber Tardy, red Absent), and a notes icon.
  3. Status toggle — tapping a student cycles through: Present → Tardy → Absent → Present. Or a dropdown/button group for explicit selection. Tardy shows a time picker. Absent shows a reason dropdown.
  4. Notes — expandable note field per student for tardy/absent reasons.
  5. Batch submit — "Submit Attendance" button that confirms all statuses for the period. Changes confirmation_status from PRE_POPULATED to CONFIRMED. Cannot be undone without admin override.
  6. Visual feedback — submit button shows count of exceptions ("Submit — 2 Tardy, 1 Absent"). Success toast after submission. Class card on dashboard updates to "Submitted" status.
UX Principles

Exception-based workflow: the default state is Present. The teacher only interacts with the 2–3 students who aren't present. A class of 25 with perfect attendance takes 2 taps: open the class, hit submit. A class with 2 tardies takes 4 taps: open, tap tardy, tap tardy, submit.

Produces → The attendance-taking page. A teacher can take attendance for a class in under 30 seconds.
Depends on → Step 8 (dashboard navigates here), Step 6 (attendance API)
10 Parent Dashboard & Attendance View
UI

The parent's view. David Chen logs in and sees Maya's attendance. Initially this is an in-app view — push notifications and email come in Cycle 3 (Communications).

Pages
  1. Parent Dashboard /dashboard — shows each child with a summary card: name, grade, today's attendance status, overall attendance rate. If Maya was marked tardy, a notification banner appears at the top: "Maya was marked tardy in Period 1 Algebra at 8:15 AM."
  2. Child Attendance /children/:id/attendance — calendar heatmap showing the month. Green days = fully present, amber = tardy, red = absent. Click a day to see per-period detail. Running stats: days present, tardy count, absence count, attendance percentage.
  3. Absence Request /children/:id/absence-request — form for submitting a same-day absence report or advance planned absence. Date picker, reason category dropdown, text explanation, optional document upload. Submits to POST /absence-requests.
Persona-Aware Routing

The /dashboard route renders different content based on the user's person_type: teachers see the teacher dashboard (Step 8), parents see the parent dashboard (this step). The AppLayout sidebar also changes. This is driven by the JWT payload's personId, which resolves to iam_person.person_type.

Produces → Parent dashboard with child attendance view and absence request. David Chen sees Maya's tardy immediately.
Depends on → Step 7 (UI shell), Step 6 (attendance API), Step 4 (seed data with parent-student links)
11 Vertical Slice Integration Test
Test

The end-to-end proof that Cycle 1 works as a complete vertical slice. This is the first CAT (Customer Acceptance Test) script — a manual walkthrough that exercises every layer.

Test Script
  1. Teacher login: Open browser to CampusOS. Log in as teacher@demo.campusos.dev via dev-login. Verify dashboard loads with 6 classes.
  2. View class roster: Click Period 1 Algebra. Verify 5–8 students shown. All marked Present (green). Verify student names match seed data.
  3. Mark tardy: Tap Maya Chen. Set status to Tardy. Enter note: "arrived 8:15." Verify Maya's row turns amber.
  4. Submit attendance: Click "Submit Attendance." Verify confirmation dialog shows "1 Tardy." Submit. Verify success toast. Verify class card on dashboard shows "Submitted."
  5. Verify database: Query sis_attendance_records for Maya in Period 1 today. Verify status=TARDY, confirmation_status=CONFIRMED.
  6. Parent login: Log in as parent@demo.campusos.dev. Verify parent dashboard loads. Verify notification shows "Maya was marked tardy."
  7. View attendance history: Click Maya → Attendance. Verify today shows amber for Period 1. Verify attendance percentage reflects the tardy.
  8. Submit absence request: Click "Report Absence." Select tomorrow. Category: Medical Appointment. Submit. Verify request appears with status PENDING.
  9. Admin verification: Log in as admin. Verify school-wide attendance dashboard shows correct stats. Verify absence request appears in queue.
Produces → A verified vertical slice. The first CampusOS feature is production-ready. Pattern established for Cycles 2–8.
Depends on → All previous steps (the complete Cycle 1 must be built)

Cycle 1 Completion Criteria

Exit Checklist
  1. Tenant schema: ~18 new SIS tables migrated across all tenant schemas.
  2. Seed data: 15 students, 10 guardians, 8 families, 6 classes, attendance pre-populated.
  3. SIS API: ~15 endpoints, all permission-protected, Swagger documented.
  4. Attendance API: ~8 endpoints including batch marking. Kafka events emitting.
  5. Teacher Dashboard: shows today's classes, click navigates to attendance.
  6. Attendance UI: roster view, status toggles, batch submit, under 30 seconds per class.
  7. Parent Dashboard: shows child attendance, tardy notification, absence request form.
  8. Vertical slice test: teacher marks tardy → parent sees it. All 9 test steps pass.
  9. Tests: unit tests for services, integration tests for API endpoints, coverage >90%.
  10. Git: all code committed, CI pipeline green.

Once Cycle 1 is complete, CampusOS is a product. It does one thing — attendance — but it does it end-to-end with real multi-tenant isolation, real IAM permissions, and real identity management. Cycle 2 (Classroom + Assignments + Grading) adds the next vertical slice on top of the same students and classes.
