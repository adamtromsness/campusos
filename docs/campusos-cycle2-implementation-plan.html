Cycle 2: Classroom, Assignments & Grading
Implementation Plan

The second vertical slice — teacher creates an assignment, student submits work, teacher grades it, parent sees the grade. The core teaching and learning workflow.
April 2026 · 10 Steps · ERD v11 · M21 Classroom

Plan Overview

Cycle 2 adds the M21 Classroom module — the teaching and learning domain. Where Cycle 1 gave teachers attendance, Cycle 2 gives them the tools they spend most of their day in: lesson planning, assignments, submissions, grading, and gradebook management. It builds directly on the SIS classes, students, and enrollments from Cycle 1.

Vertical Slice Deliverable

Teacher creates a homework assignment for Period 1 Algebra → student Maya Chen opens the assignment and submits her work → teacher reviews the submission and enters a grade → grade is published → gradebook snapshot updates asynchronously → parent David Chen sees Maya's current average on the parent dashboard.

What's In / What's Deferred

In scope: Assignment types, assignments, submissions, grades, gradebook snapshots, report cards, assignment categories with weighted grading, student progress notes. Functions TCH-001 through TCH-004.

Deferred to later cycles: Lesson planning (TCH-001 full — create the table but minimal API), quizzes & assessments (TCH-006 — question bank, auto-scoring), AI grading (cls_ai_grading_jobs — table created, service stubbed), lesson bank sharing (cls_lesson_bank_shares), video processing, AI lesson summaries, class set checkouts. These tables are created in the migration for schema completeness but their services and UI are future work.

Step-by-Step Build Plan

01 Classroom Schema — Lessons & Assignments
Schema

Create the first batch of M21 Classroom tables in a new tenant migration. These define the assignment structure: what kinds of assignments exist, the assignments themselves, and their questions/answer keys.

Tables (7)
  1. cls_lesson_types — school_id, name, icon, is_active. UNIQUE(school_id, name). Configurable lesson types.
  2. cls_lessons — class_id FK nullable (NULL for templates), teacher_id (soft FK to hr_employees), lesson_type_id FK, title, date nullable, duration_minutes, learning_objectives TEXT[], status ENUM(DRAFT,PUBLISHED,ARCHIVED,TEMPLATE), is_template BOOLEAN, bank_lesson_id FK nullable (provenance link). Table created for completeness; minimal API in this cycle.
  3. cls_assignment_types — school_id, name, weight_in_category NUMERIC(5,2), category ENUM(HOMEWORK,QUIZ,TEST,PROJECT,CLASSWORK). UNIQUE(school_id, name).
  4. cls_assignment_categories — class_id FK, name, weight NUMERIC(5,2). Weights must sum to 100 per class (app-layer validation). Enables weighted grade calculation.
  5. cls_assignments — class_id FK, lesson_id FK nullable, assignment_type_id FK, title, instructions, due_date TIMESTAMPTZ, max_points NUMERIC(6,2), grading_scale_id FK, is_ai_grading_enabled, is_extra_credit. INDEX(class_id, due_date).
  6. cls_assignment_questions — assignment_id FK, question_text, question_type ENUM(MULTIPLE_CHOICE,SHORT_ANSWER,ESSAY,TRUE_FALSE,FILE_UPLOAD), points NUMERIC(5,2), sort_order. INDEX(assignment_id, sort_order).
  7. cls_answer_key_entries — UNIQUE(question_id, option_index). correct_answer, explanation, is_correct.
Migration File

packages/database/prisma/tenant/migrations/005_cls_lessons_and_assignments.sql. Follows the same idempotent pattern as Cycle 1 migrations: CREATE TABLE IF NOT EXISTS, TEXT+CHECK for enums, soft FKs to platform schema.

Produces → 7 new tables in every tenant schema.
02 Classroom Schema — Submissions & Grading
Schema

The grading pipeline: student submissions, teacher grades, gradebook snapshots, report cards, progress notes, and the AI grading stub.

Tables (8)
  1. cls_submissions — UNIQUE(student_id, assignment_id). status ENUM(NOT_STARTED,IN_PROGRESS,SUBMITTED,GRADED,RETURNED), submitted_at TIMESTAMPTZ. Partial index on (assignment_id, submitted_at DESC) WHERE status='SUBMITTED'.
  2. cls_submission_question_grades — UNIQUE(submission_id, question_id). ai_suggested_points, ai_confidence, teacher_awarded_points, feedback. AI suggestions are read-only inputs to the teacher review UI.
  3. cls_ai_grading_jobs — submission_id FK, status ENUM(PENDING,RUNNING,COMPLETE,FAILED), ai_suggested_grade, ai_confidence, ai_reasoning, model_version, completed_at. AI must never write to cls_grades — teacher action only.
  4. cls_grades — UNIQUE(student_id, assignment_id). teacher_id FK, grade_value NUMERIC(6,2), letter_grade, feedback, graded_at, is_published BOOLEAN. Only teacher action creates or updates a confirmed grade.
  5. cls_gradebook_snapshots — UNIQUE(class_id, student_id, term_id). current_average NUMERIC(5,2), last_updated_at. Updated by async Kafka consumer with 30s debounce (ADR-010). Never updated synchronously inside a grade write transaction.
  6. cls_report_cards — student_id FK, class_id FK, term_id FK, status ENUM(DRAFT,PUBLISHED), published_at. UNIQUE(student_id, class_id, term_id).
  7. cls_report_card_entries — UNIQUE(report_card_id, subject). final_grade, grade_value, teacher_comments, effort_grade. Subject is intentional free-text, not a FK to sis_courses (report cards need labels that don't map 1:1 to courses).
  8. cls_student_progress_notes — UNIQUE(class_id, student_id, term_id). author_id FK (soft to hr_employees), note_text, overall_effort_rating, is_parent_visible, is_student_visible, published_at. One note per student per class per term.
Key Design Contracts

ADR-010 (gradebook snapshots): cls_gradebook_snapshots is never updated in the same transaction that writes a grade. A Kafka event (cls.grade.published) triggers an async consumer that debounces 30 seconds per (class_id, student_id) before recomputing current_average. This prevents gradebook recalculation from blocking grade entry.

AI boundary: cls_ai_grading_jobs and cls_submission_question_grades store AI suggestions. cls_grades stores only teacher-confirmed grades. AI must never write to cls_grades. The UI shows AI suggestions alongside teacher input but only teacher action persists a grade.

Produces → 8 new tables. Total Classroom tables: 15. Total tenant tables: ~38 (23 from Cycle 1 + 15 new).
Depends on → Step 1 (assignments must exist for submissions to reference)

  7. Progress notes: One progress note from James Rivera for Maya in Period 1 Algebra: "Excellent problem-solving skills. Consistently participates." is_parent_visible=true.

04 Classroom NestJS Module — Assignments
Code

Build the assignment management API. This is the teacher's primary workflow for creating and managing coursework.

Services & Endpoints
  1. AssignmentService — CRUD assignments. List by class with pagination. Filter by type, due date range, status. Create with questions and answer key in one transaction. Validate category weights sum to 100.
  2. AssignmentController — GET /classes/:id/assignments (list), GET /assignments/:id (detail with questions), POST /classes/:id/assignments (create), PATCH /assignments/:id (update), DELETE /assignments/:id (soft delete). All @RequirePermission('tch-002:read') or 'tch-002:write'.
  3. CategoryController — GET /classes/:id/categories, PUT /classes/:id/categories (bulk upsert with weight validation). @RequirePermission('tch-003:write').
Row-Level Authorization

Follow the pattern established in Cycle 1 review fixes: teachers can only manage assignments for their assigned classes (verified via sis_class_teachers). Students can view assignments for their enrolled classes only. Parents can view assignments for their linked children's classes.

Produces → ClassroomModule with AssignmentService, ~8 endpoints, all permission-protected with row-level auth.
Depends on → Step 3 (seed data), Cycle 1 (SisModule for class/student lookups)
05 Classroom NestJS Module — Submissions & Grading
Code

The grading pipeline: students submit work, teachers review and grade, grades are published, gradebook is updated.

Services & Endpoints
  1. SubmissionService — Create/update submission. Only the student themselves can submit (row-level: submission.student_id must match caller via sis_students). Track status transitions: NOT_STARTED → IN_PROGRESS → SUBMITTED → GRADED → RETURNED.
  2. GradeService — Enter grade for a submission. Only teachers assigned to the class can grade. Validate grade_value ≤ max_points. Publish grades (set is_published=true, emit Kafka event). Batch grade endpoint for grading a whole class at once.
  3. GradebookService — Get gradebook view for a class (all students × all assignments, with grades and current averages). Get student gradebook across all classes (for student/parent view). Compute weighted average from category weights.
  4. ProgressNoteService — Create/update progress notes. Publish to parent (set is_parent_visible=true, emit cls.progress_note.published event).
  5. Endpoints: POST /assignments/:id/submit (student submits), GET /assignments/:id/submissions (teacher sees all), GET /assignments/:id/submissions/mine (student sees own), POST /submissions/:id/grade (teacher grades), POST /classes/:id/grades/batch (batch grade), POST /grades/:id/publish (publish single), POST /classes/:id/grades/publish-all (publish all for class), GET /classes/:id/gradebook (teacher gradebook view), GET /students/:id/gradebook (student/parent view), POST /classes/:id/progress-notes, GET /students/:id/progress-notes.
Grading Rules

Grade value must be ≤ max_points (unless is_extra_credit on the assignment). Weighted average computed from cls_assignment_categories weights. Extra credit assignments excluded from the denominator. Only published grades count in the gradebook snapshot. Teachers can unpublish a grade (sets is_published=false, triggers snapshot recalculation).

Produces → SubmissionService, GradeService, GradebookService, ProgressNoteService, ~12 endpoints.
Depends on → Step 4 (assignments), Cycle 1 (students, enrollments)
06 Kafka Events & Gradebook Snapshot Worker
Code

The first Kafka consumer in the system. When a grade is published, an event triggers an async worker that recomputes the student's current average. This is the ADR-010 pattern: never update gradebook snapshots synchronously inside a grade transaction.

Events Emitted
  1. cls.grade.published — { studentId, classId, assignmentId, gradeValue, maxPoints, isExtraCredit, termId }. Emitted when a teacher publishes a grade or batch-publishes.
  2. cls.grade.unpublished — Same payload. Triggers snapshot recalculation.
  3. cls.submission.submitted — { studentId, assignmentId, submittedAt }. Future use: notify teacher.
  4. cls.progress_note.published — { studentId, classId, termId }. Future use: notify parent.
Gradebook Snapshot Consumer

Consumes cls.grade.published and cls.grade.unpublished. Uses a 30-second debounce per (class_id, student_id) — if multiple grades are published in rapid succession for the same student in the same class, only one recalculation runs. The consumer queries all published grades for the student in that class, applies category weights, computes current_average, and upserts the cls_gradebook_snapshots record.

This is the first Kafka consumer — it establishes the pattern (KafkaConsumerService, topic subscription, idempotency via platform_event_consumer_idempotency) that Cycle 3 (Communications) will follow for notification delivery.

Produces → 4 Kafka events, 1 consumer (gradebook snapshot worker), debounce logic, idempotency.
Depends on → Step 5 (grade publishing), Cycle 0 (Kafka infrastructure, event consumer idempotency table)
07 Teacher Assignments UI
UI

The teacher's assignment management interface. Extends the existing class view from Cycle 1 with a new "Assignments" tab alongside "Attendance."

Pages & Components
  1. Assignment list /classes/:id/assignments — table of all assignments for the class. Columns: title, type, due date, max points, submissions count, average grade. Sort by due date. Filter by type. "Create Assignment" button.
  2. Create/edit assignment /classes/:id/assignments/new — form with: title, type (dropdown from cls_assignment_types), category (dropdown from cls_assignment_categories), instructions (rich text or plain), due date/time picker, max points, extra credit toggle. Optional: add questions (for quizzes — question text, type, points, answer options).
  3. Category weight management — modal or inline editor showing categories with weight sliders/inputs. Validation: must sum to 100%. Affects grade calculation for all students in the class.
  4. Class view tabs — the class detail page now has tabs: Attendance | Assignments | Gradebook. Attendance is Cycle 1. Assignments is this step. Gradebook is Step 8.
Produces → Assignment list, create/edit form, category management. Teacher can create and manage assignments in the browser.
Depends on → Step 4 (assignment API), Cycle 1 UI (class view shell)
08 Teacher Grading UI
UI

The teacher's grading workflow. Review submissions, enter grades, and publish to students/parents.

Pages & Components
  1. Gradebook view /classes/:id/gradebook — spreadsheet-style grid. Rows = students, columns = assignments (grouped by category). Cells show grade/max (colour-coded: green ≥80%, amber 60–79%, red <60%). Right column shows current weighted average. Click a cell to enter/edit a grade inline.
  2. Submission review /assignments/:id/submissions — list of all student submissions. Status badges (Not Started / In Progress / Submitted / Graded / Returned). Click a submission to see student's work and enter grade + feedback.
  3. Grade entry — inline in the gradebook grid or in the submission detail. Grade value input, letter grade auto-calculated from grading scale, feedback textarea. Save immediately (optimistic UI).
  4. Publish controls — "Publish All Grades" button per assignment (batch). Individual publish toggle per grade. Published grades are visible to students and parents. Unpublish option available.
  5. Progress notes — accessible from the gradebook, per student. Write mid-term narrative, set effort rating, toggle parent visibility. Publish emits notification event.
Produces → Gradebook grid, submission review, grade entry, publish workflow. The core grading experience.
Depends on → Steps 5–6 (grading API + snapshot worker), Step 7 (assignments UI)
09 Student & Parent Grade Views
UI

The student and parent views of assignments, submissions, and grades. Extends the existing persona-aware dashboard from Cycle 1.

Student Pages
  1. My assignments /assignments — list of all assignments across all enrolled classes. Filter by class, due date, status. Upcoming due dates highlighted. Overdue assignments flagged red.
  2. Assignment detail & submit /assignments/:id — view instructions, questions (if any). Text area or file upload for submission. Submit button. Shows grade and feedback after grading.
  3. My grades /grades — gradebook summary across all classes. Per-class: current average, per-assignment grades. Matches the data in cls_gradebook_snapshots.
Parent Pages
  1. Child gradebook — integrated into the existing parent dashboard from Cycle 1. Below the attendance section, a "Grades" card showing per-class current averages. Click a class to see per-assignment breakdown.
  2. Progress notes — visible on the child detail page when is_parent_visible=true. Shows teacher name, effort rating, narrative text, date.
Row-Level Auth

Students see only their own submissions and grades. Parents see only their linked children's data. Teachers see only their assigned classes. Same patterns as Cycle 1.

Produces → Student assignment list, submission form, grade view. Parent gradebook integration. The vertical slice is now visible end-to-end.
Depends on → Steps 5 & 8 (grading API + UI), Cycle 1 UI (student/parent dashboards)
10 Vertical Slice Integration Test
Test

The end-to-end proof that the teaching and learning workflow works.

Test Script
  1. Teacher creates assignment: Log in as teacher. Navigate to Period 1 Algebra → Assignments tab. Create "Chapter 4 Homework" (Homework type, 25 points, due tomorrow). Verify it appears in the list.
  2. Student sees assignment: Log in as student (Maya). Navigate to Assignments. Verify "Chapter 4 Homework" appears with due date.
  3. Student submits: Open the assignment. Type response. Click Submit. Verify status changes to SUBMITTED.
  4. Teacher grades: Log in as teacher. Open Period 1 Algebra → Gradebook. Click Maya's "Chapter 4 Homework" cell. Enter grade: 22/25. Add feedback: "Good work, watch step 3." Save.
  5. Teacher publishes: Click "Publish All Grades" for Chapter 4 Homework. Verify Kafka event cls.grade.published emitted.
  6. Gradebook updates: Wait 30+ seconds for debounce. Verify cls_gradebook_snapshots for Maya in Period 1 Algebra has updated current_average reflecting the new grade.
  7. Student sees grade: Log in as Maya. Navigate to Grades. Verify Chapter 4 Homework shows 22/25 with feedback. Verify current average updated.
  8. Parent sees grade: Log in as David Chen. Verify parent dashboard shows updated average for Maya in Algebra. Click through to see per-assignment breakdown including the new grade.
  9. Permission denial: Verify student cannot grade, parent cannot submit, teacher cannot grade another teacher's class.
Produces → Verified vertical slice: create → submit → grade → publish → snapshot updates → parent sees. Pattern established for all future classroom features.
