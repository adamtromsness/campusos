Cycle 3: Communications
Implementation Plan

The cycle that closes the loop — events from attendance and grading finally reach parents and teachers as real notifications. Plus direct messaging, announcements, and content moderation. After this, CampusOS enters Phase 2: Test & Refine.
April 2026 · 11 Steps · ERD v11 · M40 Communications

Plan Overview

Cycle 3 is the final core cycle before the test-and-refine phase. It serves two purposes: first, it adds the M40 Communications module (messaging, announcements, notifications); second, it connects every Kafka event from Cycles 1 and 2 to actual user-facing notifications. When this cycle is complete, a parent no longer needs to manually check the dashboard to see that Maya was marked tardy — they receive an in-app notification, and the infrastructure for email and push is in place.

Vertical Slice Deliverable

Teacher marks Maya tardy in Period 1 → Kafka event fires → notification consumer picks it up → in-app notification appears on David Chen's parent dashboard with a badge count → David clicks the notification → he sees the attendance detail. Separately: James Rivera sends David a direct message about Maya's progress → David sees it in his inbox → David replies → the reply goes through content moderation → James sees the reply in his inbox.

What's In / What's Deferred

In scope: ADR-057 event envelope (carried from Cycle 2). Notification pipeline (queue, preferences, log, consumers for all Cycle 1+2 events). In-app messaging (threads, messages, read tracking, Redis unread counters). Announcements with audience fan-out. Three-tier content moderation (platform + district + building). In-app notification bell with badge count. Functions COM-001 through COM-004.

Deferred to later cycles: Emergency alerts (msg_emergency_alerts — requires dedicated always-on service per Architecture Review). Communication groups with dynamic membership. Digest batching (daily/weekly email summaries). Translation requests (AI pipeline). Retention policies and archival jobs. Actual email/SMS/push provider integration (Sendgrid, Twilio, FCM) — the queue and log are built, but delivery is stubbed for now. These tables are created in the migration for schema completeness.

Step-by-Step Build Plan

00 ADR-057 Event Envelope (Carry-Over)
Infrastructure

Implement the canonical Kafka event envelope before adding new consumers. This was deferred from Cycle 2 and is now a prerequisite — Cycle 3 adds multiple consumers that depend on a consistent event format.

Envelope Schema
  1. event_id — UUIDv7, unique per event. Already emitted as a header; moves into the envelope body.
  2. event_type — string matching the topic name (e.g. att.student.marked_tardy).
  3. event_version — integer, starts at 1. Increment on breaking payload changes.
  4. occurred_at — ISO timestamp of when the domain event happened.
  5. published_at — ISO timestamp of when the event was published to Kafka.
  6. tenant_id — school UUID. Already in headers; moves into envelope.
  7. source_module — string (e.g. attendance, classroom, communications).
  8. correlation_id — UUIDv7 for request tracing. Propagated from the HTTP request context.
  9. payload — the domain-specific event data (what is currently the entire message body).
Topic Naming

Topics follow {env}.{domain}.{entity}.{verb}. In dev: dev.att.student.marked_tardy. The env prefix is set via environment variable. All existing topic references in Cycle 1 and 2 producers/consumers are updated.

Migration

Refactor KafkaProducerService.emit() to accept domain payload and auto-wrap in the envelope. Update the Cycle 2 gradebook snapshot worker to read from the envelope's payload field. Update all Cycle 1 attendance event emitters. Backward compatibility: consumers that read headers continue to work during the migration.

Produces → Canonical event envelope. All existing producers migrated. Gradebook worker updated. Foundation for all Cycle 3 consumers.
01 Communications Schema — Messaging
Schema

The core messaging tables. Threads are partitioned by school (HASH 64 buckets per ADR-047) for inbox performance at scale. Messages are RANGE-partitioned by created_at monthly for retention management.

Tables (6)
  1. msg_thread_types — school_id, name, allowed_participant_roles TEXT[], is_system. UNIQUE(school_id, name). Examples: TEACHER_PARENT, CLASS_DISCUSSION, ADMIN_STAFF.
  2. msg_threads — HASH(school_id) with 64 buckets (ADR-047). school_id, thread_type_id FK, subject, created_by FK(platform_users), is_archived. Partition pruning eliminates 63/64 partitions on inbox queries.
  3. msg_thread_participants — UNIQUE(thread_id, platform_user_id). role ENUM(OWNER,PARTICIPANT,OBSERVER), joined_at, left_at. Determines who can read/write in a thread.
  4. msg_messages — RANGE(created_at) monthly via pg_partman, 24-month retention. thread_id, sender_id FK, body TEXT, is_edited, is_deleted (soft delete). school_id denormalised for partition-pruned queries. INDEX(thread_id, created_at DESC).
  5. msg_message_attachments — message_id FK, file_name, s3_key, content_type, file_size_bytes. Signed S3 URLs with 15-60 min expiry.
  6. msg_message_reads — UNIQUE(message_id, reader_id). read_at TIMESTAMPTZ. Unread count pattern: Redis HASH key inbox:{user_id}:{thread_id} stores count, incremented on msg.message.posted, cleared on thread open. Badge total = SUM of Redis HVALS. Never COUNT(*) on msg_messages.
Produces → 6 messaging tables with partitioning. Migration: 007_msg_messaging.sql.
02 Communications Schema — Notifications & Announcements
Schema

The notification pipeline and announcement system. The notification queue uses Redis SET NX for idempotency (not a DB UNIQUE constraint — avoids deadlocks during emergency fan-out).

Tables (7)
  1. msg_notification_queue — recipient_id FK, notification_type TEXT, payload JSONB, status ENUM(PENDING,SENT,FAILED,SKIPPED), idempotency_key TEXT, scheduled_for TIMESTAMPTZ. Idempotency via Redis SET NX. Partial index (status, scheduled_for) WHERE status='PENDING'.
  2. msg_notification_preferences — UNIQUE(platform_user_id, notification_type). channels TEXT[] (IN_APP, EMAIL, PUSH, SMS), is_enabled, quiet_hours_start TIME, quiet_hours_end TIME.
  3. msg_notification_log — RANGE partition by sent_at monthly. recipient_id, notification_type, channel ENUM(PUSH,EMAIL,SMS,IN_APP), status ENUM(SENT,DELIVERED,FAILED), provider_ref, sent_at.
  4. msg_announcements — school_id, author_id FK, title, body, publish_at, expires_at, audience_type ENUM(ALL_SCHOOL,CLASS,YEAR_GROUP,ROLE,CUSTOM), is_recurring, recurrence_rule (iCal RRULE). INDEX(school_id, publish_at DESC).
  5. msg_announcement_audiences — Pre-computed audience. INDEX(platform_user_id, delivery_status). Populated by announcement worker on publish. Eliminates real-time fan-out at send time.
  6. msg_announcement_reads — UNIQUE(announcement_id, reader_id). read_at.
  7. msg_alert_types — school_id, name, severity ENUM(INFO,WARNING,URGENT,EMERGENCY), default_channels TEXT[], requires_acknowledgement. Created for schema completeness; emergency alert service is deferred.
Produces → 7 tables. Migration: 008_msg_notifications_and_announcements.sql.
03 Communications Schema — Moderation & Support Tables
Schema

Content moderation, user blocks, admin audit, and platform-level notification infrastructure (push tokens, DLQ).

Tables (8)
  1. msg_moderation_policies — scope ENUM(PLATFORM,DISTRICT,BUILDING), scope_id, keywords TEXT[], keyword_action ENUM(BLOCK,FLAG_FOR_REVIEW,ESCALATE_TO_COUNSELLOR), sensitivity_threshold, escalation_rules JSONB, is_active. Three-tier: most restrictive action applies.
  2. msg_moderation_log — RANGE-partitioned by created_at monthly. message_id FK, policy_id FK, flag_type, matched_keywords, review_outcome, reviewed_at.
  3. msg_tags — school_id, name, color. UNIQUE(school_id, name). For thread tagging.
  4. msg_user_tags — UNIQUE(platform_user_id, tag_id).
  5. msg_user_blocks — UNIQUE(blocker_id, blocked_id). school_id.
  6. msg_admin_access_log — admin_id, thread_id, reason, accessed_at. FERPA audit trail for admin reading private threads.
  7. platform_push_tokens — platform_user_id FK, device_id, platform ENUM(IOS,ANDROID,WEB), token, is_active, last_seen_at. UNIQUE(platform_user_id, device_id). Platform schema table, added via Prisma migration.
  8. platform_dlq_messages — topic, partition, offset, payload JSONB, error_message, retry_count, first_failed_at, last_failed_at. Dead letter queue after 3 consumer failures. Platform schema table.
Platform Schema Changes

platform_push_tokens and platform_dlq_messages are platform-scoped (shared across tenants). Add them via Prisma migration to the platform schema, not the tenant migration pipeline.

Produces → 6 tenant tables + 2 platform tables. Total M40 tables in Cycle 3: ~21 of 29 (8 deferred: emergency alerts, digest batching, translation, retention/archival).

05 Notification Pipeline — Consumers & Queue
Code

The heart of Cycle 3: Kafka consumers that listen to events from Cycles 1, 2, and 3 and fan them out as user-facing notifications. This is where CampusOS stops being a system you check and becomes a system that tells you.

Notification Consumers
  1. AttendanceNotificationConsumer — consumes att.student.marked_tardy and att.student.marked_absent. Looks up the student's guardians via sis_student_guardians. For each guardian with portal_access=true and receives_reports=true, enqueues an in-app notification. Payload includes student name, class, period, status, time.
  2. GradeNotificationConsumer — consumes cls.grade.published. Enqueues notification to the student and their guardians. Payload includes assignment title, grade, class name.
  3. ProgressNoteNotificationConsumer — consumes cls.progress_note.published. Enqueues notification to guardians. Payload includes teacher name, class, term.
  4. AbsenceRequestNotificationConsumer — consumes att.absence.requested and att.absence.reviewed. Enqueues notification to the relevant teacher/admin (on request) or parent (on review).
  5. MessageNotificationConsumer — consumes msg.message.posted. For each thread participant except the sender, enqueues an in-app notification and increments the Redis unread counter.
NotificationQueueService

Accepts notification requests, checks the recipient's msg_notification_preferences (is this type enabled? what channels? are we in quiet hours?), applies idempotency (Redis SET NX on idempotency_key), and inserts into msg_notification_queue with status=PENDING.

NotificationDeliveryWorker

Polls msg_notification_queue for PENDING entries. For each, delivers via the appropriate channel. In Cycle 3, IN_APP is the only real delivery — it writes to a Redis sorted set (notifications:{user_id}) that the UI polls. EMAIL and PUSH channels are stubbed (log and mark as SENT, actual provider integration is Phase 3). Records delivery in msg_notification_log.

Produces → 5 Kafka consumers, NotificationQueueService, NotificationDeliveryWorker. The event loop from Cycles 1+2 is now closed.
Depends on → Step 0 (ADR-057 envelope), Steps 1-3 (notification tables), Cycle 1 (attendance events), Cycle 2 (grade events)
06 Messaging NestJS Module
Code

Direct messaging between users. Teacher-parent communication is the primary use case.

Services & Endpoints
  1. ThreadService — Create thread (validates participant roles against msg_thread_types.allowed_participant_roles). List user's threads (inbox). Archive/unarchive thread.
  2. MessageService — Post message to thread (only participants can post). Edit message (only author, within 15-min window). Soft-delete message. List messages in thread (paginated, newest first).
  3. UnreadCountService — Redis-backed. increment(userId, threadId) on new message, clear(userId, threadId) on thread open, getBadgeCount(userId) returns total unread across all threads. Never queries msg_messages for counts.
  4. ContentModerationInterceptor — NestJS interceptor on POST /messages. Before persisting, checks message body against active msg_moderation_policies (PLATFORM → DISTRICT → BUILDING, most restrictive wins). BLOCK: reject with 422 and reason. FLAG_FOR_REVIEW: persist but mark as flagged, log in msg_moderation_log. ESCALATE: persist, log, and emit msg.moderation.escalated event.
  5. Endpoints: GET /threads (inbox), POST /threads (create), GET /threads/:id/messages, POST /threads/:id/messages, PATCH /messages/:id (edit), DELETE /messages/:id (soft delete), POST /threads/:id/read (mark read), GET /notifications/unread-count (badge). All @RequirePermission('com-001:read') or 'com-001:write'.
Row-Level Authorization

Users can only see threads they're a participant in. Users can only post to threads they're a participant in with role OWNER or PARTICIPANT (not OBSERVER). Block list enforced: if blocker_id/blocked_id pair exists in msg_user_blocks, message creation is rejected.

Produces → MessagingModule with ThreadService, MessageService, UnreadCountService, ContentModerationInterceptor. ~8 endpoints.
07 Announcements NestJS Module
Code

School-wide, class-level, or role-targeted announcements with pre-computed audience fan-out.

Services & Endpoints
  1. AnnouncementService — Create announcement (draft or publish immediately). Schedule for future publish. List announcements for a user (filtered by audience membership). Mark as read.
  2. AudienceFanOutWorker — Kafka consumer on msg.announcement.published. Resolves the audience based on audience_type: ALL_SCHOOL queries all active platform_users for the school, CLASS queries enrollments + class teachers, YEAR_GROUP queries by grade_level, ROLE queries by IAM role. Inserts rows into msg_announcement_audiences. Enqueues notifications for each audience member.
  3. Endpoints: GET /announcements (list for current user), GET /announcements/:id, POST /announcements (create/publish), PATCH /announcements/:id (edit draft), POST /announcements/:id/read (mark read), GET /announcements/:id/stats (read count, delivery stats — admin only). @RequirePermission('com-002:read') or 'com-002:write'.
Produces → AnnouncementModule with AnnouncementService, AudienceFanOutWorker. ~6 endpoints.
08 Notification Bell & Inbox UI
UI

The notification bell in the top bar that shows a badge count and a dropdown of recent notifications. This is the user-facing payoff for the entire Kafka event pipeline.

Components
  1. NotificationBell — icon in the app top bar. Shows badge count (from Redis via GET /notifications/unread-count). Polls every 30 seconds (or WebSocket in future). Click opens dropdown.
  2. NotificationDropdown — shows last 10 notifications. Each row: icon (type-specific: bell for attendance, grade for classroom, envelope for message), title, timestamp, read/unread state. Click navigates to the source (attendance record, grade detail, message thread).
  3. NotificationPage /notifications — full notification history with pagination. Filter by type (attendance, grades, messages, announcements). Mark all as read.
  4. Deep links — each notification type links to the relevant page: tardy notification → /children/:id/attendance, grade published → /grades or /students/:id/gradebook, new message → /messages/:threadId.
Produces → Notification bell, dropdown, full notification page. The first time a user sees CampusOS proactively tell them something.
09 Messaging UI
UI

The messaging inbox and conversation view. The primary teacher-parent communication channel.

Pages
  1. Inbox /messages — list of threads, newest first. Each row: participant names/avatars, subject, last message preview, timestamp, unread badge. Click opens thread.
  2. Thread view /messages/:threadId — message list (oldest first, paginated). Each message: sender avatar, name, timestamp, body. Own messages styled right-aligned. Mark thread as read on open (clears Redis counter). Reply input at bottom.
  3. Compose /messages/new — select recipient(s), choose thread type, enter subject, write message. Recipient picker filters by allowed_participant_roles for the selected thread type (e.g. TEACHER_PARENT only shows teachers and parents).
  4. Moderation feedback — if a message is blocked by content moderation, the sender sees a clear error: "This message was not sent because it contains content that violates school policy." No detail about which keyword triggered it (prevents gaming).
Produces → Inbox, thread view, compose, moderation feedback. Teachers and parents can communicate directly.
10 Announcements UI
UI

The announcement feed and creation interface.

Pages
  1. Announcement feed /announcements — list of announcements targeted at the current user. Pinned/active at top, expired grayed out. Each shows: title, author, timestamp, audience badge (ALL SCHOOL / YOUR CLASS / etc.), read/unread state. Click to expand full body.
  2. Create announcement /announcements/new (admin/teacher only) — title, body (rich text), audience selector (ALL_SCHOOL, specific CLASS dropdown, YEAR_GROUP dropdown, ROLE dropdown), schedule or publish now, expiry date. Preview before publishing.
  3. Announcement stats (admin only) — per-announcement: total audience, read count, read %, delivery timeline chart.
Produces → Announcement feed, create form, stats dashboard. School-wide communication working.
11 Vertical Slice Integration Test
Test

The end-to-end proof that the notification loop works and messaging is functional.

Test Script
  1. Notification from attendance: Log in as teacher. Mark Maya tardy in Period 1. Log in as parent (David). Verify notification bell shows badge count. Click bell → see "Maya was marked tardy in Period 1 Algebra." Click notification → navigates to attendance detail.
  2. Notification from grading: Log in as teacher. Publish a grade for Maya. Log in as parent. Verify grade notification appears. Click → navigates to gradebook.
  3. Direct messaging: Log in as teacher. Navigate to Messages. Compose new message to David Chen (TEACHER_PARENT thread). Send. Log in as parent. Verify unread badge in inbox. Open thread. See teacher's message. Reply: "Thanks for letting me know."
  4. Content moderation: Log in as any user. Send a message containing a moderation keyword. Verify message is blocked with appropriate error. Verify msg_moderation_log has an entry.
  5. Announcements: Log in as admin. Create announcement "Early Dismissal Friday" targeting ALL_SCHOOL. Publish. Log in as teacher, parent, student. Verify all see the announcement in their feed. Mark as read. Verify read count increases in admin stats.
  6. Notification preferences: Log in as parent. Disable attendance notifications. Log in as teacher. Mark another student tardy. Verify David does NOT receive a notification (preference honored).
  7. Permission denial: Verify student cannot create announcements. Verify parent cannot access moderation log. Verify user cannot read threads they're not a participant in.
Produces → Verified notification loop + messaging + announcements + moderation. Phase 1 (Core) is complete. CampusOS is ready for Phase 2: Test & Refine.

Phase 1 Complete → Phase 2 Begins

After Cycle 3, CampusOS handles three complete school workflows end-to-end: attendance (teacher marks → parent notified), grading (teacher grades → gradebook updates → parent sees average), and communication (direct messaging + announcements + notifications). The platform enters Phase 2: Test & Refine — persona walkthroughs, UI design review, edge case testing, and creation of the UI design guide before expanding with Cycles 4–8.
