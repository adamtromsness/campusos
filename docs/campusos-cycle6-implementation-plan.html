Cycle 6: Enrollment & Payments
Implementation Plan

The admissions pipeline and family billing engine. A family applies, gets an offer, accepts, enrolls, receives an invoice, and pays — all within CampusOS. Completes the full parent journey from first contact to first tuition payment.
April 2026 · 12 Steps · ERD v11 · M81 Enrollment + M84 Payments (Core Subset)

Plan Overview

Cycle 6 spans two ERD modules: M81 (Enrollment) and M84 (Payments & Billing). The ERD defines 62 tables across these modules plus M83 (Finance/GL). This cycle implements the ~22 core tables that deliver the enrollment-to-payment pipeline. The GL, financial aid, lunch accounts, discount rules, and advanced billing features are deferred to later cycles.

The architecture follows the ERD's hard boundary: M81 Enrollment never writes to pay_ tables. The EnrollmentConfirmedWorker emits enr.student.enrolled, and the PaymentAccountWorker consumes it to create the family account. This event-driven decoupling means enrollment and payments can evolve independently.

Vertical Slice Deliverable

Admin opens an enrollment period for Fall 2026 → parent submits an application for a new student → admin reviews the application → admin issues an offer → parent accepts the offer → enr.student.enrolled fires → PaymentAccountWorker creates the family account and links the student → admin generates a tuition invoice from the fee schedule → parent views the invoice on their dashboard → parent pays via Stripe (stubbed) → payment recorded → ledger entry created → invoice status updates to PAID → parent sees receipt.

What's In / What's Deferred

In scope (M81 Enrollment): enr_enrollment_periods, enr_admission_streams, enr_intake_capacities, enr_capacity_summary, enr_applications, enr_application_screening_responses, enr_application_documents, enr_application_notes, enr_offers, enr_waitlist_entries. Functions STU-003 (core), STU-004 (withdrawal initiation only).

In scope (M84 Payments): pay_family_accounts, pay_family_account_students, pay_fee_categories, pay_fee_schedules, pay_invoices, pay_invoice_line_items, pay_payments, pay_ledger_entries, pay_refunds, pay_stripe_accounts, pay_payment_plans, pay_payment_plan_installments. Function FIN-001 (core).

Deferred: enr_tour_slots/bookings (campus tours). enr_reenrollment_windows (re-enrollment). enr_withdrawal_checklists (full withdrawal workflow — STU-004 advanced). pay_financial_aid_applications/awards (FIN-002). pay_lunch_accounts/transactions (food service billing — Cycle 15). pay_discount_rules, pay_late_payment_policies, pay_auto_invoice_rules (advanced billing automation). pay_credit_notes, pay_payment_reversals, pay_debt_writeoffs (advanced AR). pay_family_account_access, pay_payment_authorisation_requests (co-payer). pay_payment_allocations (multi-invoice payments). pay_ledger_balance_checkpoints (performance optimization). pay_invoice_generation_runs (batch invoicing). All of M83 Finance/GL (fin_ tables — separate cycle).

Stripe Integration (ADR-003)

Stripe Connect is the payment gateway. Each school gets a Stripe managed account (pay_stripe_accounts). Payment intents are created server-side; the frontend uses Stripe.js Elements for PCI-compliant card capture. In Cycle 6, the Stripe integration is stubbed — the pay_stripe_accounts table and the PaymentService accept Stripe references but don't make actual Stripe API calls. A development-mode flag lets the payment flow complete with a mock charge. Real Stripe wiring is a Phase 3 ops task once the billing workflow is validated.

Step-by-Step Build Plan

01 Enrollment Schema — Periods & Applications
Schema

The enrollment pipeline: when the school accepts applications, what intake streams exist, capacity per grade, and the application itself with screening responses and documents.

Tables (8)
  1. enr_enrollment_periods — school_id, academic_year_id FK(sis_academic_years), name TEXT, opens_at TIMESTAMPTZ, closes_at TIMESTAMPTZ, status ENUM(UPCOMING, OPEN, CLOSED), allows_mid_year_applications BOOLEAN DEFAULT false. INDEX(school_id, status).
  2. enr_admission_streams — Named parallel intake streams within an enrollment period (Standard Intake, Music Scholarship, SEND Specialist). enrollment_period_id FK, name TEXT, grade_level TEXT nullable, opens_at TIMESTAMPTZ, closes_at TIMESTAMPTZ, is_active BOOLEAN. UNIQUE(enrollment_period_id, name).
  3. enr_intake_capacities — Places available per grade per period. enrollment_period_id FK, stream_id FK nullable, grade_level TEXT NOT NULL, total_places INT, reserved_places INT DEFAULT 0. UNIQUE(enrollment_period_id, COALESCE(stream_id, nilUUID), grade_level).
  4. enr_capacity_summary — Materialized capacity dashboard. Updated on every application status change. enrollment_period_id FK, grade_level TEXT, total_places INT, reserved INT, applications_received INT, offers_issued INT, offers_accepted INT, waitlisted INT, available INT (computed). UNIQUE(enrollment_period_id, grade_level).
  5. enr_applications — school_id, enrollment_period_id FK, stream_id FK nullable, student_first_name TEXT, student_last_name TEXT, student_date_of_birth DATE, applying_for_grade TEXT, guardian_person_id UUID (soft to iam_person), guardian_email TEXT, guardian_phone TEXT, admission_type ENUM(NEW_STUDENT, TRANSFER, MID_YEAR_ADMISSION), status ENUM(DRAFT, SUBMITTED, UNDER_REVIEW, ACCEPTED, REJECTED, WAITLISTED, WITHDRAWN, ENROLLED), submitted_at TIMESTAMPTZ, reviewed_by UUID (soft), reviewed_at TIMESTAMPTZ. INDEX(school_id, enrollment_period_id, status).
  6. enr_application_screening_responses — application_id FK, question_key TEXT, response_value JSONB. UNIQUE(application_id, question_key). Flexible schema for school-configured screening questions.
  7. enr_application_documents — application_id FK, document_type TEXT, s3_key TEXT NOT NULL, uploaded_at TIMESTAMPTZ. Signed S3 URLs only.
  8. enr_application_notes — application_id FK, note_type ENUM(INTERVIEW_NOTES, ASSESSMENT_RESULT, STAFF_OBSERVATION, REFERENCE_CHECK, VISIT_NOTES, GENERAL), note_text TEXT NOT NULL, is_confidential BOOLEAN DEFAULT false, created_by UUID (soft). INDEX(application_id, created_at DESC).
Produces → 8 new tables. Migration: 018_enr_enrollment_and_applications.sql.
02 Enrollment Schema — Offers & Waitlist
Schema

The offer lifecycle and waitlist management. Offers are separate from application status — they model the offer lifecycle independently.

Tables (2)
  1. enr_offers — UNIQUE(application_id). school_id, offer_type ENUM(UNCONDITIONAL, CONDITIONAL), offer_conditions TEXT[] nullable, conditions_met BOOLEAN nullable, offer_letter_s3_key TEXT, issued_at TIMESTAMPTZ NOT NULL, response_deadline TIMESTAMPTZ NOT NULL, family_response ENUM(ACCEPTED, DECLINED, DEFERRED) nullable, family_responded_at TIMESTAMPTZ, deferral_target_year_id FK(sis_academic_years) nullable, status ENUM(ISSUED, ACCEPTED, DECLINED, EXPIRED, WITHDRAWN, CONDITIONS_NOT_MET) DEFAULT 'ISSUED'. INDEX(school_id, status). INDEX(response_deadline) WHERE status='ISSUED' for expiry sweep.
  2. enr_waitlist_entries — school_id, enrollment_period_id FK, application_id FK, grade_level TEXT NOT NULL, priority_score NUMERIC(5,2), position INT, status ENUM(ACTIVE, OFFERED, ENROLLED, EXPIRED, WITHDRAWN), added_at TIMESTAMPTZ, offered_at TIMESTAMPTZ. UNIQUE(enrollment_period_id, application_id). INDEX(school_id, enrollment_period_id, grade_level, position).
Offer Lifecycle

Application ACCEPTED → admin creates an offer (ISSUED) with a response deadline → parent receives notification → parent ACCEPTS/DECLINES/DEFERS → on ACCEPT: application status → ENROLLED + Kafka event enr.student.enrolled fires → SIS worker creates the student records + PaymentAccountWorker creates the family account. On DECLINED: offer status → DECLINED, next waitlisted applicant may be offered. EXPIRED: a sweep job checks offers past response_deadline and marks them EXPIRED.

Produces → 2 new tables. Migration: 019_enr_offers_and_waitlist.sql. Total enrollment tables: 10.
03 Payments Schema — Family Accounts & Fees
Schema

The billing foundation: family accounts that aggregate charges across students, fee categories and schedules that define what the school charges, and the Stripe Connect account for payment processing.

Tables (5)
  1. pay_family_accounts — school_id, account_holder_id UUID (soft to iam_person — the primary guardian), account_number TEXT, status ENUM(ACTIVE, SUSPENDED, CLOSED), payment_authorisation_policy ENUM(ACCOUNT_HOLDER_ONLY, ANY_AUTHORISED) DEFAULT 'ACCOUNT_HOLDER_ONLY'. UNIQUE(school_id, account_holder_id). Created automatically by the PaymentAccountWorker when a student enrolls.
  2. pay_family_account_students — UNIQUE(family_account_id, student_id). Links students to family accounts for billing. student_id FK(sis_students), added_by UUID (soft), added_at TIMESTAMPTZ.
  3. pay_fee_categories — school_id, name TEXT NOT NULL (e.g. "Tuition", "Registration", "Activity Fee", "Technology Fee"), is_active BOOLEAN. UNIQUE(school_id, name).
  4. pay_fee_schedules — school_id, academic_year_id FK, name TEXT, fee_category_id FK, grade_level TEXT nullable (null = all grades), amount NUMERIC(10,2) NOT NULL, is_recurring BOOLEAN, recurrence ENUM(ONE_TIME, MONTHLY, QUARTERLY, SEMESTER, ANNUAL) DEFAULT 'ANNUAL'. UNIQUE(school_id, academic_year_id, name).
  5. pay_stripe_accounts — school_id UUID UNIQUE, stripe_account_id TEXT NOT NULL UNIQUE, onboarding_complete BOOLEAN, created_at TIMESTAMPTZ. Stripe Connect managed account per school. Stubbed in Cycle 6.
Produces → 5 new tables. Migration: 020_pay_family_accounts_and_fees.sql.
04 Payments Schema — Invoices, Payments & Ledger
Schema

The billing engine: invoices with line items, payments with Stripe references, an immutable ledger, refunds, and payment plans for installment billing.

Tables (7)
  1. pay_invoices — school_id, family_account_id FK, title TEXT NOT NULL, total_amount NUMERIC(10,2), due_date DATE, status ENUM(DRAFT, SENT, PARTIAL, PAID, OVERDUE, CANCELLED), sent_at TIMESTAMPTZ. INDEX(school_id, status). INDEX(family_account_id, due_date).
  2. pay_invoice_line_items — invoice_id FK, description TEXT NOT NULL, quantity NUMERIC(6,2) DEFAULT 1, unit_price NUMERIC(10,2), total NUMERIC(10,2), fee_schedule_id FK nullable. INDEX(invoice_id).
  3. pay_payments — invoice_id FK, family_account_id FK, amount NUMERIC(10,2) NOT NULL, payment_method ENUM(CARD, BANK_TRANSFER, CASH, CHEQUE, WAIVER), stripe_payment_intent_id TEXT, status ENUM(PENDING, COMPLETED, FAILED, REFUNDED), paid_at TIMESTAMPTZ, receipt_s3_key TEXT. INDEX(family_account_id, paid_at DESC).
  4. pay_ledger_entries — IMMUTABLE (no UPDATE or DELETE). family_account_id FK, entry_type ENUM(CHARGE, PAYMENT, REFUND, CREDIT, ADJUSTMENT), amount NUMERIC(10,2) NOT NULL, reference_id UUID, description TEXT, created_at TIMESTAMPTZ. INDEX(family_account_id, created_at DESC). BRIN on created_at. RANGE partitioned by created_at annually (2025–2030). Balance = SUM(amount) WHERE family_account_id=$1, cached in Redis ledger:balance:{id} TTL=30s.
  5. pay_refunds — payment_id FK, family_account_id FK, amount NUMERIC(10,2) NOT NULL, refund_category ENUM(OVERPAYMENT, WITHDRAWAL, PROGRAMME_CANCELLED, ERROR_CORRECTION, GOODWILL, OTHER), reason TEXT NOT NULL, stripe_refund_id TEXT, status ENUM(PENDING, COMPLETED, FAILED), authorised_by UUID (soft), authorised_at TIMESTAMPTZ. INDEX(family_account_id, created_at DESC).
  6. pay_payment_plans — family_account_id FK, invoice_id FK, total_amount NUMERIC(10,2), installment_count INT, frequency ENUM(MONTHLY, QUARTERLY), start_date DATE, status ENUM(ACTIVE, COMPLETED, DEFAULTED, CANCELLED). UNIQUE(invoice_id). Auto-generates installment rows on creation.
  7. pay_payment_plan_installments — plan_id FK, installment_number INT, amount NUMERIC(10,2), due_date DATE, status ENUM(UPCOMING, DUE, PAID, OVERDUE), payment_id FK nullable (links to the payment that covered this installment). UNIQUE(plan_id, installment_number). INDEX(due_date, status).
Ledger Integrity

pay_ledger_entries is IMMUTABLE — no UPDATE, no DELETE. Every financial action (invoice sent, payment received, refund issued) creates a new ledger entry. Balance is always derivable from SUM(amount). Corrections use ADJUSTMENT entries, never mutations. This is the accounting foundation that M83 Finance/GL will consume via Kafka events in a future cycle.

Produces → 7 new tables. Migration: 021_pay_invoices_payments_ledger.sql. Total payment tables: 12. Total Cycle 6 tables: 22.

06 Enrollment NestJS Module
Code

The admissions pipeline API. Application submission, review, offer management, and enrollment confirmation.

Services & Endpoints
  1. EnrollmentPeriodService — GET /enrollment-periods, GET /enrollment-periods/:id (with streams + capacities), POST /enrollment-periods, PATCH /enrollment-periods/:id (open/close). Admin only for writes (stu-003:admin).
  2. ApplicationService — POST /applications (parent submits — validates period is OPEN), GET /applications (admin: all, parent: own), GET /applications/:id, PATCH /applications/:id (admin reviews — status transitions), POST /applications/:id/notes (admin adds review notes). Kafka: enr.application.submitted, enr.application.status_changed.
  3. OfferService — POST /applications/:id/offer (admin issues offer), PATCH /offers/:id/respond (parent accepts/declines/defers), GET /offers (admin: all, parent: own). On ACCEPT: application status → ENROLLED, emits enr.student.enrolled with student info + guardian info + grade level. The EnrollmentConfirmedWorker consumes this to create sis_students + sis_enrollments + sis_guardians records if they don't exist (idempotent).
  4. WaitlistService — GET /waitlist (by period + grade), PATCH /waitlist/:id/offer (admin moves waitlisted applicant to offer stage). Auto-manages position ordering.
  5. CapacitySummaryService — Recomputes enr_capacity_summary on every application/offer status change. Called internally, not via endpoint. Idempotent UPSERT.
Row-Level Authorization

Parents can only see their own applications (matched via guardian_person_id = actor.personId). Admins see all applications in the enrollment period. Teachers do not have enrollment access. Students cannot submit applications (parents do).

Produces → EnrollmentModule with 5 services, ~14 endpoints + 3 Kafka events.
07 Payments NestJS Module
Code

The billing engine: family accounts, invoice generation, payment processing (Stripe stubbed), and the immutable ledger.

Services & Endpoints
  1. FamilyAccountService — GET /family-accounts (admin: all, parent: own), GET /family-accounts/:id (with balance from ledger), GET /family-accounts/:id/students. Admin can manually create accounts; normally created by PaymentAccountWorker.
  2. FeeScheduleService — GET /fee-schedules, POST /fee-schedules, PATCH /fee-schedules/:id, GET /fee-categories, POST /fee-categories. Admin only (fin-001:admin).
  3. InvoiceService — POST /invoices (admin creates from fee schedule or ad hoc), GET /invoices (admin: all, parent: own family), GET /invoices/:id (with line items), PATCH /invoices/:id/send (status DRAFT → SENT, emits pay.invoice.created, creates CHARGE ledger entry), PATCH /invoices/:id/cancel. POST /invoices/generate-from-schedule (bulk: create invoices for all enrolled families from a fee schedule).
  4. PaymentService — POST /invoices/:id/pay (creates payment intent — Stripe stubbed in dev mode, marks COMPLETED immediately with mock intent ID), GET /payments (parent: own, admin: all), GET /payments/:id. On COMPLETED: creates PAYMENT ledger entry, updates invoice status (PARTIAL or PAID based on running total vs total_amount). Emits pay.payment.received.
  5. LedgerService — GET /family-accounts/:id/ledger (paginated, newest first), GET /family-accounts/:id/balance (SUM with Redis cache). Read-only — entries are created by other services, never directly. Balance = SUM(amount) cached in Redis ledger:balance:{accountId} TTL=30s, invalidated on write.
  6. RefundService — POST /payments/:id/refund (admin authorizes), GET /refunds. Creates REFUND ledger entry. Emits pay.refund.issued.
  7. PaymentPlanService — POST /invoices/:id/payment-plan (creates plan + auto-generates installment rows), GET /payment-plans/:id (with installments). Installments link to payments when paid.
  8. PaymentAccountWorker — Kafka consumer on enr.student.enrolled. Creates pay_family_accounts row (if not exists for the guardian) + pay_family_account_students link. Uses claim-after-success idempotency.
Ledger Immutability

pay_ledger_entries has no UPDATE or DELETE service methods. Every financial action creates a new row. CHARGE entries have positive amounts. PAYMENT entries have negative amounts (reduce balance). REFUND entries have positive amounts (increase balance back). Balance = SUM(amount) should equal the outstanding amount across all invoices. A future nightly reconciliation job can verify this invariant.

Produces → PaymentsModule with 7 services + 1 Kafka consumer, ~18 endpoints + 3 Kafka events.

12 Vertical Slice Integration Test
Test

The full enrollment-to-payment pipeline, proving the complete parent journey works.

Test Script
  1. Open enrollment: Admin creates an enrollment period for Fall 2026. Verify it appears as OPEN.
  2. Parent applies: Parent submits an application for a new student (new guardian, not David Chen). Verify application status = SUBMITTED. Verify capacity summary updates.
  3. Admin reviews: Admin views the application, adds a note, changes status to UNDER_REVIEW then ACCEPTED.
  4. Admin issues offer: Admin creates an UNCONDITIONAL offer with a 14-day deadline. Verify parent receives notification.
  5. Parent accepts: Parent responds ACCEPTED. Verify application status → ENROLLED. Verify enr.student.enrolled Kafka event fires.
  6. Student records created: Verify sis_students, sis_enrollments, and sis_guardians records were created by the EnrollmentConfirmedWorker (or equivalent).
  7. Family account created: Verify PaymentAccountWorker consumed enr.student.enrolled and created pay_family_accounts + pay_family_account_students link.
  8. Invoice generated: Admin generates a tuition invoice from the fee schedule for the new family. Verify invoice with correct line items. Admin sends the invoice. Verify CHARGE ledger entry.
  9. Parent pays: Parent views the invoice on their billing dashboard. Clicks Pay. Enters amount (full). Stubbed payment completes. Verify PAYMENT ledger entry. Verify invoice status → PAID. Verify pay.payment.received event.
  10. Balance check: Verify family account balance = $0 (all paid). Verify ledger entries sum to 0.
  11. Partial payment test: Using the existing Chen family's $400 tech fee invoice: parent pays $200. Verify invoice status → PARTIAL. Pay remaining $200. Verify → PAID.
  12. Permission denials: Teacher cannot access admissions. Student cannot submit applications. Parent cannot see other families' invoices. Teacher cannot issue refunds.
Produces → Verified enrollment-to-payment pipeline. Complete parent journey from application to paid invoice.
