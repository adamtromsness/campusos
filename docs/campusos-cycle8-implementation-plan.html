Cycle 8: Service Tickets
Implementation Plan

Shared reactive ticketing for IT, facilities, and HR support. SLA tracking, vendor management, problem linking, and a knowledge base. The last module of Wave 1 — completes the Core Operations platform.
May 2026 · 10 Steps · ERD v11 · M60 Service Tickets (13 tables — 10 core in scope)

Plan Overview

Cycle 8 builds the M60 Service Tickets module — a shared reactive ticketing engine that IT, facilities, and HR support all use. This is NOT a facilities management system (that's M65, a future cycle). M60 handles reactive requests: "my projector is broken," "Room 204 needs a new light bulb," "I can't log in." The module integrates with Cycle 7's Task Worker (ticket assignment creates a task on the assignee's to-do list) and the Approval Workflows engine (ticket escalation can route through an approval chain).

The ERD defines 13 M60 tables. This cycle implements the 10 core tables that deliver the ticket lifecycle. The remaining 3 (tkt_ticket_asset_links, tkt_vendor_updates, tkt_knowledge_articles) are deferred because they depend on M62 IT Infrastructure (asset registry) and vendor portal access, neither of which exists yet.

Vertical Slice Deliverable

Teacher submits a ticket: "Projector not working in Room 101" under IT → Hardware → auto-assigned to the IT admin based on subcategory rules → SLA clock starts (4h response / 24h resolution for HIGH priority) → IT admin sees the ticket in their queue, adds an internal comment ("Ordered replacement lamp") → reassigns to facilities vendor → vendor reference recorded → admin resolves the ticket → requester gets a notification → ticket auto-closes after 48h if requester doesn't reopen → SLA metrics show response and resolution times on the admin dashboard.

What's In / What's Deferred

In scope: tkt_categories (hierarchical), tkt_subcategories (with auto-assignment rules), tkt_sla_policies (per category + priority), tkt_vendors (external vendor registry), tkt_tickets (full lifecycle with SLA tracking), tkt_ticket_comments (public + internal), tkt_ticket_attachments, tkt_ticket_tags, tkt_ticket_activity (immutable audit log), tkt_problems + tkt_problem_tickets (root-cause grouping). Functions IT-001, FAC-001.

Deferred: tkt_ticket_asset_links (requires M62 tech_assets — IT Infrastructure cycle). tkt_vendor_updates (requires platform_vendor_accounts — vendor portal). tkt_knowledge_articles (knowledge base — future polish). SLA breach cron worker (schema ready, cron deferred to ops). Ticket auto-close after 48h in RESOLVED state (documented, deferred). Email-to-ticket inbound (Phase 3 ops). CSAT survey on ticket close (Phase 3 ops).

Integration with Cycles 4–7

Task Worker (Cycle 7): When a ticket is assigned, the system emits tkt.ticket.assigned. A new auto-task rule (seeded in Step 3) translates this into a TODO task on the assignee's to-do list: "Resolve ticket: Projector not working in Room 101 — SLA: 24h." When the ticket is resolved, the task auto-completes.

Notifications (Cycle 3): Ticket lifecycle events (submitted, assigned, commented, resolved, closed) fire Kafka events that the notification pipeline consumes to inform the requester and assignee.

Rooms (Cycle 5): Tickets can reference a sch_rooms location via a soft location_id column, so "Room 101 projector" links to the scheduling room for context.

HR Employees (Cycle 4): Internal assignees reference hr_employees. Vendor assignees reference tkt_vendors. A ticket is assigned to one or the other, never both (CHECK constraint).

Step-by-Step Build Plan

01 Ticket Schema — Categories, SLA, Vendors
Schema

The configuration layer: what kinds of tickets exist, how fast they should be resolved, and who can fix them externally.

Tables (4)
  1. tkt_categories — Hierarchical ticket categories. school_id, name TEXT NOT NULL, parent_category_id FK(tkt_categories) nullable (top-level when null), icon TEXT nullable, is_active BOOLEAN DEFAULT true. UNIQUE(school_id, name). Examples: IT (parent) → Hardware, Software, Network, Account Access. Facilities → Electrical, Plumbing, HVAC, Cleaning. HR Support → Payroll, Benefits.
  2. tkt_subcategories — Leaf-level classification with auto-assignment rules. category_id FK, name TEXT NOT NULL, default_assignee_id UUID FK(hr_employees) nullable (auto-assign when set), auto_assign_to_role TEXT nullable (fallback: resolve via IAM role like the workflow engine). UNIQUE(category_id, name).
  3. tkt_sla_policies — Per-(category, priority) SLA targets. school_id, category_id FK, priority ENUM(LOW, MEDIUM, HIGH, CRITICAL), response_hours INT NOT NULL, resolution_hours INT NOT NULL. UNIQUE(school_id, category_id, priority). The SLA clock starts at ticket creation; first_response_at on tkt_tickets records when an assignee first responds; resolved_at records resolution. SLA breach = now() - created_at > resolution_hours.
  4. tkt_vendors — External vendor registry. school_id, vendor_name TEXT NOT NULL, vendor_type ENUM(IT_REPAIR, FACILITIES_MAINTENANCE, CLEANING, ELECTRICAL, PLUMBING, HVAC, SECURITY, GROUNDS, OTHER), contact_name TEXT, contact_email TEXT, contact_phone TEXT, website TEXT, is_preferred BOOLEAN DEFAULT false, notes TEXT, is_active BOOLEAN DEFAULT true. UNIQUE(school_id, vendor_name).
Produces → 4 new tables. Migration: 028_tkt_categories_sla_vendors.sql.
02 Ticket Schema — Tickets, Comments, Activity
Schema

The operational layer: the ticket itself, its comment thread, attachment storage, tags, immutable activity log, and the problem management tables for root-cause grouping.

Tables (7)
  1. tkt_tickets — school_id, category_id FK, subcategory_id FK nullable, requester_id UUID (soft to platform_users), assignee_id UUID FK(hr_employees) nullable, vendor_id UUID FK(tkt_vendors) nullable, vendor_reference TEXT nullable, vendor_assigned_at TIMESTAMPTZ nullable, title TEXT NOT NULL, description TEXT, priority ENUM(LOW, MEDIUM, HIGH, CRITICAL) DEFAULT 'MEDIUM', status ENUM(OPEN, IN_PROGRESS, VENDOR_ASSIGNED, PENDING_REQUESTER, RESOLVED, CLOSED, CANCELLED), sla_policy_id FK nullable, location_id UUID nullable (soft to sch_rooms), first_response_at TIMESTAMPTZ, resolved_at TIMESTAMPTZ, closed_at TIMESTAMPTZ. CHECK: NOT (assignee_id IS NOT NULL AND vendor_id IS NOT NULL) — internal OR vendor, never both. INDEX(school_id, status, created_at DESC). INDEX(assignee_id, status). INDEX(vendor_id, status).
  2. tkt_ticket_comments — ticket_id FK, author_id UUID (soft to platform_users), body TEXT NOT NULL, is_internal BOOLEAN DEFAULT false (internal = staff-only, hidden from requester), created_at TIMESTAMPTZ. INDEX(ticket_id, created_at ASC).
  3. tkt_ticket_attachments — ticket_id FK, s3_key TEXT NOT NULL, filename TEXT, uploaded_by UUID (soft), uploaded_at TIMESTAMPTZ.
  4. tkt_ticket_tags — ticket_id FK, tag TEXT NOT NULL. UNIQUE(ticket_id, tag). Free-form tags for filtering.
  5. tkt_ticket_activity — IMMUTABLE audit log. ticket_id FK, actor_id UUID (soft), activity_type ENUM(STATUS_CHANGE, REASSIGNMENT, COMMENT, ATTACHMENT, ESCALATION, VENDOR_ASSIGNMENT, SLA_BREACH), metadata JSONB (captures before/after for status changes, assignee changes, etc.), created_at TIMESTAMPTZ. INDEX(ticket_id, created_at ASC). No UPDATE, no DELETE.
  6. tkt_problems — Root-cause grouping for related tickets. school_id, title TEXT NOT NULL, description TEXT NOT NULL, category_id FK, status ENUM(OPEN, INVESTIGATING, KNOWN_ERROR, RESOLVED), root_cause TEXT nullable (set when KNOWN_ERROR or RESOLVED), resolution TEXT nullable, workaround TEXT nullable, assigned_to_id UUID FK(hr_employees) nullable, vendor_id UUID FK(tkt_vendors) nullable, created_by UUID (soft), resolved_at TIMESTAMPTZ. INDEX(school_id, status). On RESOLVED: batch-update all linked tickets still OPEN/IN_PROGRESS to RESOLVED.
  7. tkt_problem_tickets — Link table. problem_id FK, ticket_id FK. UNIQUE(problem_id, ticket_id). INDEX(ticket_id).
Produces → 7 new tables. Migration: 029_tkt_tickets_and_activity.sql. Total Cycle 8 tables: 11 (ERD says 13; tkt_ticket_asset_links + tkt_vendor_updates deferred).

04 Ticket NestJS Module — Core CRUD + Lifecycle
Code

The ticket lifecycle API. Submit, assign, reassign, escalate to vendor, resolve, close, cancel. SLA clock is computed, not stored — the service calculates breach status from created_at + sla_policy.resolution_hours vs now().

Services & Endpoints
  1. TicketService — POST /tickets (submit — auto-assigns if subcategory has default_assignee_id or auto_assign_to_role; auto-links SLA policy by category + priority; emits tkt.ticket.submitted). GET /tickets (requester: own, assignee: assigned, admin: all; filters: status, priority, category, assignee, vendor, dateRange). GET /tickets/:id. PATCH /tickets/:id/assign (admin assigns to internal employee; clears vendor; records first_response_at if first assignment; emits tkt.ticket.assigned). PATCH /tickets/:id/assign-vendor (admin assigns to vendor; clears internal assignee; records vendor_reference + vendor_assigned_at; status → VENDOR_ASSIGNED). PATCH /tickets/:id/resolve (assignee or admin; sets resolved_at; emits tkt.ticket.resolved). PATCH /tickets/:id/close (admin or requester; sets closed_at). PATCH /tickets/:id/reopen (requester reopens a RESOLVED ticket; clears resolved_at). PATCH /tickets/:id/cancel (requester or admin). All lifecycle transitions use SELECT FOR UPDATE inside executeInTenantTransaction per the convention.
  2. CategoryService — GET /ticket-categories (tree with subcategories), POST /ticket-categories (admin), PATCH /ticket-categories/:id (admin). POST /ticket-subcategories, PATCH /ticket-subcategories/:id.
  3. SlaService — GET /ticket-sla (full SLA matrix for the school), POST /ticket-sla (admin creates/updates). computeSlaBreach(ticket) internal helper returning {isResponseBreached, isResolutionBreached, responseRemaining, resolutionRemaining}.
  4. VendorService — GET /ticket-vendors, POST /ticket-vendors (admin), PATCH /ticket-vendors/:id (admin).
SLA Clock

SLA is computed, not stored as a countdown. response_breached = first_response_at IS NULL AND (now() - created_at) > response_hours. resolution_breached = resolved_at IS NULL AND (now() - created_at) > resolution_hours. The admin dashboard shows these computed values. A future SLA breach cron can emit tkt.sla.breached for alerting.

Produces → TicketModule with 4 services, ~18 endpoints + 3 Kafka events.
05 Ticket NestJS Module — Comments, Activity, Problems
Code
Services & Endpoints
  1. CommentService — GET /tickets/:id/comments (requester sees public only; staff sees all; is_internal filtered server-side). POST /tickets/:id/comments (staff can set is_internal=true; requesters always public). Records first_response_at on the ticket if this is the first staff comment. Emits tkt.ticket.commented. Creates an activity log entry.
  2. ActivityService — GET /tickets/:id/activity (immutable timeline — read-only). All lifecycle events auto-create activity rows via a private recordActivity() helper called from TicketService and CommentService. No direct write endpoint.
  3. ProblemService — GET /problems (admin), GET /problems/:id (with linked tickets), POST /problems (admin creates from one or more ticket IDs), PATCH /problems/:id (update status, root_cause, resolution, workaround), POST /problems/:id/link (link additional tickets), PATCH /problems/:id/resolve (batch-resolves all linked tickets still OPEN/IN_PROGRESS — each ticket gets a RESOLVED status + resolved_at + activity log entry + tkt.ticket.resolved emit).
Produces → 3 more services, ~10 endpoints. Total ticket endpoints: ~28.
06 Ticket Notification Consumer
Code

Connects the ticket lifecycle to the Cycle 3 notification pipeline and the Cycle 7 task engine.

Components
  1. TicketNotificationConsumer — Kafka consumer on tkt.ticket.submitted, tkt.ticket.assigned, tkt.ticket.commented, tkt.ticket.resolved. On submitted: notifies school admins (same pattern as absence-request notification). On assigned: notifies the assignee. On commented (public): notifies the requester. On resolved: notifies the requester with "Your ticket has been resolved."
  2. Auto-task rule integration: The tkt.ticket.assigned rule from Step 3 feeds the existing Task Worker (Cycle 7). When a ticket is assigned, the worker creates a TODO task on the assignee's list. When the ticket is resolved, the service emits tkt.ticket.resolved — a new auto-task condition on this topic marks the linked task DONE (via source_ref_id matching).
Produces → 1 Kafka consumer + auto-task rule wiring.
07 Helpdesk UI — Submit + My Tickets
UI

The staff-facing ticket submission and tracking interface. A new "Helpdesk" app tile on the home page for anyone with it-001:read or fac-001:read.

Pages
  1. Submit Ticket /helpdesk/new — category picker (hierarchical dropdown: IT → Hardware), subcategory, title (required), description (rich text), priority selector (default MEDIUM), optional location dropdown (populated from sch_rooms), optional file attachments. Submit creates the ticket and navigates to the detail page.
  2. My Tickets /helpdesk — list of tickets the user submitted OR is assigned to. Filter chips: Open / In Progress / Resolved / All. Each row: ticket number, title, category, priority pill, status pill, SLA indicator (green/amber/red based on time remaining), assigned to, created date. Sort by created_at desc.
  3. Ticket Detail /helpdesk/:id — header with title, priority, status, SLA countdown, assigned to, category/subcategory, location. Comment thread (public comments for everyone, internal comments for staff only with amber "Internal" badge). Reply input with internal toggle for staff. Requester sees Reopen button on RESOLVED tickets, Cancel button on OPEN. Activity timeline at the bottom showing all status changes and reassignments.
Produces → "Helpdesk" app tile, ticket submission, my tickets, ticket detail.
08 Helpdesk Admin UI — Queue + Dashboard
UI

The IT/facilities admin's operational command center.

Pages
  1. Admin Ticket Queue /helpdesk/admin — all tickets in the school. Multi-filter: status, priority, category, assignee, vendor, date range, SLA status (breached/at risk/healthy). Bulk actions: assign, change priority, close. Per-row: assign button opens a modal with employee picker (from hr_employees) or vendor picker (from tkt_vendors with preferred vendors first). SLA indicator on each row.
  2. SLA Dashboard /helpdesk/admin/sla — overview stats: open tickets by priority, average response time, average resolution time, SLA compliance % (tickets resolved within SLA / total resolved). Breakdown by category. Breached tickets highlighted red at the top. Time-series chart of ticket volume + resolution rate (computed from activity log).
  3. Category Management /helpdesk/admin/categories — tree view of categories with subcategories. Add/edit/deactivate. Auto-assignment rule editor per subcategory (employee picker or role selector). Admin only (it-001:admin or fac-001:admin).
  4. Vendor Management /helpdesk/admin/vendors — list of vendors with type, contact, preferred badge. Add/edit/deactivate. Admin only.
Produces → Admin queue, SLA dashboard, category management, vendor management.
09 Problem Management UI
UI

Root-cause analysis for related tickets. An admin groups multiple tickets under a single problem, documents the root cause, and resolves them in batch.

Pages
  1. Problems List /helpdesk/admin/problems — all problems with status, linked ticket count, assigned to. Filter by status.
  2. Problem Detail /helpdesk/admin/problems/:id — title, description, category, status timeline (OPEN → INVESTIGATING → KNOWN_ERROR → RESOLVED). Root cause + resolution + workaround fields (editable). Linked tickets list with status. "Link more tickets" button (ticket search modal). "Resolve problem" button — sets root_cause + resolution (required), batch-resolves all linked open tickets, emits per-ticket events.
  3. Create from ticket — on the ticket detail page, admin sees "Create problem from this ticket" button that pre-populates the problem with the ticket's category and links it.
Produces → Problem management for root-cause grouping + batch resolution.

Test
Test Script
