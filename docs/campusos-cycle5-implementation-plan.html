Cycle 5: Scheduling & Calendar
Implementation Plan

Bell schedules, timetable slots, school calendar, room bookings, daily coverage, and substitute management. Consumes the hr.leave.coverage_needed event from Cycle 4 to auto-flag classes that need coverage when a teacher takes leave.
April 2026 · 10 Steps · ERD v11 · M22 Academic Scheduling (Core Subset)

Plan Overview

Cycle 5 builds the M22 Academic Scheduling module — the operational backbone that connects everything. When a teacher takes leave (Cycle 4), the schedule knows which classes need coverage. When attendance is taken (Cycle 1), the schedule defines what period it is. When a parent views their child's timetable, this module provides the data. The full ERD has 27 scheduling tables; this cycle implements the 12 core tables that deliver the daily operations workflow.

Vertical Slice Deliverable

Admin configures the bell schedule (8 periods, M-F) → assigns timetable slots (Rivera teaches Algebra in Room 101, Period 1) → Rivera's Cycle 1 class now has a room + time → Rivera submits a sick leave request (Cycle 4) → admin approves → hr.leave.coverage_needed fires → CoverageConsumer creates coverage-needed records for Rivera's classes on the leave dates → admin assigns a substitute → substitute notification fires → school calendar shows the coverage for that day.

What's In / What's Deferred

In scope: sch_bell_schedules, sch_periods, sch_rooms, sch_timetable_slots (with teacher + room double-booking EXCLUSION constraints), sch_room_bookings, sch_room_change_requests, sch_calendar_events, sch_calendar_day_overrides, sch_substitution_timetable, sch_coverage_requests. Plus the CoverageConsumer for hr.leave.coverage_needed. Functions SCH-001, SCH-003, SCH-004, SCH-005.

Deferred to Cycle 5b or later: sch_rotation_cycles + sch_rotation_calendar (A/B day schedules — ADR-053 advanced scheduling). sch_scheduling_requests + sch_scheduling_candidates + sch_scheduling_candidate_slots (automated constraint solver). sch_exam_sessions + sch_exam_session_rooms + sch_exam_seatings + sch_exam_invigilator_assignments (exam scheduling). sch_coteaching_arrangements (co-teaching). sch_pull_out_interventions (specialist pull-outs). sch_cross_school_staff_assignments (district staff). sch_cover_arrangements + sch_cover_arrangement_classes + sch_cover_split_students (complex cover beyond simple substitution). SCH-002 (Course Selection) — depends on enrollment module.

Step-by-Step Build Plan

01 Scheduling Schema — Bell Schedules & Rooms
Schema

The foundation: what periods exist in a school day, and what rooms are available to schedule into.

Tables (3)
  1. sch_bell_schedules — school_id, name TEXT NOT NULL, schedule_type ENUM(STANDARD, EARLY_DISMISSAL, ASSEMBLY, EXAM, CUSTOM), is_default BOOLEAN DEFAULT false. UNIQUE(school_id, name). A school might have: "Standard Day", "Early Dismissal", "Exam Schedule".
  2. sch_periods — bell_schedule_id FK, name TEXT NOT NULL (e.g. "Period 1", "Lunch", "Advisory"), day_of_week SMALLINT (0=Mon..6=Sun, nullable for rotation schedules), start_time TIME NOT NULL, end_time TIME NOT NULL, period_type ENUM(LESSON, BREAK, LUNCH, REGISTRATION, ASSEMBLY). CHECK: start_time < end_time. UNIQUE(bell_schedule_id, COALESCE(day_of_week, -1), start_time). Rotation-day column exists in ERD but deferred (requires sch_rotation_cycles).
  3. sch_rooms — school_id, name TEXT NOT NULL, capacity INT, room_type ENUM(CLASSROOM, LAB, GYM, HALL, LIBRARY, OFFICE, OUTDOOR), has_projector BOOLEAN, has_av BOOLEAN, floor TEXT, building TEXT, is_active BOOLEAN DEFAULT true, fac_space_id UUID (DISPLAY-ONLY soft ref to future fac_spaces). UNIQUE(school_id, name).
Produces → 3 new tables. Migration: 015_sch_bell_schedules_and_rooms.sql.
02 Scheduling Schema — Timetable & Bookings
Schema

The timetable is the master schedule — which class meets in which room with which teacher during which period. The EXCLUSION constraints enforce no double-booking at the database level.

Tables (3)
  1. sch_timetable_slots — school_id, class_id UUID FK(sis_classes), period_id UUID FK(sch_periods), teacher_id UUID FK(hr_employees) nullable, room_id UUID FK(sch_rooms), effective_from DATE NOT NULL, effective_to DATE. UNIQUE(class_id, period_id, effective_from). Two EXCLUSION constraints using btree_gist: (1) Teacher double-booking: EXCLUDE USING gist (teacher_id WITH =, period_id WITH =, daterange(effective_from, COALESCE(effective_to,'infinity'),'[)') WITH &&). (2) Room double-booking: EXCLUDE USING gist (room_id WITH =, period_id WITH =, daterange(effective_from, COALESCE(effective_to,'infinity'),'[)') WITH &&). Emits sch.timetable.updated on INSERT/UPDATE.
  2. sch_room_bookings — school_id, room_id FK, booked_by UUID FK (soft to hr_employees), booking_purpose TEXT, start_at TIMESTAMPTZ, end_at TIMESTAMPTZ, status ENUM(CONFIRMED, CANCELLED). Application checks sch_timetable_slots before confirming — room cannot be double-booked against a timetable slot.
  3. sch_room_change_requests — timetable_slot_id FK, requested_by UUID FK(hr_employees), current_room_id FK, requested_room_id FK nullable (null = "any available"), request_date DATE NOT NULL, reason TEXT NOT NULL, status ENUM(PENDING, APPROVED, REJECTED, AUTO_APPROVED). INDEX(school_id, status, request_date).
EXCLUSION Constraints — btree_gist

The timetable uses PostgreSQL EXCLUSION constraints with the btree_gist extension. This requires CREATE EXTENSION IF NOT EXISTS btree_gist at the top of the migration. The constraints prevent: a teacher being assigned to two classes in the same period at the same time, and a room being assigned to two classes in the same period at the same time. These are schema-enforced invariants — the application layer cannot violate them even with a bug.

Produces → 3 new tables with EXCLUSION constraints. Migration: 016_sch_timetable_and_bookings.sql. Requires btree_gist extension.
Depends on → Step 1 (sch_periods, sch_rooms), Cycle 1 (sis_classes), Cycle 4 (hr_employees)
03 Scheduling Schema — Calendar & Coverage
Schema

The school calendar, day-type overrides, coverage request tracking, and substitution timetable. The coverage table is what the hr.leave.coverage_needed consumer writes to.

Tables (4)
  1. sch_calendar_events — school_id, title TEXT NOT NULL, description TEXT, event_type ENUM(HOLIDAY, PROFESSIONAL_DEVELOPMENT, EARLY_DISMISSAL, ASSEMBLY, EXAM_PERIOD, PARENT_EVENT, FIELD_TRIP, CUSTOM), start_date DATE NOT NULL, end_date DATE NOT NULL, all_day BOOLEAN DEFAULT true, start_time TIME, end_time TIME, bell_schedule_id FK nullable (override bell schedule for this day), affects_attendance BOOLEAN DEFAULT false (when true, attendance pre-population skips this day), is_published BOOLEAN DEFAULT false, created_by UUID (soft). INDEX(school_id, start_date, end_date).
  2. sch_calendar_day_overrides — school_id, override_date DATE NOT NULL, bell_schedule_id FK (override the default bell schedule for this date), is_school_day BOOLEAN DEFAULT true (false = no classes, e.g. snow day), reason TEXT. UNIQUE(school_id, override_date). Overrides take precedence over the default bell schedule and calendar events.
  3. sch_coverage_requests — school_id, timetable_slot_id FK, absent_teacher_id FK(hr_employees), leave_request_id UUID (soft to hr_leave_requests), coverage_date DATE NOT NULL, status ENUM(OPEN, ASSIGNED, COVERED, CANCELLED), assigned_substitute_id FK(hr_employees) nullable, assigned_at TIMESTAMPTZ, notes TEXT. UNIQUE(timetable_slot_id, coverage_date). INDEX(school_id, coverage_date, status). Created automatically by the CoverageConsumer when a leave request is approved.
  4. sch_substitution_timetable — original_slot_id FK(sch_timetable_slots), effective_date DATE NOT NULL, substitute_id UUID FK(hr_employees), room_id FK(sch_rooms), coverage_request_id FK(sch_coverage_requests). UNIQUE(original_slot_id, effective_date). Created when a substitute is assigned to a coverage request. The substitute sees this slot in their schedule for that day.
Coverage Flow

When a teacher's leave is approved (Cycle 4), the hr.leave.coverage_needed Kafka event fires with the employee's affected class IDs and date range. The CoverageConsumer in this cycle reads the event, queries sch_timetable_slots for the teacher's classes on those dates, and creates one sch_coverage_requests row per (slot, date) pair with status=OPEN. Admin then assigns a substitute via the coverage board UI, which creates the sch_substitution_timetable row and fires sch.coverage.assigned.

Produces → 4 new tables. Migration: 017_sch_calendar_and_coverage.sql. Total scheduling tables: 10.
Depends on → Steps 1–2, Cycle 4 (hr_employees, hr_leave_requests)

05 Scheduling NestJS Module — Timetable & Rooms
Code

The scheduling backbone: bell schedule management, timetable CRUD with conflict detection, and room management with booking.

Services & Endpoints
  1. BellScheduleService — GET /bell-schedules, GET /bell-schedules/:id (with periods), POST /bell-schedules, PATCH /bell-schedules/:id. POST /bell-schedules/:id/periods (bulk upsert periods). Admin only for writes (sch-001:admin). Read for all staff (sch-001:read).
  2. TimetableService — GET /timetable (full school timetable, filterable by class, teacher, room, day), GET /timetable/teacher/:employeeId (teacher's schedule), GET /timetable/class/:classId (class schedule), GET /timetable/room/:roomId (room schedule), POST /timetable/slots (create slot — the EXCLUSION constraint catches double-booking at the DB level; the service wraps the constraint violation in a friendly 409 Conflict response), PATCH /timetable/slots/:id, DELETE /timetable/slots/:id. Emits sch.timetable.updated on writes.
  3. RoomService — GET /rooms (list with availability for a given date/period), GET /rooms/:id, POST /rooms, PATCH /rooms/:id. Admin only for writes (sch-005:admin).
  4. RoomBookingService — GET /room-bookings (filter by room, date range), POST /room-bookings (checks against timetable slots before confirming), PATCH /room-bookings/:id/cancel. @RequirePermission('sch-005:write').
  5. RoomChangeRequestService — POST /room-change-requests (teacher submits), GET /room-change-requests (admin queue), PATCH /room-change-requests/:id/approve (creates a one-day timetable override), PATCH /room-change-requests/:id/reject. @RequirePermission('sch-005:write').
Conflict Handling

When POST /timetable/slots tries to double-book a teacher or room, PostgreSQL's EXCLUSION constraint fires error code 23P01 (exclusion violation). The service catches this and returns 409 Conflict with a message like "Teacher James Rivera is already scheduled for Period 1 during this date range" or "Room 101 is already booked for Period 1". The app layer doesn't need to re-implement the conflict check — the database is the authority.

Produces → SchedulingModule with 5 services, ~18 endpoints.
Depends on → Step 4 (seed data), Cycle 1 (SIS classes), Cycle 4 (hr_employees)
06 Scheduling NestJS Module — Calendar & Coverage
Code

The school calendar and the coverage workflow. This step lands the CoverageConsumer that finally connects Cycle 4's leave system to Cycle 5's scheduling.

Services & Endpoints
  1. CalendarService — GET /calendar (events for a date range, filterable by type), GET /calendar/:id, POST /calendar, PATCH /calendar/:id, DELETE /calendar/:id. GET /calendar/day/:date (returns the effective bell schedule for a specific date: checks day overrides first, then calendar events, then default). @RequirePermission('sch-003:read') or 'sch-003:write'.
  2. DayOverrideService — GET /calendar/overrides, POST /calendar/overrides (e.g. declare a snow day), DELETE /calendar/overrides/:date. Admin only (sch-003:admin).
  3. CoverageService — GET /coverage (daily coverage board — all OPEN + ASSIGNED requests for a date), GET /coverage/:id, PATCH /coverage/:id/assign (assign substitute — creates substitution timetable row, fires sch.coverage.assigned), PATCH /coverage/:id/cancel. @RequirePermission('sch-004:read') or 'sch-004:write'.
  4. CoverageConsumer — Kafka consumer on hr.leave.coverage_needed (group: coverage-consumer). For each affected class in the event payload, creates an sch_coverage_requests row with status=OPEN. Uses the deterministic event_id from Cycle 4's REVIEW fix (the consumer's idempotency claim catches redeliveries). Fires sch.coverage.needed with the list of open slots for the admin dashboard.
  5. SubstitutionService — GET /substitutions (for a date range), GET /substitutions/teacher/:employeeId (a substitute's assignments). Read-only — substitution rows are created by the coverage assignment workflow, not directly.
Kafka Events
  1. Consumes: hr.leave.coverage_needed (from Cycle 4) — creates coverage requests.
  2. Emits: sch.timetable.updated (on timetable changes), sch.coverage.needed (when coverage requests are created), sch.coverage.assigned (when a substitute is assigned — notifies the substitute + the absent teacher).
Produces → CalendarService, CoverageService, CoverageConsumer, SubstitutionService, DayOverrideService. ~12 endpoints + 1 Kafka consumer + 3 event types.
07 Schedule & Timetable UI
UI

Admin-facing schedule management. A new "Schedule" app tile on the home page.

Pages
  1. Bell Schedule Editor /schedule/bell-schedules — list of bell schedules. Click one to see its periods in a timeline/grid. Add/edit periods. Mark one as default. Admin only (sch-001:admin).
  2. Timetable Grid /schedule/timetable — week-view grid: periods as rows, days as columns. Each cell shows class name + teacher + room. Color-coded by department. Filter by teacher, room, or class. Click a cell to edit (admin) or view detail (teacher). The EXCLUSION constraint means the grid can never show overlaps.
  3. Room Management /schedule/rooms — list of rooms with type, capacity, amenities. Click a room to see its schedule for the week (which classes use it when). Create/edit rooms. Admin only for writes.
  4. Room Bookings /schedule/room-bookings — calendar view of room bookings. Teachers can book rooms for free periods. Availability check prevents double-booking against the timetable.
Produces → Schedule app tile, bell schedule editor, timetable grid, room management, room bookings.
08 Calendar & Coverage UI
UI

The school calendar and the daily coverage board — the "air traffic control" for school operations.

Pages
  1. School Calendar /calendar — month/week view showing holidays, PD days, events, early dismissals. Color-coded by event type. Published events visible to all; draft events visible to admin only. A new "Calendar" app tile on the home page for everyone with sch-003:read.
  2. Daily Coverage Board /schedule/coverage — shows today's coverage status. Each row: period, class, absent teacher, status (OPEN/ASSIGNED/COVERED), assigned substitute. OPEN rows highlighted red. Admin can click OPEN → assign a substitute from a dropdown of available employees (those not already scheduled for that period). Assigning fires the Kafka event and sends a notification to the substitute. Admin only for writes (sch-004:write).
  3. Coverage History /schedule/coverage/history — filter by date range, teacher, substitute. Shows past coverage assignments for reporting.
Produces → Calendar app tile, school calendar, daily coverage board, coverage history.
09 Teacher & Parent Schedule Views
UI

The non-admin schedule views. Teachers see their own timetable; parents see their child's timetable.

Pages
  1. My Schedule (Teacher) — integrated into the existing Classes view. A "My Schedule" tab shows the teacher's week-view timetable: period, class, room, with any substitution overrides highlighted. Also shows upcoming coverage assignments if the teacher is assigned as a substitute for another teacher.
  2. Child's Timetable (Parent) — integrated into the existing "My Children" view. Each child card gains a "Schedule" link showing their weekly class schedule: period, class name, teacher, room. Substitutions shown with the substitute teacher's name highlighted.
  3. Student's Timetable (Student) — integrated into "My Classes" with a "My Schedule" tab showing the same weekly view.
  4. Room Availability Checker — on the room bookings page, a quick "check availability" widget: select a date + period → see which rooms are free. Available to anyone with sch-005:read.
Produces → Teacher/parent/student schedule views integrated into existing navigation.
10 Vertical Slice Integration Test
Test

End-to-end proof that the scheduling module connects the leave system to the coverage workflow.

Test Script
  1. Verify timetable: GET the school timetable. Verify Rivera's 6 classes appear with correct periods, rooms, and times.
  2. Verify teacher schedule: Log in as teacher. View "My Schedule." Verify all 6 periods show with correct rooms.
  3. Verify parent view: Log in as parent. View Maya's timetable. Verify her enrolled classes appear with correct schedule.
  4. Double-booking prevention: As admin, try to create a timetable slot that conflicts with Rivera's existing Period 1. Verify 409 Conflict with a clear message.
  5. Leave → coverage flow: Verify the seeded coverage request exists (from Cycle 4's leave approval). Verify status=ASSIGNED with Park as substitute. Verify the substitution timetable row exists.
  6. Live leave → coverage: As Rivera, submit a new 1-day leave request. As admin, approve it. Wait for hr.leave.coverage_needed Kafka event. Verify CoverageConsumer creates OPEN coverage requests for Rivera's classes on that date.
  7. Assign substitute: As admin, open the coverage board. Assign Park to one of the OPEN slots. Verify sch.coverage.assigned fires. Verify Park receives a notification.
  8. Calendar: View the school calendar. Verify Spring Break, PD day, and events appear. Verify the snow day override shows as a non-school day.
  9. Room booking: As teacher, book Main Hall for a free period. Verify the booking appears. Try to book a room that's in use during that period — verify conflict prevention.
  10. Permission denials: Parent cannot access timetable admin. Student cannot book rooms. Teacher cannot assign substitutes.
Produces → Verified scheduling vertical slice. Leave → coverage → substitute assignment flow working end-to-end.
