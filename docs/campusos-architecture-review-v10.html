1. Overall System Architecture

1.1 Architectural Approach: Modular Monolith

✓ Recommendation Start with a well-structured modular monolith with strong domain boundaries enforced at the DB and application layers. Extract high-load or specialized services (video processing, AI inference, notifications) as standalone microservices only when justified by scale or team ownership needs. The 32-module ERD maps directly to this decomposition strategy.

A modular monolith means a single deployable unit divided internally into cohesive, loosely-coupled modules with explicit boundaries. Each module owns its domain logic, its table prefix, and its Kafka event topics. Modules communicate via well-defined internal APIs or events — never by reading another module's tables directly (ADR-028: cross-module reference taxonomy).

1.3 Service Decomposition — Extract From Day One

Six services have fundamentally different infrastructure or safety requirements and must be separate from the monolith from day one:

Video Processing Service — GPU workloads, long-running jobs, S3 integration. Cannot share a JVM heap with OLTP traffic.
AI Inference Service — Model serving, rate limits, cost management, PII masking middleware. Isolated behind the internal AI Gateway.
Notification / Push Service — Fan-out to thousands of recipients, WebSocket connections, push gateway integrations. Redis pub/sub backbone. Handles volume and delivery reliability for routine notifications.
Scheduling Solver Service — Stateless external constraint solver (OR-Tools or equivalent). Per ADR-056, the solver is invoked by SchedulingWorker and returns complete candidate solution sets. It has no persistent state and no runtime connection to the application layer. Needs explicit per-request timeout, maximum concurrent jobs per school, and defined failure mode when timeout is reached.
Transport Dispatch Service — Manages driver WebSocket sessions (trn_driver_sessions), processes real-time run messages (trn_run_messages), updates route manifests, and emits dispatch events during active route runs. The driver app maintains a long-lived bidirectional connection from a moving vehicle — this connection lifecycle is architecturally distinct from the Notification Service. Safety implication: a bus driver losing their WebSocket connection during a run needs reconnection logic that is independent of API load from other modules. Shares the database but has its own connection management and deployment unit.
Emergency Alert Service — Handles emergency.alert.issued events with no lag tolerance. Needs: a dedicated always-on process, reserved Kafka consumer priority, and a direct WebSocket/SMS/push fan-out path that bypasses the normal notification queue. Critically, the emergency execution path must be designed to operate under degraded conditions — it must function with degraded DB (local fallback store for active incident state), degraded Kafka (direct publish channel bypass), and degraded auth (pre-validated emergency tokens with short TTL cached at service startup). This service must have zero shared hot-path dependencies with the main monolith API. See Section 18 — Operational Risk Governance for the minimal dependency execution path specification.

1.4 Cross-Module Communication (ADR-028)

All cross-module data access is governed by a three-category reference taxonomy defined in ADR-028:

HARD FK Database-enforced foreign key. Used for references within the same module or to shared platform infrastructure. Guarantees referential integrity at the DB layer.
SOFT INTEGRITY Application-layer reference registered in platform_reference_health. Used for cross-module domain references where a DB FK would create a compile-time module dependency. Validated by a background health monitor.
DISPLAY-ONLY No enforcement. Column comment required. Used for advisory links (e.g., sch_rooms.fac_space_id → fac_spaces.id) where the reference is informational only and not operationally critical.

1.5 Module Boundary Enforcement Architecture

Policy without enforcement erodes. The following three-tier boundary enforcement strategy ensures the modular monolith cannot silently become a distributed big ball of mud:

Tier 1 — Separate PostgreSQL Schemas for High-Risk Domains Three module groups are placed in separate PostgreSQL schemas within each tenant, providing DB-level enforcement where a boundary violation causes the most damage: hlth schema — Health and SEND data. The classroom module must never read hlth_* tables directly — it reads only the sis_student_active_accommodations read model (ADR-030). Separate schema means a missing explicit cross-schema grant makes an illegal read a query error, not a code review miss. fin schema — Finance and GL data. M81 must never write to fin_* tables directly — only M83 PostingService has write grants, via the event-driven posting contract (ADR-058). The GL audit trail is protected at the DB layer. dpo schema — Compliance and GDPR data. Only the DPO worker and compliance admin role hold table-level access grants. No other module can read or write dpo_* tables regardless of application code. Rationale for limiting to three schemas: 32 separate schemas × every tenant = tens of thousands of schemas, making Flyway migrations and PgBouncer connection management operationally untenable. The three selected domains are those where a boundary violation causes irreversible compliance or financial damage.
Tier 2 — DB Role Separation for All Other Modules For all modules not in a separate schema (sis_, cls_, hr_, pay_, msg_, tkt_, trn_, sch_, ext_, fds_, fac_, lib_, etc.), DB-level enforcement is achieved through role-based grants rather than schema separation: Each module has a dedicated PostgreSQL role (e.g., role_sis, role_cls, role_hr). Each role is granted SELECT/INSERT/UPDATE only on its own prefix tables. No module role has blanket access to the full tenant schema. Example: role_cls holds grants on cls_* tables only. A ClassroomRepository running as role_cls that constructs a query joining to sis_students will receive a permission denied error at the DB level — not a code review comment. Cross-module reads that are legitimately needed (e.g., classroom reading student enrollment status) go through the owning module's API or a designated read model — never through a direct table join from another module's role.
Tier 3 — Application Layer and CI Static Analysis DB enforcement catches runtime violations. CI enforcement catches them at compile time: Repository scoping: each module's repository classes are strictly scoped to their own prefix. No repository class may reference a table name outside its declared module prefix. This is enforced by package visibility rules (Java JPMS or equivalent) — cross-module repository injection is a compile error. CI static analysis: a custom lint rule (implementable in one day as a ArchUnit / SonarQube rule or equivalent) scans all repository and query classes and rejects any that directly reference a table name outside their module prefix. Cross-module access must go through an interface — not a raw query string. Code review gate: any PR touching more than one module prefix requires a second reviewer who explicitly signs off on the cross-module access pattern and confirms it is the correct ADR-028 category (SOFT INTEGRITY vs DISPLAY-ONLY). This three-tier strategy means a boundary violation must defeat: DB role grants, application package visibility, CI static analysis, and code review — four independent gates rather than one.
⚠ Risk Boundary enforcement is the most fragile rule in a modular monolith — developers will bypass it under performance pressure, and 'just one join' compounds. The three-tier strategy above must be implemented from the first migration and the first repository class. Retrofitting schema separation or role grants after domain tables exist is extremely disruptive.

2. Multi-Tenancy Architecture

2.1 Strategy: Schema-per-Tenant

✓ Recommendation Schema-per-tenant isolation. Each school gets its own PostgreSQL schema. All tenant tables are structurally identical; the connection search_path changes per request. A shared platform schema holds cross-tenant data: tenants, billing, global config, IAM infrastructure, and substitutes/volunteers.
Strategy Isolation Recommendation
Separate DB per tenant Highest — but very high cost and operational overhead Enterprise/HIPAA-level only
Schema per tenant High — recommended ✓ CampusOS architecture
Row-level (shared schema) Lowest — high cross-tenant leak risk Avoided for regulated data

2.2 Platform Schema vs Tenant Schema

The platform schema contains all cross-tenant data and is accessible from every tenant schema via search_path:

Cross-tenant identity: iam_person (HASH-partitioned 32 buckets), platform_users (HASH-partitioned by email domain), platform_students (HASH-partitioned). These exist at platform level because a substitute, volunteer, or guardian may have relationships with multiple schools.
Auth infrastructure: iam_identity_provider, iam_federated_identity, iam_scope, iam_scope_type, iam_role_assignment, iam_effective_access_cache.
Operational infrastructure: platform_tenant_routing, platform_connection_pool_config, platform_tenant_quotas, platform_import_jobs, audit_log, platform_legal_holds.

- Tenant schemas hold all domain module tables (sis_, cls_, hr_, pay_, etc.).

- For multi-school organisations (districts/MATs), cross-school entities like sis_families use SOFT INTEGRITY organisation_id references, not cross-schema JOINs (ADR-020).

2.3 Tenant Routing and Scaling

TenantRoutingDataSource — resolves the DataSource based on tenant context at request time. Reads from platform_tenant_routing cached in Redis (ADR-042, 500ms TTL). Falls back to DB on cache miss.
Connection pooling — PgBouncer in TRANSACTION mode (required for schema-per-tenant). Per-tenant pool sizes governed by platform_connection_pool_config (ADR-044).
Tenant tiers — SMALL / MEDIUM / LARGE / ENTERPRISE map to infrastructure profiles: API rate limits, connection pool sizes, storage quotas, AI token daily limits, Kafka consumer isolation mode (ADR-041–046).
Noisy-neighbor detection — platform_tenant_usage_metrics tracks hourly usage. NoisyNeighborDetector flags tenants at 2× rolling median and triggers throttle/isolate actions (ADR-045).
Cluster rebalance — Large/Enterprise tenants can be migrated to dedicated RDS instances via logical replication with is_frozen write-gate (ADR-031).
⚠ Risk Never store tenant-identifying data only in JWT claims without server-side validation. Always re-verify tenant membership from the database on sensitive operations. The is_frozen gate must be respected by all application layers during tenant rebalance.

3. IAM Subsystem Architecture (ADR-036, ADR-055)

The IAM subsystem was fully designed in ADR-036 and completed with ADR-055. It supersedes the original RBAC description. All 21 IAM tables live in the platform schema.

3.1 Identity Model — Three Distinct Concepts

iam_person The universal human identity record. One row per real-world person. Holds name, date of birth, national_id_hash, and person_type. HASH-partitioned 32 buckets. This is the FK target for all domain table references to a human being (ADR-055).
platform_users The authentication account. One row per login credential holder. Holds email, MFA status, account_status, last_seen_at, and person_id FK(iam_person). Reserved exclusively for auth/session/audit references. Not used for human identity FKs in domain tables.
Domain projections hr_employees, sis_students, sis_guardians — school-scoped role context records. Use person_id FK(iam_person) for identity and account_id FK(platform_users) for auth access. The correct FK for 'what role is this person playing' (ADR-055 category 3).

3.2 FK Contract (ADR-055) — Three Mutually Exclusive Categories

Category FK Target Use When
Human identity FK(iam_person) Column represents who this person is: person_id, patron_id, data_subject_id, member_id
Auth / session / audit FK(platform_users) Column represents who is acting: created_by, reviewed_by, actor_id, account_id
Role context FK(hr_employees), FK(sis_students), etc. Column represents what role a person plays in a school context

3.3 Access Grant Model

iam_role_assignment is the sole mechanism for granting access. There is no parallel access grant path. Key design decisions:

Scoped — every assignment is bound to an iam_scope (DISTRICT, SCHOOL, DEPARTMENT, CLASS, ACTIVITY, etc.). Assignments without a scope are invalid.
Time-bounded — effective_from and effective_to with an expiry sweep worker. NULL effective_to means indefinite.
Provenance-labelled — source ENUM(MANUAL, HR_SYNC, SIS_DERIVED, GUARDIAN_RELATIONSHIP, WORKFLOW_APPROVAL, EMERGENCY) with source_ref_id linking to the originating record.
Guardian access — always derived from iam_relationship_access_rule evaluation, never implicit. Changing a guardian relationship triggers immediate reassessment within 60 seconds (I-17).
Break-glass grants — iam_break_glass_grant for emergency access with mandatory justification, mandatory expiry, immediate notification, and post-event review requirement (I-18).

3.4 Authentication Delegation

CampusOS authorises — the IdP authenticates. Authentication is delegated entirely to the configured external identity provider (Keycloak, Auth0, Cognito). CampusOS stores IdP state in iam_federated_identity but never implements authentication logic itself. Supports SAML 2.0 and OIDC for Google Workspace and Microsoft Entra ID SSO (very common in K-12).

3.5 Permission Cache Architecture

iam_effective_access_cache is the sole source for hot-path permission checks. Direct JOINs across role_assignment → role → permission tables are forbidden on the request path. Cache design:

- Redis is the primary cache (5-minute TTL). iam_effective_access_cache is the persistent fallback.

- Staleness SLA: 500ms maximum for active sessions. Pre-warm target: p95 < 200ms after LOGIN or ROLE_CHANGE events (ADR-043).

- Cache hit rate target: > 0.99 after pre-warm.

- Rebuild trigger: any iam_role_assignment INSERT/UPDATE, iam_relationship_access_rule change, or role_permission change.

4. Database Architecture

4.1 Partitioning Strategy

All high-volume tables use PostgreSQL declarative partitioning. Strategy choices are documented table-by-table in the ERD; the platform-wide rules are:

HASH partitioning Universal identity tables (iam_person, platform_users, platform_students) — 32 buckets each. Enables horizontal scale of the platform schema without range-based hot spots. Online migration via pg_partman 5-step procedure (ADR-041).
RANGE by time All append-only operational tables: messages (monthly), attendance (annual), audit_log (monthly), pay_ledger_entries (annual), fin_gl_entries (annual), iam_access_change_events (monthly). T1/T2/T3 archival tiers (ADR-026).
RANGE by tenant bucket msg_threads: HASH(school_id) 64 buckets (increased from 16 — ADR-047). Large districts create hot partitions at 16 buckets.
No partitioning Most domain tables per tenant (sis_students, cls_assignments, hr_employees) — individual tenant schemas are naturally bounded; partitioning adds overhead without benefit at typical school scale.

4.2 Data Lifecycle — Archive Tiers (ADR-026)

T1 — Hot PostgreSQL primary. Active partitions. Fully indexed. Typical retention: 12 months for operational data, 7 years for FERPA-regulated records.
T2 — Warm PostgreSQL with detached/read-only partitions. Queryable via archive_query_log. Estimated cost tracked per query.
T3 — Cold S3 as Parquet files. Analytical access only via platform_archive_query_log. Signed URL delivery.
T4 — Legal hold platform_legal_holds prevents partition detach or row archival. Two-party release required.

4.3 Immutability Contracts

The following tables are immutable — no UPDATE or DELETE grants are given to the application database role (P-01 Platform Invariant):

- audit_log — every access to student PII, health records, and financial data

- pay_ledger_entries — financial transaction ledger

- fin_gl_entries — GL entries once batch is POSTED

- iam_role_assignment_history — access change audit trail

- trn_dispatch_events — transport safety log

- trn_route_change_log — safeguarding record of route changes

- iam_access_change_event, iam_auth_event — IAM audit logs

4.4 Cross-Module Reference Health (ADR-028)

All SOFT INTEGRITY references are registered in platform_reference_health. A background worker validates that each registered reference resolves correctly. A CI check at build time validates that every SOFT INTEGRITY column comment has a corresponding registry entry (P-03 Platform Invariant). This is the primary mechanism for catching cross-module referential drift without DB-level FKs.

5. Platform Event Model (ADR-057)

All Kafka events across all modules conform to a standard envelope and contract. This was formalised in ADR-057 after peer review identified inconsistent event semantics as a scalability risk.

5.1 Canonical Event Envelope

Required fields on every CampusOS Kafka event event_id: UUID v7 — globally unique, monotonically increasing. Used as the idempotency key by all consumers. event_type: STRING — dot-separated domain.entity.verb (e.g., sis.student.enrolled, pay.invoice.created). event_version: STRING — semver. Minor increments are backward-compatible field additions. Major increments require a new topic. occurred_at: TIMESTAMPTZ — when the domain event occurred. Consumers building read models must use this for ordering, not published_at. published_at: TIMESTAMPTZ — when the message was written to Kafka. Used for lag monitoring only. tenant_id: UUID — the school_id. All consumers filter on this. Cross-tenant fan-out is forbidden. source_module: STRING — the module prefix (e.g., sis, pay, hlth). Used for DLQ routing and observability. correlation_id: UUID nullable — the originating request ID for distributed tracing. payload: JSONB — event-specific data, typed per event_type and event_version.

5.2 Topic Naming and Ownership

Topics follow the convention: {environment}.{domain}.{entity}.{verb} — e.g., prod.sis.student.enrolled, prod.pay.invoice.created, prod.hlth.allergy_alert.changed.

- Each topic is owned by exactly one source module — the sole producer.

- Other modules are consumers only. A module never publishes to another module's topic.

- Topic creation requires a platform event registry entry (a configuration file in the platform repository, not a database table).

- Schema changes require a PR to the event registry with review from the owning module team and all affected consumer teams.

5.3 Idempotency and Replay

Every consumer must be idempotent with respect to event_id. The standard pattern:

- INSERT event_id into a consumer-local idempotency table (consumer_name, event_id, processed_at) with a UNIQUE constraint.

- If INSERT succeeds: process the event.

- If INSERT fails (duplicate): skip processing, acknowledge the message.

- Replay: reset consumer group offset to desired position. Clear the idempotency table for the replayed range first — this is a supervised operational procedure.

5.4 Dead Letter Queue

Every consumer has a DLQ topic at {original_topic}.dlq. On three consecutive processing failures for the same event_id, the message moves to the DLQ with failure_count, last_error, last_failed_at, consumer_name appended to the envelope. DLQ messages older than 15 minutes trigger an SRE alert. Manual replay after root cause resolution.

6. Scheduling Engine Contract (ADR-056)

Academic scheduling was identified in peer review as having a strong data model but undefined execution model. ADR-056 specifies the complete behavioral contract.

6.1 Architecture

✓ Recommendation The scheduling engine is an external constraint solver invoked by a background worker (SchedulingWorker). It is stateless — it receives a complete problem description in one call and returns a complete solution set. CampusOS owns all state; the solver owns only computation.

Input tables assembled by SchedulingWorker before invoking the solver: sch_student_subject_choices (demand), sis_classes (class definitions), hr_employees + sch_cross_school_staff_assignments (teacher availability), sch_rooms (room inventory), sch_periods (period grid), sch_scheduling_constraints (active constraint profile), sch_rotation_calendar (date-to-rotation mapping).

6.2 Constraint Model

Hard constraints (H — DB-enforced, cannot be violated) H1: teacher cannot be in two places simultaneously (DB EXCLUSION CONSTRAINT on sch_timetable_slots). H2: room cannot be double-booked (same). H3: student cannot be in two concurrent classes. H4: class size cannot exceed room capacity. H5: cross-school staff cannot exceed max_periods_per_week.
Soft constraints (S — scored and minimised) S1: avoid consecutive periods for same teacher. S2: honour teacher period preferences. S3: minimise student free periods. S4: avoid same-subject on consecutive days. S5: distribute co-teaching arrangements evenly.

6.3 Promotion Lifecycle

Four explicit stages, each gated by an explicit state change:

DRAFT Solver output written to sch_scheduling_candidates (review_status=PENDING) and sch_scheduling_candidate_slots.
REVIEWED Admin reviews candidates. Slots with has_clash=true are highlighted.
APPROVED Admin selects one candidate. SchedulingActivationWorker begins promotion.
ACTIVE Worker creates/updates sch_timetable_slots rows. sch.timetable.updated Kafka events fire. Calendar delta-update worker runs.

A candidate cannot be activated if any has_clash=true slots remain unresolved. Three conflict resolution options: (1) remove the slot; (2) reassign to alternative room/teacher; (3) override with documented justification stored in sch_scheduling_candidate_slots.clash_description — only permitted for room conflicts, never teacher conflicts.

6.4 Rebuild Triggers

Five defined triggers that mark a schedule as stale and prompt admin-initiated regeneration: new enrolment with unscheduled subject choices, teacher availability change, room added/closed, constraint profile updated, class added or removed. Rebuilds are never automatic — emergency mid-year changes use sch_cover_arrangements, not full regeneration.

7. Finance Architecture (ADR-038, ADR-051, ADR-058)

7.1 M81 / M84 / M83 Boundary (restated for v10)

M81 Enrollment & Payments (Wave 1) Owns the transaction: invoice, payment, family account, Stripe event, refund, credit note, debt write-off. AR pipeline. Produces Kafka events consumed by M83.
M83 Finance & Accounting (Wave 4) Owns the interpretation: GL entries, chart of accounts, budgets, fund accounting, AP vouchers, financial reports. Consumes M81 events to post double-entry GL batches. Never writes to M81 tables.

7.2 Minimal GL Backbone (ADR-058) — Pre-Staged in Wave 1

The GL backbone was designed alongside M81 to validate the event integration contract. 9 fin_ tables are physically documented under M81 and will migrate to M83 at Wave 4 commencement:

fin_chart_of_accounts — account hierarchy with type (ASSET/LIABILITY/EQUITY/REVENUE/EXPENSE), normal balance (DEBIT/CREDIT), fund FK, and is_system protection for control accounts.
fin_funds — fund accounting classification (GENERAL, SPECIAL_REVENUE, CAPITAL_PROJECTS, DEBT_SERVICE, PERMANENT, ENTERPRISE).
fin_accounting_periods — fiscal period control gate. OPEN/CLOSED/LOCKED status. LOCKED periods are permanently sealed — no further posting without CFO override.
fin_journal_batches — atomic posting unit. Balance enforced by validate_batch_balance(batch_id) DB procedure which aggregates actual fin_gl_entries rows — not a generated column on application-maintained header totals (ADR-059 fix). PostingService writes all entries, then calls validate_batch_balance(), then sets status=POSTED — all in one transaction. Rollback on any failure. source_event_id UNIQUE index provides Kafka idempotency.
fin_gl_entries — individual debit/credit lines. IMMUTABLE once batch is POSTED — DB trigger + application role permission loss. RANGE-partitioned by period_start DATE annually — partition key is denormalised from fin_accounting_periods.period_start at INSERT time, aligning partitions with fiscal year boundaries (ADR-059 fix: period_id UUID cannot be a RANGE partition key in PostgreSQL).
fin_budgets + fin_budget_lines — budget header per fund per fiscal year with account-level line allocations. Actuals computed on demand from fin_gl_entries, cached in Redis.

9. Security Architecture

9.1 Authentication

Authentication is fully delegated to external identity providers per ADR-036. CampusOS never implements authentication logic — it authorises access and delegates auth to the configured IdP.

- Supported: SAML 2.0 and OIDC (Google Workspace, Microsoft Entra ID, Keycloak, Auth0, Cognito)

- Credential type: short-lived JWTs (15-minute access tokens) with refresh token rotation in HttpOnly cookies

- Auto-provisioning: first login from a trusted IdP can create a platform_users row (iam_identity_provider.auto_provision_accounts)

- Trust levels: HIGH = district-managed corporate IdP; LOW = consumer social login (different session constraints)

9.2 Role-Based Access Control

The RBAC model uses permission-level checks, not role name checks. Every @PreAuthorize annotation checks a permission code (e.g., grade:write, student_record:read), never a role name. Roles are named collections of permissions, configurable per tenant.

The full access control stack: IAM Gateway checks iam_effective_access_cache (Redis primary, DB fallback) → permission code resolution → resource ownership check (a teacher can grade:write only in their own classroom). All permission checks happen without touching iam_role_assignment on the hot path.

9.3 Data Isolation

- Every DB query must include tenant_id / schema context. Spring interceptor validates tenant context before any repository call.

- Row-level security (RLS) as defense-in-depth even with schema isolation.

- All S3 assets served via signed URLs with 15–60 minute expiry. No public bucket URLs.

- PII fields: AES-256 at rest, KMS-managed keys. Health records use a separate HIPAA-compliant KMS key.

9.4 Compliance

FERPA (US) Immutable audit_log records every access to student PII. data_subject_id INDEX enables DPO cross-module queries. 7-year minimum retention. compliance_ferpa_requests table for formal records requests. DPO module (dpo_) provides full SAR/erasure/portability lifecycle.
COPPA Age-gating in enrollment flow. Students under 13 require parental consent. data_subject_is_self flag on platform_students transitions rights at age 18.
HIPAA hlth_* tables use a separate KMS key. Access restricted to health_record:read permission. Classroom module never reads hlth_* directly — reads sis_student_active_accommodations read model instead (ADR-030).
GDPR / UK GDPR dpo_ compliance suite: 30/45-day SAR deadlines, erasure requests with legal hold checks, processing restrictions, ROPA (processing activities register), DPIAs for high-risk processing, 72-hour breach notification records.
DBS/Background checks Reference numbers stored; raw registry data never persisted (ADR-015). Same pattern for iam_federated_identity — IdP credentials are references, not copies.

13. Pre-Build Architectural Decisions (ADR-059)

Eight architectural issues were identified and resolved before Wave 1 build commences. All are reflected in ERD v10. These decisions are documented here for engineering teams — they affect implementation patterns across multiple modules.

Fix 1 — fin_gl_entries Partition Key Issue: fin_gl_entries was defined as RANGE partition by period_id (UUID). PostgreSQL RANGE partitioning requires an ordered type — UUIDs are not ordered meaningfully. Decision: Added period_start DATE NOT NULL to fin_gl_entries, denormalised from fin_accounting_periods.period_start and set at INSERT time. RANGE partition by period_start ANNUAL. Partitions align with fiscal years — the natural audit and reporting boundary. period_start is immutable once set because GL entries cannot move between periods after posting.
Fix 2 — fin_journal_batches Balance Validation Issue: is_balanced GENERATED ALWAYS AS (total_debits = total_credits) checked application-maintained header aggregates, not actual entry rows. A batch could show balanced while having unbalanced entries. Decision: Removed total_debits, total_credits, and is_balanced from fin_journal_batches. Balance is validated by validate_batch_balance(batch_id) — a DB procedure that aggregates SUM(amount) by entry_type from actual fin_gl_entries rows. PostingService write order: (1) write all entries, (2) call validate_batch_balance(), (3) set status=POSTED — all inside one transaction. Rollback on any failure. Implementation note: the procedure must be called after all entries are written, not before. Calling it before writing entries produces a spurious 'balanced at zero' result.
Fix 3 — Event Idempotency Table Issue: ADR-057 specified a per-consumer idempotency table pattern but no such table existed in the schema. Decision: platform_event_consumer_idempotency added to M0 platform schema. Schema: (consumer_name TEXT, event_id UUID, processed_at TIMESTAMPTZ). UNIQUE(consumer_name, event_id). RANGE partition monthly, 90-day retention. Implementation note: the table is shared across all consumers — event_id is globally unique (UUIDv7) so consumer_name + event_id is sufficient without tenant_id. However, at Wave 1 scale the shared write path may create index contention. If profiling shows contention, partition by HASH(consumer_name) to give each consumer its own partition. Each service must use its logical name consistently as consumer_name — define these in a platform constants file, not per-service strings.
Fix 4 — iam_scope_membership Removed Issue: iam_scope_membership recorded scope membership as a prerequisite before role assignment. iam_role_assignment already implies scope membership — maintaining a separate table creates dual sources of truth. Decision: iam_scope_membership removed. Scope membership is derived from active iam_role_assignment rows — an account is a member of a scope if and only if it has at least one ACTIVE assignment there. IAM cache rebuild aggregates this directly. Implementation note: any UI feature that needs to show 'which scopes does this account belong to' should query SELECT DISTINCT scope_id FROM iam_role_assignment WHERE account_id=$1 AND status='ACTIVE'. Do not introduce a replacement membership table.
Fix 5 — GDPR Erasure Strategy for Immutable Tables Issue: audit_log, pay_ledger_entries, and fin_gl_entries are immutable but may contain PII. No anonymisation path existed. Decision (hybrid): audit_log.metadata JSONB receives field-level pseudonymisation via PseudonymisationWorker on approved dpo_erasure_requests — replacing data subject identifiers with an opaque token, logged in dpo_pseudonymisation_log. fin_gl_entries.description and pay_ledger_entries are retained on legal obligation grounds (FERPA 7-year minimum, financial record retention law) — denial basis documented in dpo_erasure_requests.denied_categories. Implementation note: PseudonymisationWorker must hold a legal-hold check (platform_legal_holds) before processing any erasure. The pseudonymisation token stored in dpo_pseudonymisation_log must be irreversible — not a lookup key back to the original value.
Fix 6 — Identity Projection Sync (person.identity.updated) Issue: platform_students and platform_substitute_profiles hold denormalised name/DOB snapshots from iam_person but no mechanism existed to keep them in sync on update. Decision: person.identity.updated Kafka event emitted on iam_person UPDATE to first_name, last_name, date_of_birth, or preferred_name. IdentityProjectionSyncWorker consumes and updates platform_students, platform_substitute_profiles, and platform_volunteer_profiles snapshots. Implementation note: the worker must be idempotent. The event carries the full updated iam_person snapshot so the worker can perform a blind UPDATE — no need to read before writing. Legal name changes must go through iam_person (the authoritative source) first; updating a projection table directly will be overwritten on the next identity event.
Fix 7 — sch_rooms / fac_spaces Creation Protocol Issue: no process defined for keeping sch_rooms and fac_spaces in sync. A room added in Facilities could be invisible to scheduling. Decision: fac.space.created event (consumed by SchedulingWorker → creates draft sch_rooms entry with is_active=false and requires_scheduling_setup=true). fac.space.deactivated event (sets sch_rooms.is_active=false). Scheduling Admin reviews and activates draft rooms. Implementation note: sch_rooms.requires_scheduling_setup=true means the room exists but is not yet available for timetabling. The scheduling UI must surface a 'rooms pending setup' task. This ensures no physical room is invisible to the scheduler, and no room is assigned in schedules without explicit admin confirmation.
Fix 8 — Cross-School Family Identity (platform_families) Issue: sis_families is tenant-scoped — siblings at different schools in a district could not be identified without cross-schema JOINs. Decision: platform_families and platform_family_members added to M0 platform schema. sis_families gains platform_family_id FK nullable. FamilyMemberSyncWorker keeps platform_family_members in sync with sis_family_members events. Cross-school sibling query is now a platform-schema JOIN. Implementation note: standalone schools leave platform_family_id=null with no operational impact. For district deployments, the family provisioning workflow must create the platform_families record first, then link sis_families.platform_family_id at tenant provisioning. Do not create platform_families records retroactively from sis_families — the linking must be intentional and managed.

14. Pre-Build Engineering Concerns

Five implementation concerns identified during pre-build review. These are not ERD defects — the schema is correct — but each requires an explicit implementation contract before the affected module is built. All five are resolved with documented mitigations.

Concern 1 — platform_event_consumer_idempotency Write Contention Risk: all Kafka consumers write to one shared table on every event processed. Under Wave 1 load with a dozen concurrent consumers, the UNIQUE(consumer_name, event_id) index becomes a hot write contention point. Mitigation: monitor p95 INSERT latency on this table from day one. If contention appears, add a HASH partition by consumer_name — each consumer gets its own partition and the index is partitioned accordingly. This is a one-step online migration via pg_partman. Action before build: define a consumer_name registry (platform constants, not per-service strings). Agree on the HASH partition trigger threshold (e.g. p95 > 5ms on INSERT).
Concern 2 — IAM Cache Rebuild Cascade on Bulk Provisioning Risk: at academic year start, hundreds of iam_role_assignment rows are created in a batch. Each INSERT fires a cache rebuild event. If the IAMCachePrewarmWorker processes them serially, the 500ms staleness SLA is breached for every account until the queue drains. Mitigation: add deduplication to iam_cache_prewarm_queue — if an account_id already has a QUEUED entry, do not INSERT a new one; let the existing rebuild run. This converts O(n) rebuild events per account into O(1). Action before build: add a UNIQUE partial index on iam_cache_prewarm_queue(account_id) WHERE status='QUEUED'. The INSERT-on-conflict-do-nothing pattern becomes the standard cache invalidation path.
Concern 3 — Task Worker as Platform-Wide Single Point of Failure Risk: the Task Worker is invoked by absence requests, room changes, approval escalations, library fines, IEP completions, equipment overdue, bus no-shows, withdrawal processing, and more. If the worker is backed up, all of these notifications are delayed simultaneously across all modules. Mitigation: define two Task Worker tiers — SAFETY_CRITICAL (bus no-show, allergen override, emergency) and STANDARD (library reminders, assignment notifications, room change requests). Safety-critical tasks bypass the standard queue and use a dedicated high-priority consumer with reserved capacity. Standard tasks go through the normal queue. Action before build: categorise every tsk_auto_task_rules trigger_event_type as SAFETY_CRITICAL or STANDARD. Define separate Kafka topics or consumer groups for each tier. Define queue depth SLA: SAFETY_CRITICAL tasks delivered within 10 seconds; STANDARD within 60 seconds.
Concern 4 — Scheduling Solver Resource Governance Risk: sch_scheduling_requests has no timeout, no concurrent job limit, and no defined behaviour when a large school's solver run exceeds the time budget. Two schools submitting simultaneously compete for the same SchedulingWorker resources. Mitigation: add timeout_seconds INT and max_solutions_to_return INT to sch_scheduling_requests. Add a per-tenant concurrent job limit (max 1 RUNNING job per school at a time — enforced by the worker before starting). Define failure mode on timeout: write whatever partial solutions exist as FAILED candidates with error_message='SOLVER_TIMEOUT'; notify the scheduling admin. Action before build: set a default timeout of 600 seconds (10 minutes). Define the performance SLA for acceptance testing: a school with 400 sections must produce at least one candidate solution within 600 seconds.
Concern 5 — GL Posting Transaction Write Order Risk: fin_journal_batches.validate_batch_balance() checks the actual fin_gl_entries rows. If PostingService calls the procedure before all entries are written, it validates against an incomplete set and produces a false 'balanced' result. Mitigation: enforce write order in PostingService as a documented contract: (1) BEGIN TRANSACTION; (2) INSERT all fin_gl_entries rows for the batch; (3) CALL validate_batch_balance(batch_id) — procedure raises EXCEPTION if debit sum ≠ credit sum; (4) UPDATE fin_journal_batches SET status='POSTED'; (5) COMMIT. Any other order is a bug. Add an integration test that verifies a partial-entry batch cannot be posted. Action before build: add validate_batch_balance() to the ERD as a documented DB procedure alongside fin_journal_batches. Define the EXCEPTION message format so the application can surface it to the Finance Admin.

RESOLVED — ADR-060: iam_person.person_types[] Multi-Type Identity Decision: person_type ENUM NOT NULL → person_types TEXT[] NOT NULL. A person can simultaneously hold multiple identity types (e.g. {STAFF, GUARDIAN} for a teacher who is also a parent). 16. Operational Risk Governance

Seven operational risks identified in principal-architect peer review. These are not design gaps — the architecture is correct — but execution risks that surface at scale. Each is documented with its mitigation contract and implementation timing.

16.1 Platform Schema as a Separate Scaling Tier

The platform schema centralises high-write-volume tables: iam_person, platform_users, platform_event_consumer_idempotency, platform_families. Even with HASH partitioning, this creates shared WAL pressure and cross-tenant write contention during spikes — most critically at academic year start when bulk provisioning fires simultaneously across all schools.

Immediate mitigation Treat platform schema as its own scaling tier from day one. It should have: dedicated connection pool separate from tenant schema pools, independent WAL monitoring, its own read replica for identity projection reads, and separate backup and PITR windows.
Medium-term (Wave 3–4) If platform schema write latency p95 exceeds 10ms sustained, isolate IAM tables (iam_person, iam_role_assignment, iam_effective_access_cache) onto a dedicated PostgreSQL instance with logical replication from the primary. This is the logical sharding path — plan the replication topology now even if the trigger is not yet reached.
Event idempotency platform_event_consumer_idempotency is the highest-volume write table in the platform schema. Monitor INSERT latency per consumer_name from day one. Trigger: add HASH partition by consumer_name if p95 INSERT latency exceeds 5ms. This is a one-step pg_partman migration.
Back-to-school spike Academic year provisioning must be a controlled batch operation — not an event fan-out. iam_role_assignment bulk inserts must use a dedicated provisioning job with rate limiting, not concurrent Kafka consumer processing. The deduplication guard on iam_cache_prewarm_queue (UNIQUE partial index WHERE status=QUEUED) is the immediate mitigation for cache rebuild cascades.

16.2 Event Schema Governance Lifecycle

The ADR-057 event model provides correct foundations: versioned envelope, topic ownership, idempotency contract. What decays over 12–18 months without active governance is schema drift — breaking changes smuggled as minor versions, consumer drift, and 'temporary' fields becoming permanent.

Schema registry Introduce a schema registry (Confluent Schema Registry or a lightweight custom equivalent) before Wave 2 events are published. Every event_type + event_version must have a registered Avro or JSON Schema. The registry is the contract, not the ERD documentation.
Backward compatibility CI check Add a CI gate that compares the proposed event schema against the registered version and rejects: removed fields, renamed fields, type changes, and new required fields. Additive changes (new optional fields) are allowed on minor version increments. Breaking changes require a new major version and a new topic.
Event deprecation lifecycle Define a three-stage lifecycle: ACTIVE → DEPRECATED (consumers notified, migration window opens, minimum 90 days) → SUNSET (topic sealed, old version consumers must have migrated). The event registry tracks lifecycle state per topic per version. No topic is sunset without all registered consumers confirming migration.
Topic ownership enforcement No team may publish to a topic they do not own. The registry enforces producer identity at publish time. Ownership changes require an ADR.

16.3 Worker Orchestration Standard

The platform now has a significant worker footprint: SchedulingWorker, IdentityProjectionSyncWorker, CalendarDeltaWorker, GLPostingWorker, AI pipelines, Task Workers (two tiers), TransportDispatchService, EmergencyAlertService, FamilyMemberSyncWorker, PseudonymisationWorker, NoisyNeighborDetector, IAMCachePrewarmWorker, DistrictAnalyticsWorker. Without a standard, each worker implements its own retry policy, failure handling, and observability — creating a coordination complexity that compounds with each wave.

Retry standard All workers must implement: exponential backoff starting at 1 second, jitter of ±20% to prevent retry storms, maximum 5 retries before DLQ routing, retry count logged on the message envelope. No worker implements a fixed-interval retry. No worker retries indefinitely.
Circuit breakers Each worker that calls an external dependency (DB, Redis, external solver, AI Gateway, push provider) must implement a circuit breaker: CLOSED (normal), OPEN (dependency failing — fail fast, do not retry), HALF-OPEN (probe recovery). Circuit state is emitted as a metric. SAFETY_CRITICAL workers (Emergency Alert, Transport Dispatch) use a read-only local cache in OPEN state rather than failing.
Global worker dashboard All workers emit to a common observability namespace: worker_name, tenant_id, event_type, processing_duration_ms, retry_count, circuit_state. A single SRE dashboard shows queue depth, DLQ size, circuit state, and p95 processing time across all workers. Worker health is a first-class platform metric, not per-service logging.
Orchestration layer (Wave 2+) For multi-step workflows with compensation logic (GL posting → validation → status update; scheduling solve → candidate review → activation), introduce Temporal or a Temporal-pattern equivalent. Temporal provides: durable execution across process restarts, built-in retry with backoff, workflow versioning, and a UI for inspecting in-flight workflows. This is not required for Wave 1 single-step consumers but should be evaluated before Wave 2 financial workflows are built.

16.4 Scheduling Solver Fallback and Explainability

The 600-second SLA was set against clean test data. Real school data has conflicting legacy constraints, teacher overrides, and edge cases that can make a fully-scheduled solution impossible. A solver that returns no result is a support incident, not a feature.

Partial schedule mode The solver must always return something. Define a 'best partial schedule' mode: if a complete solution is not found within the SLA, return the highest-coverage partial candidate (most sections scheduled, fewest hard constraint violations). sch_scheduling_candidates gains a completion_percentage NUMERIC and unscheduled_section_count INT. The admin UI surfaces partial candidates with explicit warnings — never silently.
Constraint relaxation strategy Define a ranked relaxation order for soft constraints when a full solution is not reachable: (1) relax teacher period preferences; (2) allow consecutive periods for same teacher; (3) allow same-subject on consecutive days; (4) relax room preference (accept any available room). Hard constraints (teacher double-booking, room capacity) are never relaxed. The relaxation path taken is recorded on sch_scheduling_candidates.relaxations_applied JSONB for explainability.
Conflict explainability When the solver cannot schedule a section, it must return a structured conflict reason: which hard constraints are in tension, which teacher/room/period combinations were exhausted. This is surfaced in the admin UI as 'why this section could not be scheduled' — not a generic failure message. sch_scheduling_candidates.conflict_report JSONB stores the structured output.
Admin trust Scheduling is the feature most likely to lose admin trust quickly if it silently fails. The partial schedule + explainability contract means the admin always knows what was scheduled, what was not, and why — even on a difficult input.

16.5 Analytics Cost Governance

The CQRS read model approach (rpt_ tables, replica routing, ADR-049 incremental snapshots) is the correct foundation at Wave 1–3 scale. District-level cross-school analytics adds aggregation cost that compounds as tenant count grows.

Current approach (Wave 1–3) rpt_ read models materialised by background workers. Replica-routed reads. Incremental snapshots (ADR-049) prevent full Kafka replay. DistrictAnalyticsWorker (ADR-060) aggregates school-level snapshots nightly. This is sufficient for the first 200–500 schools.
Cost trigger Monitor: DistrictAnalyticsWorker nightly job duration and compute cost per school. Trigger for next tier: nightly job duration exceeds 4 hours OR compute cost per school exceeds $X/month (define threshold at Wave 3 launch based on actual unit economics).
Lakehouse hedge (Wave 4+) When the cost trigger is reached: introduce a lakehouse layer (Apache Iceberg or Delta Lake) fed by Kafka events. Analytics queries run against the lakehouse via Athena or Trino — not against PostgreSQL replicas. The rpt_ read models remain for operational dashboards; the lakehouse serves historical analysis, district-wide reporting, and board financial reports. This is not a now decision — but the Kafka event model (ADR-057) is the correct foundation for it. Every event published today is future lakehouse fodder.
Avoid premature optimization Do not introduce a lakehouse before the cost trigger. The operational complexity of maintaining two analytics stacks before you need it creates more risk than the compute cost it saves.

16.6 Emergency Alert Minimal Dependency Execution Path

The Emergency Alert Service and Transport Dispatch Service are correctly extracted from the monolith. However, both still depend on shared upstream infrastructure (Kafka, platform DB, auth service) for their normal operation. Under an outage — which is precisely when emergency alerts matter most — these dependencies can degrade the services that must not degrade.

Emergency Alert minimal path The Emergency Alert Service must maintain a local incident cache (Redis or in-process) of active emergency declarations refreshed every 30 seconds. On DB degradation: serve from local cache. On Kafka degradation: publish directly to push/SMS provider bypassing Kafka entirely (direct API call, not event). On auth degradation: accept pre-validated emergency tokens (short-TTL tokens issued at service startup and refreshed every 5 minutes) rather than validating against the auth service on each request.
Transport Dispatch minimal path The Transport Dispatch Service must maintain a local session cache of active driver sessions. On DB degradation: driver WebSocket connections remain active; manifests are served from the local session cache; stop updates are buffered locally and flushed when DB recovers. On Kafka degradation: dispatch events are written to a local outbox table in the service's own minimal DB (a small embedded store, not the platform DB) and replayed when Kafka recovers.
Dependency audit Before each Wave 1 service is built, produce a dependency audit: list every external call on the hot path, classify each as REQUIRED (service cannot function without it) or DEGRADABLE (service can operate in reduced mode without it). Platform DB and Kafka must be DEGRADABLE for Emergency Alert and Transport Dispatch. Auth must be DEGRADABLE for Emergency Alert.
Testing Add chaos engineering tests for Emergency Alert and Transport Dispatch: simulate DB outage, Kafka outage, and auth service outage. Verify the service continues to operate in degraded mode. These tests run in the CI pipeline, not just in production drills.

16.7 Summary — Operational Discipline as the Success Condition

✓ Recommendation All seven peer-review risks are addressed above with specific mitigations, triggers, and action items. The platform is architecturally sound. Execution success depends on: (1) enforcing the three-tier boundary strategy from the first line of code; (2) implementing the worker orchestration standard before Wave 2; (3) introducing event schema governance before the first Wave 2 event is published; and (4) building the Emergency Alert minimal dependency path before going to production. These are not optional governance items — they are the difference between a system that scales and one that requires an architectural intervention at scale.

19. Read Model Ownership

CampusOS uses a CQRS-lite pattern (ADR-008): domain modules write to their operational tables, background workers materialise read models into rpt_ tables for dashboards, analytics, and reporting. Without explicit ownership, read models accumulate duplicated logic, inconsistent metrics, and conflicting definitions of the same business concept.

19.1 Ownership Rules

One owner per read model — each rpt_ table has exactly one owning module. The owning module's worker is the sole writer. No other module writes to a rpt_ table it does not own — reads are permitted for display.
Naming convention — rpt_{owning_module_prefix}_{concept}. Examples: rpt_sis_daily_attendance_summary (owned by M20 SIS), rpt_fin_aged_debtors (owned by M81/M83 Finance), rpt_trn_fleet_status (owned by M61 Transportation), rpt_district_summary (owned by M110 Analytics).
Worker naming convention — {Domain}ReadModelWorker. Example: SISAttendanceReadModelWorker, FinanceARReadModelWorker, DistrictAnalyticsWorker. The worker name must match the owning module — a worker named 'ReportingWorker' that writes to five different module read models is an ownership violation.
Update trigger documentation — every rpt_ table must document its update trigger in the ERD: either (a) Kafka event consumer naming the specific topic(s), (b) nightly scheduled job with timing, or (c) on-demand rebuild with the initiating endpoint. No rpt_ table without a documented update path.

20. Database Migration Governance

658 tables across 32 modules and a schema-per-tenant deployment means migrations run against every tenant schema on every deploy. Without explicit governance, migrations become the most dangerous part of the deployment pipeline — a breaking change in a migration is irreversible at tenant scale.

20.1 Migration Tooling and Versioning

Tooling Flyway with strict versioning mode. All migration scripts are version-controlled alongside application code. Migration version numbers are immutable — once a migration is merged, its version number and checksum are frozen. No modification of existing migration scripts ever.
Version format V{major}.{minor}__{description}.sql. Major version increments on wave boundaries. Minor version increments on individual changes. Example: V1.0042__add_sis_absence_reason_category.sql. Descriptions must be human-readable — not ticket numbers alone.
Multi-tenant execution TenantAwareMigrationRunner discovers all tenant schemas at deploy time and runs pending migrations against each. Migrations for the platform schema run first, then tenant schemas in batches (not serially — parallelise to a configurable batch size, default 20 concurrent schemas). Failures are logged per tenant and do not abort the full deployment — a per-tenant failure is isolated and replayed.
hlth / fin / dpo schemas The three separate module schemas each have their own Flyway migration history table. hlth, fin, and dpo migrations are versioned independently of the tenant schema — they can evolve at different cadences. A change to fin_ tables does not require a full tenant schema migration cycle.

20.2 Backward-Compatible Schema Evolution Rules

Permitted (no migration coordination required) ADD COLUMN nullable — always safe. Add new nullable columns freely. ADD COLUMN with DEFAULT — safe as long as the default does not require a full-table rewrite (avoid DEFAULT with a function call on large tables). ADD INDEX — safe (CONCURRENTLY required on tables > 10k rows). ADD TABLE — always safe. ADD ENUM value — safe in PostgreSQL 12+ (appends to end of enum; existing rows unaffected). ADD FOREIGN KEY — safe with NOT VALID + VALIDATE CONSTRAINT in two separate migrations (avoids full table lock).
Requires shadow column pattern (coordinate with application team) RENAME COLUMN — never rename directly. Add new column, dual-write in application (write both old and new), backfill new column, switch reads to new column, drop old column. Each step is a separate migration + deployment. CHANGE COLUMN TYPE — same shadow column pattern. Never ALTER COLUMN TYPE directly on a populated table. REMOVE COLUMN — deprecate first (mark nullable, stop writing), then drop in a later release after confirming no reads remain. Minimum 2 deployments. RENAME TABLE — treat as add new table + migrate data + drop old table across multiple releases.
Never permitted (requires ADR to override) DROP TABLE or DROP COLUMN without shadow column deprecation cycle. ALTER COLUMN NOT NULL on a table with existing rows (use CHECK constraint + backfill instead). Truncating or modifying data in a migration script. Migrations that take an exclusive lock for more than 2 seconds on a table with > 1000 rows (use CONCURRENTLY, NOT VALID, or online schema change tooling). Cross-schema FK additions from tenant schema to hlth/fin/dpo schemas — these must go through the ADR-028 SOFT INTEGRITY pattern.

20.3 Rollback Strategy

PostgreSQL DDL is transactional within a session. All migrations must be written inside an explicit transaction so that a mid-migration failure leaves the schema unchanged. Flyway's default behavior wraps each script in a transaction — do not disable this.

- Forward-only policy: migrations are never rolled back in production. If a migration causes an issue, the fix is a new forward migration. Rollback scripts are written and tested in staging but are the path of last resort.

- Staging must mirror production: staging runs against an anonymised snapshot of production data at production scale before any migration is promoted. A migration that takes 30 seconds on staging against a 10-row table will take hours against a production tenant with 10 years of attendance records.

- Migration lock: Flyway's migration lock table prevents concurrent migrations. In a multi-instance deploy, only one instance runs migrations — others wait. Lock timeout is 5 minutes; after timeout the deploy fails rather than proceeding without a migration lock.

21. Multi-Region Strategy

CampusOS is designed for global deployment from day one — GDPR, FERPA, COPPA, and international school markets require data residency decisions before the first tenant is onboarded. The multi-region architecture does not need to be implemented in Wave 1, but the tenant data model must accommodate it.

21.1 Current Deployment Model (Wave 1–3)

Single-region active Wave 1–3 deploys to a single AWS region (primary region). All tenants are hosted in this region. This is the correct starting model — operational simplicity, single RDS cluster, single MSK cluster, unified observability.
Region selection Choose the primary region based on the majority of initial customers and their data residency requirements. If the customer base is primarily US K-12: us-east-1 or us-west-2. If UK/EU schools are a significant early market: eu-west-1 (Ireland) or eu-west-2 (London). This decision is made at product launch, not at design time.
Data residency groundwork platform_tenant_routing gains a home_region TEXT column at Wave 1 (even if all tenants have the same value). This is the hook for multi-region routing — adding it later requires a migration and an application change simultaneously. Add it now when it is cheap.
GDPR Article 44+ compliance For EU/UK tenants, data must not be transferred to a country without adequate protection without explicit consent or a transfer mechanism (Standard Contractual Clauses). Single-region deployment in eu-west-1 or eu-west-2 for EU/UK tenants satisfies this with no transfers. Hosting EU tenants in us-east-1 requires SCCs documented in dpo_data_processing_agreements.

21.2 Multi-Region Target Architecture (Wave 4+)

Active-passive (recommended first step) A second region is brought online as a warm standby. Platform schema and tenant schemas are replicated via AWS RDS global database or logical replication. In a primary region failure, DNS failover routes traffic to the standby. RTO target: < 15 minutes. RPO target: < 30 seconds (replication lag). This is the correct first multi-region step — operationally simple, no split-brain risk.
Active-active (future consideration) Each region hosts a subset of tenants (tenant-region affinity). Cross-region reads are prohibited — a tenant's data never leaves its home region except for explicit disaster recovery. This model requires: tenant routing at the API gateway layer (before any DB connection), cross-region identity resolution for substitute/volunteer profiles (platform schema is per-region or globally replicated read-only), and a defined conflict resolution strategy (last-write-wins is not acceptable for financial data). Active-active is a significant operational commitment — do not pursue until single-region scale is exhausted.
Tenant-region affinity Each tenant is assigned to a home_region at provisioning (from platform_tenant_routing.home_region). The TenantRoutingDataSource (ADR-042) uses home_region to select the correct regional DB endpoint. A tenant's data never moves regions without an explicit migration procedure (the ADR-031 cluster rebalance pattern extended to cross-region).
Data residency enforcement dpo_erasure_requests and dpo_subject_access_requests must be processed in the tenant's home_region — GDPR data subject rights apply to the data in its actual location. The DPO workflow routes requests to the correct regional worker based on platform_tenant_routing.home_region. Cross-region DPO operations require explicit data transfer documentation in dpo_data_processing_agreements.

21.3 Actions Required Before First Non-US Tenant

- Add home_region TEXT NOT NULL to platform_tenant_routing (default to current single region). Add region_residency_requirement TEXT nullable (e.g., 'EU', 'UK', 'US') for compliance routing.

- Audit all S3 bucket configurations: EU/UK tenants must have their lesson videos, uploaded documents, and paystub PDFs in a bucket in their residency region. CloudFront distribution must not cache PII content outside the residency region.

- Populate dpo_data_processing_agreements with the platform's own data processing record — the school is a data controller, CampusOS is a data processor, and this relationship must be documented before any EU/UK tenant is onboarded.

- Review platform_event_consumer_idempotency: Kafka cross-region replication for event idempotency records is required in a multi-region deployment. Without it, a consumer in Region B may reprocess an event that was already processed in Region A before failover.

25. Observability Model

Observability is a first-class platform concern, not a tooling afterthought. This section defines structured standards for traces, logs, metrics, and alerts. These standards must be implemented from the first deployed service.

25.1 Trace Propagation Standard

trace_id UUID generated at the origin of a user action or scheduled job. Propagated unchanged across all services, DB calls, Kafka messages, and worker invocations for the lifetime of the operation. The trace_id in a Kafka message envelope (ADR-057 correlation_id field) is this value. Never regenerated mid-operation.
span_id UUID generated at each service boundary. A new span_id is created when the operation crosses into a new service, worker, or async boundary. parent_span_id links it to the calling context.
tenant_id The school_id. Always present on every trace, log line, and metric. Primary dimension for multi-tenant debugging and capacity planning.
Propagation HTTP headers (X-Trace-Id, X-Span-Id, X-Tenant-Id) on all internal service calls. Kafka envelope correlation_id carries trace_id. Worker jobs carry trace_id from the triggering event. Scheduled jobs generate fresh trace_id at invocation. OpenTelemetry is the instrumentation standard -- AWS X-Ray or Jaeger as the backend.

25.2 Structured Log Schema

Every log line must conform to this schema. Deviations are build failures in CI (log schema validation linter).

Required fields timestamp (ISO 8601 UTC), level (DEBUG/INFO/WARN/ERROR/FATAL), service_name, trace_id, span_id, tenant_id (always present unless platform-level with no tenant context), message (human-readable).
Conditional fields user_id (when acting on behalf of a human), event_type (on all Kafka consumer log lines -- the ADR-057 event_type), worker_name (on all worker log lines), duration_ms (on completed operations), error_code (on ERROR/FATAL -- machine-readable, not exception class name), module (the CampusOS module prefix e.g. 'sis', 'fin').
Prohibited Stack traces in message field (attach as error_detail object). PII in any field -- use IDs only, never names or emails. Free-form JSON blobs that are not schema-validated. Log lines without trace_id on the hot path.
Tooling AWS CloudWatch Logs as primary sink. OpenSearch for query and analysis. CI log schema linter rejects any log statement missing required fields.

27. Cross-Module Interaction Contract (v10)

These rules govern how modules communicate. They are the enforcement doctrine for module isolation.

Rule Definition
1. No Cross-Module Writes No module writes to another module's tables. If Module A needs to cause a state change in Module B, Module A emits a Kafka event and Module B's worker writes to its own tables. The only exception is M0 Platform Core — platform-level workers serve all modules.
2. Event-Driven Propagation Cross-module state changes propagate through Kafka events following ADR-057. Each event type has exactly one owning producer module. Consumer idempotency via platform_event_consumer_idempotency. DLQ after three failures.
3. Read Models Are Consumer-Owned rpt_ read models in M110 Analytics are materialised by background workers from source module tables via replica-routed queries. Source modules do not write to or read from rpt_ tables.
4. Reference Taxonomy (ADR-028) Three types: HARD FK (within module or to platform_), SOFT INTEGRITY (cross-module, app-layer, registered in platform_reference_health), DISPLAY-ONLY (no enforcement, comment required). Platform workers may do read-only PK lookups across modules.
5. Identity References (ADR-055) iam_person for human identity. platform_users for auth/session/audit. Domain projections (hr_employees.id, sis_students.id) for role context. Three categories are mutually exclusive per column.
