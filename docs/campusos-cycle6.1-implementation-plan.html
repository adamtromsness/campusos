Cycle 6.1: Profile & Household
Implementation Plan

A cross-cutting self-service feature that lets every persona maintain their own personal profile and edit shared household details. Slots between Cycles 6 and 7 as a Phase 2 polish pass on identity data.
Cycle 6.1 · April 2026 · 8 Steps · Extends platform.iam_person + platform.platform_families · Reuses sis_emergency_contacts / hr_emergency_contacts

Plan Overview

Cycles 1–6 stood up the academic, communications, HR, scheduling, and billing pipelines, but profile data has grown organically — iam_person still only carries first/last name + email, addresses live nowhere, and there is no parent-facing "edit my details" surface. Cycle 6.1 adds the missing self-service profile + household management layer without inventing new identity primitives.

The model splits cleanly into two halves. Personal fields (preferred name, mobile phone, personal email, language) hang off platform.iam_person — the canonical identity table per ADR-055. Shared household fields (address, home phone, mailing address) extend platform.platform_families — the existing platform-scoped families table that already groups David and Maya Chen. A new platform.platform_family_members link table adds a household role (HEAD_OF_HOUSEHOLD, SPOUSE, CHILD, etc.) so editing rights can be enforced. Emergency contacts continue to live in sis_emergency_contacts (non-employees) and hr_emergency_contacts (employees) — the profile UI just reads/writes those existing tables.

Vertical Slice Deliverable

David Chen logs in → clicks his avatar → "My Profile" → updates his preferred name to "Dave" and his mobile phone → switches to the "My Household" tab → edits the shared address to "1234 Oak Street" → saves → success toast. Maya Chen logs in → sees the same household address (read-only, role=CHILD) → opens her own profile → views demographics tab with date_of_birth pre-filled. James Rivera logs in → profile page reads from his hr_emergency_contacts row, not sis_emergency_contacts. An admin opens /profile/{Maya's personId} and edits Maya's first_name (the one field self-service cannot touch).

What's In / What's Deferred

In scope: Personal fields on iam_person (preferred name, middle name, suffix, previous names, DOB, three phone numbers + types, language, personal email, notes, profile_updated_at). Household fields on existing platform_families (address, home phone, home language, mailing address). New platform_family_members link table with role enum. New sis_student_demographics. Guardian employment fields on existing sis_guardians. usr-001 permission code for self-service profile read/write. Profile NestJS module + Households NestJS module. Profile UI with tabs (Personal / Household / Emergency / Demographics or Employment / Account). Avatar menu link.

Deferred: Multi-household membership (a person can only belong to one household this cycle — UNIQUE on platform_family_members.person_id). Mailing-address validation against postal services. Profile photo upload. Account-level password change UI (the page just deep-links to Keycloak). Multi-language UI (preferred_language is stored but the app remains English-only). Audit trail for name changes (the previous_names array is a simple TEXT[] this cycle — a proper history table is future work).

Schema Discipline (ADR-055, ADR-001/020)

iam_person remains the canonical identity table. New columns are additive and nullable so existing seed data keeps loading. platform_family_members.person_id is a real DB-enforced FK because both sides are in the platform schema. The tenant-side sis_student_demographics.student_id → sis_students(id) is a real intra-tenant FK. No cross-schema FKs are introduced.

Step-by-Step Build Plan

01 Platform Schema — iam_person + platform_families Extensions
Schema

One Prisma migration on the platform schema covers all three platform-side changes: new personal columns on iam_person, new household columns on the existing platform_families, and the new platform_family_members link table. Every new column is nullable (or has a sensible default) so existing rows keep loading.

Changes (one Prisma migration)
  1. Extend platform.iam_person — add 14 columns: middle_name TEXT, preferred_name TEXT, suffix TEXT, previous_names TEXT[], date_of_birth DATE, primary_phone TEXT, secondary_phone TEXT, work_phone TEXT, phone_type_primary TEXT CHECK IN ('MOBILE','HOME','WORK'), phone_type_secondary TEXT CHECK IN ('MOBILE','HOME','WORK'), preferred_language TEXT DEFAULT 'en', personal_email TEXT, notes TEXT, profile_updated_at TIMESTAMPTZ. All nullable. The email column on platform_users remains the login email and is unchanged.
  2. Extend platform.platform_families — add 11 columns: address_line1 TEXT, address_line2 TEXT, city TEXT, state TEXT, postal_code TEXT, country TEXT, home_phone TEXT, home_language TEXT DEFAULT 'en', mailing_address_same BOOLEAN DEFAULT true, plus mailing variants mailing_line1 / mailing_line2 / mailing_city / mailing_state / mailing_postal_code / mailing_country (all nullable, only consulted when mailing_address_same=false). One notes TEXT column for free-form household notes.
  3. New platform.platform_family_members — id UUID PK, family_id UUID FK(platform_families) ON DELETE CASCADE, person_id UUID FK(iam_person) UNIQUE (one household per person this cycle), role TEXT NOT NULL CHECK IN ('HEAD_OF_HOUSEHOLD','SPOUSE','CHILD','GRANDPARENT','OTHER_GUARDIAN','SIBLING','OTHER'), is_primary_contact BOOLEAN DEFAULT false, joined_at TIMESTAMPTZ DEFAULT now(), created_at / updated_at. UNIQUE(family_id, person_id) (belt-and-braces on top of the per-person UNIQUE). Partial UNIQUE INDEX on (family_id) WHERE is_primary_contact = true so each household has at most one primary contact.
Migration shape

Prisma migration is the right tool here because both iam_person and platform_families already live in the Prisma platform schema. Add the columns to schema.prisma, run prisma migrate dev against campusos_dev, commit the generated SQL. The migration is purely additive — no data backfill needed, no breaking enum/constraint changes.

Produces → 1 Prisma migration. iam_person grows from N to N+14 columns; platform_families grows by 16 columns; new platform_family_members table. Platform table count grows by 1.
02 Tenant Schema — Demographics & Guardian Employment
Schema

One tenant SQL migration adds the per-student demographics table and four employment columns to the existing sis_guardians table. Migration number follows the running sequence (currently 022 after Cycle 6 Step 4's 021).

Tables (1 new + 1 extended)
  1. sis_student_demographics (NEW) — id UUID PK, student_id UUID FK(sis_students) ON DELETE CASCADE, gender TEXT nullable, ethnicity TEXT nullable, primary_language TEXT nullable, birth_country TEXT nullable, citizenship TEXT nullable, medical_alert_notes TEXT nullable (a brief flag like "EpiPen in nurse's office"; not a full health record — that lives in M30 Health). UNIQUE(student_id) so each student has at most one demographics row. created_at / updated_at TIMESTAMPTZ.
  2. sis_guardians (EXTEND) — add 4 columns: employer TEXT, employer_phone TEXT, occupation TEXT, work_address TEXT. All nullable. relationship_to_student already exists from Cycle 1 — verify in the migration and skip if present.
Migration discipline

Use CREATE TABLE IF NOT EXISTS + ADD COLUMN IF NOT EXISTS per the Cycle 1+ idempotency convention. No ; inside any string literal or COMMENT block (the splitter cuts on every semicolon regardless of quoting context — tripped multiple times in Cycles 4–6). Pre-flight every migration with the existing Python audit script.

Produces → 1 SQL migration 022_sis_student_demographics_and_guardian_employment.sql. Tenant base table count: 107 (was 106 after Cycle 6 Step 4).
03 Permission Catalogue — usr-001
Seed

One new function USR-001 (Profile Management) with three tiers (read/write/admin) joins the 444-permission catalogue. Self-service is the design intent — every persona holds usr-001:read and usr-001:write for their own profile. The endpoint-level row-scope check (actor.personId === requestedPersonId) is the actual access gate; admins bypass via iam-001:write.

Changes
  1. packages/database/data/permissions.json — add usr-001:read, usr-001:write, usr-001:admin entries. Function name "Profile Management". Description "Self-service profile and household editing". The existing seed-iam.ts reconciler picks them up automatically.
  2. seed-iam.ts role-permission map — grant usr-001:read + usr-001:write to Teacher, Student, Parent, Staff, Counsellor, VP, School Admin, Platform Admin (every seeded role). Grant usr-001:admin to School Admin + Platform Admin only.
  3. Cache rebuild — tsx src/build-cache.ts after the seed update. Each persona's effective-access cache row count grows by 2; admins by 3.
Why a fresh function code

The existing iam-001 permission is already in use for admin-only person CRUD (creating accounts, editing names). Reusing it for self-service profile editing would force every parent / student to hold iam-001:read, leaking name-change capability. A separate usr-001 code keeps self-service distinct from admin person management. iam-001:write remains the admin override for editing fields the user themself cannot change (first_name, last_name).

Produces → 3 new permission codes (catalogue total 447). Updated effective access cache. No code changes — the new codes wire into @RequirePermission calls in Steps 5–6.

05 Profile NestJS Module
Code

The self-service profile API. One service + one controller + 4 endpoints. Persona-aware response composition: every GET /profile returns the iam_person fields plus the persona-specific data (demographics for STUDENT, guardian employment for GUARDIAN, employee profile for STAFF) plus the relevant emergency contact row.

Service & endpoints
  1. GET /profile/me — gated on usr-001:read. Resolves actor.personId from ActorContextService.resolveActor(...) and returns the full profile shape: iam_person columns + (if STUDENT) sis_student_demographics row + (if GUARDIAN) sis_guardians employment columns + (if STAFF) hr_employees primary position + emergency contact (read from sis_emergency_contacts by default; hr_emergency_contacts when actor.employeeId is non-null) + household membership (the caller's platform_family_members row + the linked platform_families).
  2. PATCH /profile/me — gated on usr-001:write. Accepts a partial payload of personal fields. Server-side ALLOW-LIST: preferred_name, middle_name, suffix, previous_names, primary_phone, secondary_phone, work_phone, phone_type_primary, phone_type_secondary, preferred_language, personal_email, notes. Cannot change first_name, last_name, email (login), or date_of_birth after initial set (admin-only per ADR-055). Guardian-extras path: if persona is GUARDIAN, also accepts employer / employer_phone / occupation / work_address. Student demographics path: if persona is STUDENT, accepts primary_language only (other demographics are admin-only). Updates profile_updated_at. Phone type CHECK constraints from Step 1 are the schema-side belt-and-braces.
  3. GET /profile/:personId — admin view, gated on iam-001:read. Returns the same shape as /profile/me but for an arbitrary person. School admins see anyone in their school; platform admins see anyone.
  4. PATCH /profile/:personId — admin edit, gated on iam-001:write. Full ALLOW-LIST including first_name, last_name, date_of_birth (the fields self-service cannot touch). Logs an audit row in the future iam_admin_access_log — deferred this cycle, for now just sets profile_updated_at.
Emergency-contact branching

The profile response embeds an optional emergencyContact object. Resolution rule: if actor.employeeId is non-null (or for admin-view-of-someone-else, if the target person has an hr_employees row), read from hr_emergency_contacts WHERE employee_id = …. Otherwise read from sis_emergency_contacts WHERE person_id = …. PATCH writes back to the same source. No data is duplicated. The UI is unaware of which table is being used.

Validation

DTOs use class-validator with @IsIn on the phone type enum (kept in sync with the SQL CHECK constraints). @IsEmail on personal_email. @IsISO8601() on date_of_birth (admin path only). String length caps on free-form fields (notes 2000 chars, name fields 100 chars). Required-field check is endpoint-policy not schema-policy — the schema keeps everything nullable for backward compat with existing rows; the PATCH endpoint refuses to clear primary_phone only on persons whose row already has one populated (no breakage of old rows, but you can't delete a phone you've already added through the UI).

Produces → ProfileModule with 1 service + 1 controller + 4 endpoints. Mounts under AppModule after IamModule.
06 Households NestJS Module
Code

Household read + edit, with role-based gating that's enforced at the service layer rather than via IAM permission codes. The trick: household editing is not a tenant-scoped admin function — it's family self-service, so the gate is "are you HEAD_OF_HOUSEHOLD or SPOUSE in this household" (or admin via iam-001:write).

Service & endpoints
  1. GET /households/my — gated on usr-001:read. Resolves actor.personId, looks up platform_family_members for that person, returns the household with all members (each member shape: name from iam_person, role, is_primary_contact, joined_at). Students/children get the same payload but canEdit=false in the response so the UI hides edit controls.
  2. PATCH /households/:id — gated on usr-001:write + service-layer assertCanEditHousehold(id, actor) check. Updates the shared fields: address_line1 / line2 / city / state / postal_code / country / home_phone / home_language / mailing_address_same / mailing_*. Refuses with 403 if the actor is neither HEAD_OF_HOUSEHOLD nor SPOUSE in this household and lacks iam-001:write.
  3. POST /households/:id/members — same gate as PATCH. Adds a person to the household. Validates the person isn't already in another household (UNIQUE on person_id would 23505 anyway; this check is for the friendlier 409 response).
  4. PATCH /households/:id/members/:memberId — same gate. Updates role or is_primary_contact. Promoting a new primary contact atomically clears the old one in the same tx (the partial UNIQUE INDEX from Step 1 is the schema-side belt-and-braces).
  5. DELETE /households/:id/members/:memberId — same gate. Refuses if the member being removed is the last HEAD_OF_HOUSEHOLD (loud 400 with the message "Households must always have at least one head of household. Promote another member first."). Cannot remove the calling user themself unless they have iam-001:write — this avoids accidental self-eviction.
Locked-read concurrency

Every PATCH/POST/DELETE wraps the read-then-write in executeInTenantTransaction (or for platform-only writes, a regular Prisma transaction) with a SELECT … FOR UPDATE on the household row. Two concurrent “promote me to primary contact” clicks have to serialise; the partial UNIQUE INDEX on (family_id) WHERE is_primary_contact = true is the schema-side fallback. This pattern is the Cycles 4–6 review carry-over: every state-machine transition must lock the row inside the transaction.

Kafka emit

Member changes (add / remove / role change / primary-contact flip) emit iam.household.member_changed via KafkaProducerService.emit({sourceModule:'iam'}). No consumer in this cycle — the topic is forward-compatible for a future M40 announcement that fans out a "your household member changed" notification.

Produces → HouseholdsModule with 1 service + 1 controller + 5 endpoints + 1 Kafka emit. Mounts under AppModule alongside ProfileModule.
07 Profile UI — Tabbed Page + Avatar Menu
UI

One new route at /profile with five tabs, plus the avatar dropdown integration. No new app tile on the launchpad — the profile is intentionally accessed only via the avatar menu, matching iOS / SaaS convention and keeping the launchpad clean.

Routes & tabs
  1. /profile — client-side tabbed view. Tabs are persona-conditional (Demographics shows only for STUDENT; Employment shows only for GUARDIAN; Account is visible to everyone).
    • Personal Info — Name section (first/last read-only with a "Contact admin to change" hint, middle, suffix, preferred name, previous names as a chip list). Contact section (primary phone required *, secondary, work, each with type selector). Personal email. Language preference dropdown. Notes textarea. Per-field validation with inline error text. Save button at the bottom of the tab.
    • My Household — Household name + shared address + home phone + home language + mailing address (toggle "same as home"). Member list with name + role badge + primary-contact star. canEdit from the API drives whether the form is editable or read-only. Add-member modal (HEAD/SPOUSE only) with person picker + role dropdown. Per-member role dropdown.
    • Emergency Contact — Name + phone + relationship. Single record per person. PATCH writes through to sis_emergency_contacts or hr_emergency_contacts depending on persona; the UI is unaware.
    • Demographics (STUDENT only) — date_of_birth (read-only after set), gender, ethnicity, primary_language, birth_country, citizenship, medical_alert_notes. Self-service can edit primary_language; the rest are admin-only (greyed out with "Admin only" hint). Admin viewing this same tab via /profile/:personId sees them all editable.
    • Employment (GUARDIAN only) — employer, employer_phone, occupation, work_address.
    • Account — Login email (read-only). Password change link that deep-links to Keycloak ({KEYCLOAK_URL}/realms/{realm}/account). Profile completeness percentage bar.
  2. /profile/:personId — admin view of any person's profile. Same tab layout but: the Personal tab's first/last are editable (one extra "Identity (admin only)" sub-section); the Demographics tab's full row is editable.
  3. Avatar dropdown — apps/web/src/components/shell/TopBar.tsx avatar menu gains a "My Profile" link above the existing "Sign out" item. Visible to every persona that holds usr-001:read (which is everyone after Step 3).
Profile completeness formula

Completeness = (filled required fields + filled recommended fields) / (all required + all recommended). Required (weight 1): first_name, last_name, primary_phone, household role (if a member of one). Recommended (weight 0.5): preferred_name, address (any line1), emergency contact name+phone, employment (if GUARDIAN), demographics primary_language (if STUDENT). The formula lives in a single helper profileCompleteness(profile, persona) in apps/web/src/lib/profile-format.ts so it's testable in isolation.

React Query hooks

New apps/web/src/hooks/use-profile.ts with useMyProfile, useProfile(personId), useUpdateMyProfile, useUpdateProfile(personId), useMyHousehold, useUpdateHousehold(id), useAddHouseholdMember(id), useUpdateHouseholdMember(id, memberId), useRemoveHouseholdMember(id, memberId). 30s staleTime on reads, mutation invalidations cover ['profile', 'me'] + ['household', 'mine'].

UI design discipline

Follows the existing principles: clean layout, no clutter, save button per tab with toast feedback, required fields marked with *, validation errors inline. No new icons needed — the avatar menu uses the existing UserCircleIcon. Match the typographic hierarchy of /staff/[id] (Cycle 4) which already does tabbed person detail.

Produces → 2 new routes (/profile, /profile/:personId), 1 hooks file, 1 helper module, avatar menu link. No new app tile.
