Plan Overview

Cycle 10 builds the M23 Health module — 14 of the 17 ERD tables (telehealth deferred). This is the most access-restricted module in the system: hlth_* tables require a dedicated health_record:read permission. Teachers never read hlth_* directly — they read sis_student_active_accommodations (the ADR-030 read model) populated by a Kafka consumer when IEP accommodations change. Parents see their own child's health summary, immunisation status, and medication schedule. The school nurse is the primary operator.

The module spans 5 domains: (1) core health records + conditions + immunisations, (2) medication management with scheduling and missed-dose audit, (3) nurse visits with live office roster, (4) IEP/504 plans with goals, services, and accommodation sync, (5) screenings + dietary profiles. Each domain has its own schema step so the migration sequence is clear.

Data Sensitivity — FERPA + HIPAA-Adjacent

All hlth_* tables are flagged for a separate HIPAA-compliant KMS key in the ERD. For the dev/demo phase, the tables ship without field-level encryption but the access control layer is strict from day one: a dedicated health_record:read permission gates every read endpoint. An immutable hlth_health_access_log records every access for HIPAA audit. Teachers see ONLY the sis_student_active_accommodations read model — they never query hlth_* tables.

Vertical Slice Deliverable

Nurse creates Maya's health record with blood type + allergies (peanuts, SEVERE) → adds an active medical condition (asthma, MODERATE) with management plan → records 3 immunisations (2 CURRENT, 1 OVERDUE) → sets up a daily medication (albuterol inhaler, PRN) with a morning schedule slot → logs today's administration → logs a nurse visit (Maya came in wheezing, treatment given, parent notified, not sent home) → creates an IEP plan with 2 goals + 1 accommodation (EXTENDED_TIME for ALL_ASSESSMENTS) → iep.accommodation.updated fires → sis_student_active_accommodations updated via Kafka consumer (ADR-030) → teacher sees the accommodation on Maya's classroom profile without touching hlth_* → nurse runs a vision screening on Maya → screening flagged for follow-up → dietary profile created (peanut allergy, SEVERE, with POS allergen alert flag) → parent sees Maya's health summary with immunisation compliance and medication schedule.

What's In / What's Deferred

In scope (14 tables): hlth_student_health_records, hlth_medical_conditions, hlth_immunisations, hlth_medications, hlth_medication_schedule, hlth_medication_administrations, hlth_nurse_visits, hlth_iep_plans, hlth_iep_goals, hlth_iep_goal_progress, hlth_iep_services, hlth_iep_accommodations, hlth_screenings, hlth_dietary_profiles. Plus hlth_health_access_log (HIPAA audit). Functions HLT-001 through HLT-005.

Deferred (3 tables): hlth_telehealth_providers, hlth_telehealth_sessions, hlth_telehealth_documents (require video platform integration + platform_signature_requests — future cycle). Screening schedule auto-generation (manual this cycle). Field-level encryption (tagged in ERD for HIPAA KMS key — access control ships now, encryption is Phase 3 ops). Nurse-visit pattern alerting (threshold-based — future). Medication inventory management (out of ERD scope).

Step-by-Step Build Plan

01 Health Records Schema — Records, Conditions, Immunisations
Schema

The core health identity: one record per student, with linked conditions and immunisation tracking. Plus the HIPAA access log that every health read operation writes to.

Tables (4)
  1. hlth_student_health_records — student_id UUID FK(sis_students) UNIQUE (one record per student), school_id UUID, blood_type TEXT nullable, allergies JSONB (structured: [{allergen, severity, reaction, notes}]), emergency_medical_notes TEXT, physician_name TEXT, physician_phone TEXT. The master health record. Created by the nurse; updated as conditions evolve.
  2. hlth_medical_conditions — health_record_id UUID FK, condition_name TEXT NOT NULL, diagnosis_date DATE, is_active BOOLEAN, severity ENUM(MILD, MODERATE, SEVERE), management_plan TEXT. INDEX(health_record_id, is_active).
  3. hlth_immunisations — health_record_id UUID FK, vaccine_name TEXT NOT NULL, administered_date DATE, due_date DATE, administered_by TEXT, status ENUM(CURRENT, OVERDUE, WAIVED). INDEX(health_record_id, vaccine_name). The compliance dashboard aggregates OVERDUE counts.
  4. hlth_health_access_log — IMMUTABLE. school_id UUID, accessed_by UUID (soft to platform_users), student_id UUID FK(sis_students), access_type ENUM(VIEW_RECORD, VIEW_CONDITIONS, VIEW_IMMUNISATIONS, VIEW_MEDICATIONS, VIEW_VISITS, VIEW_IEP, VIEW_SCREENING, VIEW_DIETARY, EXPORT), accessed_at TIMESTAMPTZ DEFAULT now(), ip_address TEXT nullable. INDEX(student_id, accessed_at DESC). INDEX(accessed_by, accessed_at DESC). No UPDATE, no DELETE. Every health read endpoint writes a row here before returning data.
Produces → 4 new tables. Migration: 032_hlth_health_records.sql.
02 Medication Schema — Meds, Schedule, Administration
Schema

The medication management domain: prescribed medications linked to health records, daily schedule slots, and dose administration logs with missed-dose audit tracking.

Tables (3)
  1. hlth_medications — health_record_id UUID FK, medication_name TEXT NOT NULL, dosage TEXT, frequency TEXT, route ENUM(ORAL, TOPICAL, INHALER, INJECTION, OTHER), prescribing_physician TEXT, is_self_administered BOOLEAN, is_active BOOLEAN DEFAULT true. INDEX(health_record_id, is_active).
  2. hlth_medication_schedule — medication_id UUID FK, scheduled_time TIME NOT NULL, day_of_week SMALLINT nullable (0–6; null = every day), notes TEXT. The daily schedule that the nurse dashboard renders as a checklist. INDEX(medication_id).
  3. hlth_medication_administrations — medication_id UUID FK, schedule_entry_id UUID FK(hlth_medication_schedule) nullable (null for PRN/unscheduled), administered_by UUID FK(hr_employees), administered_at TIMESTAMPTZ nullable (null when was_missed=true), dose_given TEXT, notes TEXT, parent_notified BOOLEAN DEFAULT false, was_missed BOOLEAN DEFAULT false, missed_reason ENUM(STUDENT_ABSENT, STUDENT_REFUSED, MEDICATION_UNAVAILABLE, PARENT_CANCELLED, OTHER) nullable. CHECK: NOT (was_missed = true AND administered_at IS NOT NULL). INDEX(medication_id, administered_at DESC). Partial INDEX(schedule_entry_id, was_missed) WHERE was_missed = true — missed dose audit query.
Produces → 3 new tables. Migration: 033_hlth_medications.sql.
03 IEP/504 + Nurse + Screening + Dietary Schema
Schema

The remaining 7 tables: the IEP/504 plan domain (5 tables with accommodation sync), nurse visits with live roster, screenings, and dietary profiles.

Tables (8)
  1. hlth_iep_plans — student_id UUID FK(sis_students) UNIQUE (one active plan per student), plan_type ENUM(IEP, 504), status ENUM(DRAFT, ACTIVE, REVIEW, EXPIRED), start_date DATE, review_date DATE, end_date DATE, case_manager_id UUID FK(hr_employees). On plan changes: emit iep.accommodation.updated Kafka event → sis_student_active_accommodations updated by SIS consumer (ADR-030).
  2. hlth_iep_goals — iep_plan_id UUID FK, goal_text TEXT NOT NULL, measurement_criteria TEXT, baseline TEXT, target_value TEXT, current_value TEXT, goal_area TEXT, status ENUM(ACTIVE, MET, NOT_MET, DISCONTINUED).
  3. hlth_iep_goal_progress — goal_id UUID FK, recorded_by UUID FK, progress_value TEXT, observation_notes TEXT, recorded_at TIMESTAMPTZ. INDEX(goal_id, recorded_at DESC).
  4. hlth_iep_services — iep_plan_id UUID FK, service_type TEXT (speech, OT, PT, counselling...), provider_name TEXT, frequency TEXT, minutes_per_session INT, delivery_method ENUM(PULL_OUT, PUSH_IN, CONSULT).
  5. hlth_iep_accommodations — iep_plan_id UUID FK, accommodation_type TEXT (EXTENDED_TIME, ALTERNATIVE_ASSESSMENT, ASSISTIVE_TECH, READ_ALOUD, REDUCED_DISTRACTION...), description TEXT, applies_to ENUM(ALL_ASSESSMENTS, ALL_ASSIGNMENTS, SPECIFIC), specific_assignment_types TEXT[] nullable, effective_from DATE, effective_to DATE. On INSERT/UPDATE/DELETE: the parent IepPlanService emits iep.accommodation.updated.
  6. hlth_nurse_visits — visited_person_id UUID NOT NULL (soft — sis_students.id or hr_employees.id per visited_person_type), visited_person_type ENUM(STUDENT, STAFF) DEFAULT 'STUDENT', school_id UUID, nurse_id UUID FK(hr_employees), visit_date TIMESTAMPTZ, status ENUM(IN_PROGRESS, COMPLETED) DEFAULT 'COMPLETED', signed_in_at TIMESTAMPTZ, signed_out_at TIMESTAMPTZ nullable, reason TEXT, treatment_given TEXT, parent_notified BOOLEAN, sent_home BOOLEAN, sent_home_at TIMESTAMPTZ, follow_up_required BOOLEAN, follow_up_notes TEXT, follow_up_date DATE. Partial INDEX(school_id, status) WHERE status = 'IN_PROGRESS' — live nurse office roster.
  7. hlth_screenings — school_id UUID, student_id UUID FK(sis_students), screening_type TEXT NOT NULL (VISION, HEARING, SCOLIOSIS, BMI, DENTAL, CUSTOM), screening_date DATE NOT NULL, screened_by UUID FK(hr_employees), result ENUM(PASS, REFER, RESCREEN, ABSENT), result_notes TEXT, follow_up_required BOOLEAN, follow_up_completed BOOLEAN DEFAULT false, referral_notes TEXT. INDEX(student_id, screening_date DESC).
  8. hlth_dietary_profiles — student_id UUID FK(sis_students) UNIQUE (one profile per student), school_id UUID, dietary_restrictions TEXT[] (VEGETARIAN, VEGAN, HALAL, KOSHER, GLUTEN_FREE, DAIRY_FREE...), allergens JSONB (structured: [{allergen, severity, reaction}]), special_meal_instructions TEXT, pos_allergen_alert BOOLEAN DEFAULT false (when true: POS system shows a hard-stop alert), updated_by UUID (soft). INDEX(school_id, pos_allergen_alert) WHERE pos_allergen_alert = true.
Produces → 8 new tables. Migration: 034_hlth_iep_nurse_screening_dietary.sql. Total Cycle 10 tables: 15 (including the access log).

Code
Services & Endpoints

GET /health/medications/:id/administrations (dose history). GET /health/medication-dashboard (nurse: today's schedule across all students with administered/pending/missed status per slot).

Services & Endpoints
