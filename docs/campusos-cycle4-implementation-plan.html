Cycle 4: HR & Workforce Core
Implementation Plan

Employee records, positions, leave management, certifications, and training compliance. Resolves the temporary HR-employee identity mapping from Cycle 2 and adds the Staff Directory app to the launchpad.
April 2026 · 10 Steps · ERD v11 · M80 HR/Workforce (Core Subset)

Plan Overview

Cycle 4 builds the M80 HR/Workforce module — or rather, the core of it. The full ERD defines 48 HR tables; this cycle implements the 20 most essential ones that deliver immediate value: employee records, positions, leave management, certifications, training compliance, and the staff directory. Recruitment, payroll, benefits enrollment, appraisals, grievances, and workers' comp are deferred to Cycle 4b or later.

Vertical Slice Deliverable

Admin creates an employee record for James Rivera → assigns him to the "Teacher" position → Rivera views his own profile with certifications and leave balances → Rivera submits a sick leave request → admin approves it → coverage notification fires for affected classes → Rivera's leave balance updates → compliance dashboard shows his certification expiry in 60 days.

Why Cycle 4 Matters

Identity resolution: Since Cycle 2, every teacher_employee_id, cls_grades.teacher_id, and cls_student_progress_notes.author_id column has held iam_person.id directly because hr_employees didn't exist yet. This cycle creates the employee table with person_id FK(iam_person), seeds employee records for all staff, and runs an additive migration that re-points the soft FKs. The "Temporary HR-Employee Identity Mapping" documented in CLAUDE.md is finally resolved.

Leave → coverage: When a teacher takes leave, the system needs to know which classes are affected. The SIS module already has sis_class_teachers. This cycle connects leave requests to affected classes via a Kafka event, which Cycle 5 (Scheduling) can consume to trigger substitute assignment.

What's In / What's Deferred

In scope: hr_employees, hr_positions, hr_employee_positions, hr_departments (reuses sis_departments), hr_leave_types, hr_leave_balances, hr_leave_requests, hr_staff_certifications, hr_training_requirements, hr_training_compliance, hr_cpd_requirements, hr_work_authorisation, hr_emergency_contacts, hr_onboarding_templates, hr_onboarding_checklists, hr_onboarding_tasks, hr_document_types, hr_employee_documents. Functions HR-001, HR-003, HR-004, HR-006 (core only).

Deferred: hr_job_postings, hr_applications, hr_interview_schedules, hr_offer_letters (HR-002 Recruitment — Cycle 4b). hr_pay_periods, hr_payroll_records, hr_salary_components, hr_deductions, hr_tax_information (payroll — separate cycle). hr_benefit_enrolment_periods, hr_benefit_plans, hr_benefit_elections (benefits — separate cycle). hr_appraisals, hr_appraisal_responses, hr_classroom_observations (HR-005 PD — Cycle 4b). hr_grievances, hr_grievance_actions, hr_disciplinary_actions (employee relations — separate cycle). hr_workers_comp_claims, hr_return_to_work_plans (workers comp — separate cycle).

Step-by-Step Build Plan

00 HR-Employee Identity Migration
Infrastructure

This is the step that resolves the Cycle 2 carry-over. Create the hr_employees table, seed employee records for the 5 test staff users, then run an additive migration that re-points the soft FK columns that currently hold iam_person.id to hold hr_employees.id instead.

What Changes
  1. Create hr_employees in the tenant migration. Core columns: id UUID PK, person_id UUID NOT NULL UNIQUE (soft FK to iam_person), account_id UUID NOT NULL UNIQUE (soft FK to platform_users), school_id UUID, employee_number TEXT, employment_type, employment_status, hire_date DATE.
  2. Seed employee records for: admin (Sarah Mitchell), principal (same), teacher (James Rivera), and any other staff users. Each row links person_id to their iam_person and account_id to their platform_users.
  3. Re-point soft FKs. Run UPDATE statements to replace iam_person.id values with the corresponding hr_employees.id in: sis_class_teachers.teacher_employee_id, cls_grades.teacher_id, cls_student_progress_notes.author_id, cls_lessons.teacher_id. Since hr_employees.person_id = iam_person.id, the join is straightforward.
  4. Update all service code that compares these columns against actor.personId. The new comparison is: column_value = (SELECT id FROM hr_employees WHERE person_id = $actor.personId). Or better: resolve employeeId once in ActorContextService and carry it on the actor object.
  5. Remove the "Temporary HR-Employee Identity Mapping" section from CLAUDE.md. Replace with a permanent convention: "Staff columns reference hr_employees.id. Resolve via actor.employeeId from ActorContextService."
Produces → hr_employees table with seeded records. All soft FKs resolved. Identity mapping complete.
01 HR Schema — Employees & Positions
Schema

The employee core: positions, position assignments (time-bounded), emergency contacts, and document management.

Tables (5, plus hr_employees from Step 0)
  1. hr_positions — school_id, title TEXT NOT NULL, department_id UUID (soft FK to sis_departments), is_teaching_role BOOLEAN. UNIQUE(school_id, title).
  2. hr_employee_positions — employee_id FK, position_id FK, effective_from DATE, effective_to DATE. INDEX(employee_id, effective_from DESC). Tracks position history.
  3. hr_emergency_contacts — employee_id FK, name TEXT NOT NULL, relationship TEXT, phone TEXT NOT NULL, is_primary BOOLEAN. Separate from sis_emergency_contacts (student emergency contacts).
  4. hr_document_types — school_id, name TEXT NOT NULL, is_required BOOLEAN, retention_days INT. UNIQUE(school_id, name). Examples: Contract, ID Copy, Tax Form, Background Check.
  5. hr_employee_documents — employee_id FK, document_type_id FK, file_name TEXT, s3_key TEXT, uploaded_by UUID (soft), uploaded_at TIMESTAMPTZ, expiry_date DATE. INDEX(employee_id, document_type_id).
Produces → 5 new tables + hr_employees from Step 0. Migration: 011_hr_employees_and_positions.sql.
02 HR Schema — Leave Management
Schema

Leave types, per-employee balances, and leave requests with approval workflow.

Tables (3)
  1. hr_leave_types — school_id, name TEXT NOT NULL, is_paid BOOLEAN, accrual_rate NUMERIC(5,2), max_balance NUMERIC(5,2). Examples: Sick Leave, Personal Leave, Bereavement, Professional Development.
  2. hr_leave_balances — UNIQUE(employee_id, leave_type_id, academic_year_id). accrued NUMERIC(5,2), used NUMERIC(5,2), pending NUMERIC(5,2). Updated by the leave request approval workflow.
  3. hr_leave_requests — employee_id FK, leave_type_id FK, start_date DATE, end_date DATE, days_requested NUMERIC(4,1), status ENUM(PENDING, APPROVED, REJECTED, CANCELLED), reason TEXT, is_hr_initiated BOOLEAN DEFAULT false, hr_initiated_by UUID (soft), hr_initiated_reason TEXT. INDEX(employee_id, status).
Leave Approval Flow

Employee submits → status=PENDING, pending balance incremented → Kafka event hr.leave.requested fires → admin approves → status=APPROVED, pending decremented, used incremented → Kafka event hr.leave.approved fires (consumed by Cycle 5 Scheduling for coverage). Rejection: status=REJECTED, pending decremented back. Cancellation: same reversal.

Produces → 3 new tables. Migration: 012_hr_leave_management.sql.
03 HR Schema — Certifications & Training
Schema

Professional certifications, mandatory training requirements, compliance tracking, CPD (Continuing Professional Development) hours, and work authorisation.

Tables (5)
  1. hr_staff_certifications — employee_id FK, certification_type ENUM(TEACHING_LICENCE, FIRST_AID, SAFEGUARDING_LEVEL1, SAFEGUARDING_LEVEL2, DBS_BASIC, DBS_ENHANCED, FOOD_HYGIENE, FIRE_SAFETY_WARDEN, SPECIALIST_SUBJECT, CUSTOM), certification_name TEXT, issuing_body TEXT, reference_number TEXT, issued_date DATE, expiry_date DATE, verification_status ENUM(PENDING, VERIFIED, EXPIRED, REVOKED), document_s3_key TEXT. Alert job: 90, 30, 7 days before expiry. Follows ADR-015 for DBS: store reference + status only.
  2. hr_training_requirements — school_id, position_id FK nullable (null = all staff), training_name TEXT, certification_type (links to hr_staff_certifications for auto compliance), frequency ENUM(ONE_TIME, ANNUAL, BIENNIAL, TRIENNIAL, CUSTOM), is_active BOOLEAN.
  3. hr_training_compliance — Materialised per-employee per-requirement compliance status. Updated nightly by TrainingComplianceWorker. UNIQUE(employee_id, requirement_id). is_compliant BOOLEAN, last_completed_date DATE, next_due_date DATE, linked_certification_id FK, days_until_due INT.
  4. hr_cpd_requirements — UNIQUE(school_id, position_id, academic_year_id). required_pd_hours NUMERIC(4,1), required_credit_hours NUMERIC(4,1).
  5. hr_work_authorisation — UNIQUE(employee_id). document_type ENUM(US_PASSPORT, UK_PASSPORT, UK_BRP, PERMANENT_RESIDENT_CARD, EMPLOYMENT_AUTHORISATION, OTHER), expiry_date DATE, verified_by UUID, verified_at DATE, reverification_due_date DATE.
Produces → 5 new tables. Migration: 013_hr_certifications_and_training.sql.
04 HR Schema — Onboarding
Schema

New hire onboarding with template-based checklists and task tracking.

Tables (3)
  1. hr_onboarding_templates — school_id, name TEXT, position_id FK nullable (null = generic), is_active BOOLEAN. UNIQUE(school_id, name). Template for what a new hire needs to complete.
  2. hr_onboarding_checklists — employee_id FK, template_id FK, status ENUM(NOT_STARTED, IN_PROGRESS, COMPLETED), started_at, completed_at, assigned_by UUID (soft). One checklist per new hire.
  3. hr_onboarding_tasks — checklist_id FK, title TEXT, description TEXT, category ENUM(DOCUMENT, TRAINING, SYSTEM_ACCESS, ORIENTATION, OTHER), is_required BOOLEAN, due_days_from_start INT, status ENUM(PENDING, IN_PROGRESS, COMPLETED, SKIPPED), completed_at, completed_by UUID. INDEX(checklist_id, status).
Produces → 3 new tables. Migration: 014_hr_onboarding.sql.

Populate the HR module with realistic demo data.

What Gets Seeded

06 HR NestJS Module — Employee Records & Directory
Code

The employee management API and staff directory.

Services & Endpoints
  1. EmployeeService — CRUD employees. GET /employees (staff directory — list all active employees with position, department, contact), GET /employees/:id (full profile), GET /employees/me (own profile), POST /employees (admin creates), PATCH /employees/:id (admin updates). All @RequirePermission('hr-001:read') or 'hr-001:write'.
  2. PositionService — GET /positions, POST /positions, PATCH /positions/:id. Admin only (hr-001:admin).
  3. EmployeeDocumentService — GET /employees/:id/documents, POST /employees/:id/documents (upload), DELETE /employees/:id/documents/:docId. Own documents readable by the employee; all documents readable by admin.
  4. ActorContextService extension — resolveActor() now resolves employeeId from hr_employees where person_id = actor.personId. This replaces the previous direct personId comparison for all class-teacher, grade, and progress-note authorization checks.
Row-Level Authorization

Employees can view the staff directory (everyone) and their own full profile. Admins can view and edit all employees. Teachers cannot view other employees' documents or leave details (only their own).

Produces → HRModule with EmployeeService, PositionService, EmployeeDocumentService. ~10 endpoints.
07 HR NestJS Module — Leave & Certifications
Code

Leave request workflow, certification management, and training compliance tracking.

Services & Endpoints
  1. LeaveService — POST /leave-requests (submit — validates against balance), GET /leave-requests (own history or admin queue), PATCH /leave-requests/:id/approve (admin), PATCH /leave-requests/:id/reject (admin), PATCH /leave-requests/:id/cancel (employee cancels own). Kafka events: hr.leave.requested, hr.leave.approved, hr.leave.rejected, hr.leave.cancelled. All @RequirePermission('hr-003:read') or 'hr-003:write'.
  2. CertificationService — GET /employees/:id/certifications, POST /employees/:id/certifications, PATCH /certifications/:id/verify (admin verifies). Expiry alerts via hr.certification.expiring Kafka event (consumed by notification pipeline). @RequirePermission('hr-004:read') or 'hr-004:write'.
  3. TrainingComplianceService — GET /compliance/dashboard (admin view — all employees' compliance status), GET /employees/:id/compliance (individual view). Recomputes compliance status by comparing hr_staff_certifications against hr_training_requirements.
  4. LeaveNotificationConsumer — Kafka consumer on hr.leave.approved. Resolves affected classes via sis_class_teachers for the leave date range. Emits hr.leave.coverage_needed with class IDs and dates (consumed by Scheduling in Cycle 5). Also notifies the employee of approval/rejection.
Produces → LeaveService, CertificationService, TrainingComplianceService, LeaveNotificationConsumer. ~12 endpoints + 5 Kafka events.
08 Staff Directory & Employee Profile UI
UI

A new "Staff" app tile on the home page launchpad. The staff directory is the phone book of the school.

Pages
  1. Staff Directory /staff — searchable list of all employees. Each card: name, position, department, email, phone. Search filters by name, position, department. Visible to all staff (hr-001:read).
  2. Employee Profile /staff/:id — tabbed view: Info (demographics, position history, emergency contacts), Certifications (list with expiry dates, verification status, color-coded by urgency), Leave (balances, request history), Documents (uploaded files). Own profile shows all tabs. Others' profiles show only Info tab unless the viewer is an admin.
  3. My Profile shortcut /staff/me — redirects to /staff/:myEmployeeId.
Produces → Staff Directory app, employee profile with tabs. Launchpad gets a "Staff" tile for users with hr-001:read.
09 Leave Management & Compliance Dashboard UI
UI

Leave request forms, approval queue, and the training compliance dashboard.

Pages
  1. Request Leave /leave/new — form: leave type dropdown, date range picker, days requested (auto-calculated, supports half-days), reason text. Shows current balance for the selected type. Submit creates a PENDING request.
  2. My Leave /leave — employee view of their leave balances (by type, visual bar showing used/remaining) and request history (with status badges). Cancel button on PENDING requests.
  3. Leave Approval Queue /leave/approvals (admin only) — list of PENDING requests. Each shows: employee name, type, dates, days, reason. Approve/Reject buttons with optional response note.
  4. Compliance Dashboard /compliance (admin only) — table of all employees with their compliance status across all training requirements. Color-coded: green (compliant), amber (expiring within 90 days), red (non-compliant/expired). Filter by department, position, compliance status. Click an employee to see their certification detail.
Produces → Leave request + history + approval queue. Compliance dashboard. Two new app tiles: "Leave" for all staff, "Compliance" for admins.
10 Vertical Slice Integration Test
Test

End-to-end proof that HR workflows work.

Test Script
  1. Admin creates employee: Log in as admin. Navigate to Staff → create a new employee. Verify they appear in the directory.
  2. Employee views profile: Log in as teacher. Navigate to My Profile. Verify Info tab shows position, hire date, emergency contacts. Verify Certifications tab shows Teaching Licence (expiring in 60 days, amber). Verify Leave tab shows balances.
  3. Employee submits leave: Teacher submits a 1-day sick leave request for next week. Verify balance shows pending incremented. Verify request appears in history as PENDING.
  4. Kafka event fires: Verify hr.leave.requested event emitted.
  5. Admin approves: Log in as admin. Navigate to Leave Approvals. See the pending request. Approve it. Verify status changes to APPROVED.
  6. Balance updates: Verify teacher's used incremented, pending decremented. Verify hr.leave.approved event emitted.
  7. Notification: Teacher receives a notification that their leave was approved.
  8. Compliance dashboard: Admin navigates to Compliance. Verify Rivera shows amber for Teaching Licence (60 days to expiry). Verify overall compliance percentage reflects the non-compliant item.
  9. Permission denial: Teacher cannot access Leave Approvals or edit another employee's record. Parent/student cannot access the Staff module at all.
Produces → Verified HR vertical slice. Identity mapping resolved. Leave workflow end-to-end. Compliance tracking active.
