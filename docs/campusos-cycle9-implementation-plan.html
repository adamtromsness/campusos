Cycle 9: Behaviour & Discipline
Implementation Plan

The first student-services module. Incident documentation, disciplinary actions with parent notification, and Behaviour Intervention Plans (BIPs) with measurable goals and teacher feedback. Bridges from the existing sis_student_notes (informal) to formal discipline records and behaviour support plans.
May 2026 · 10 Steps · ERD v11 · M20 SIS Discipline (4 tables) + M27 Behaviour Plans (3 tables)

Plan Overview

Cycle 9 spans two ERD modules: the 4 discipline tables from M20 SIS Core (deferred from Cycle 1 since they weren't needed for the attendance vertical slice) and 3 behaviour plan tables from M27 Student Services. Together they deliver the complete behaviour management lifecycle: a teacher logs an incident → admin reviews and assigns consequences → parents are notified when required → repeated incidents trigger a formal Behaviour Intervention Plan → the BIP tracks goals and collects teacher feedback on strategy effectiveness.

This module is the first to involve sensitive student conduct data. Row-level visibility is strict: teachers see incidents they reported or that involve students in their classes. Parents see their own children's incidents (non-confidential only). Students never see discipline records directly. Admins and counsellors see all. BIPs follow the same scoping.

Vertical Slice Deliverable

Teacher reports an incident: "Maya was involved in a verbal altercation in the hallway" under category "Disrespect" (MEDIUM severity) → admin reviews the incident, transitions to UNDER_REVIEW, assigns a consequence (Detention, 1 day) with requires_parent_notification=true → parent (David Chen) receives an IN_APP notification → admin resolves the incident → because Maya now has 2+ incidents this semester, the counsellor creates a BIP with target behaviours, replacement strategies, and 3 measurable goals → teacher receives a feedback request on BIP strategy effectiveness → teacher submits feedback → counsellor reviews and updates goal progress.

What's In / What's Deferred

In scope (M20 Discipline): sis_discipline_categories (severity-levelled catalogue), sis_discipline_action_types (with parent-notification flag), sis_discipline_incidents (OPEN → UNDER_REVIEW → RESOLVED lifecycle), sis_discipline_actions (per-incident consequences with date ranges for suspensions). Functions BEH-001.

In scope (M27 Behaviour Plans): svc_behavior_plans (BIP/BSP/SAFETY_PLAN with DRAFT → ACTIVE → REVIEW → EXPIRED lifecycle, target/replacement behaviours, reinforcement strategies), svc_behavior_plan_goals (measurable goals with progress tracking), svc_bip_teacher_feedback (structured teacher input on strategy effectiveness). Functions BEH-002.

Deferred: PBIS points/rewards system (not in ERD v11 — future enhancement). Discipline hearing/appeal workflow (would use Cycle 7 approval engine — Phase 3). Automated "repeat offender" BIP trigger (manual counsellor action this cycle). sis_student_notes integration (the table exists from Cycle 1 but the is_shared_with_counselor cross-module read is Cycle 11 scope). Mandatory reporting integration (COU-006, Cycle 11).

Integration with Wave 1

Task Worker (Cycle 7): New auto-task rule beh.incident.reported creates an ADMINISTRATIVE task on school admins: "Review incident: {student_name} — {category_name}". A second rule beh.bip.feedback_requested creates a task on the target teacher.

Notifications (Cycle 3): Parent notification on actions with requires_parent_notification=true. Admin notification on incident submission.

HR Employees (Cycle 4): reported_by and BIP created_by reference hr_employees. Teacher feedback links through hr_employees.

Students (Cycle 1): All incident and BIP tables FK to sis_students. Row-scope uses the same sis_student_guardians + sis_class_teachers patterns from Cycles 1–2.

Step-by-Step Build Plan

01 Discipline Schema — Categories, Incidents, Actions
Schema

The 4 M20 discipline tables that were deferred from Cycle 1. These are sis_ prefixed and live in the tenant schema alongside the existing SIS tables.

Tables (4)
  1. sis_discipline_categories — school_id UUID, name TEXT NOT NULL, severity ENUM(LOW, MEDIUM, HIGH, CRITICAL), description TEXT, is_active BOOLEAN DEFAULT true. UNIQUE(school_id, name). Examples: Tardiness (LOW), Disrespect (MEDIUM), Fighting (HIGH), Weapons (CRITICAL).
  2. sis_discipline_action_types — school_id UUID, name TEXT NOT NULL, requires_parent_notification BOOLEAN DEFAULT false, is_active BOOLEAN DEFAULT true, description TEXT. UNIQUE(school_id, name). Examples: Verbal Warning (no notification), Detention (notify), In-School Suspension (notify), Out-of-School Suspension (notify), Expulsion Referral (notify).
  3. sis_discipline_incidents — school_id UUID, student_id UUID FK(sis_students), reported_by UUID FK(hr_employees), category_id UUID FK(sis_discipline_categories), description TEXT NOT NULL, incident_date DATE NOT NULL, incident_time TIME nullable, location TEXT nullable, witnesses TEXT nullable, status ENUM(OPEN, UNDER_REVIEW, RESOLVED) DEFAULT 'OPEN', resolved_by UUID nullable (soft to hr_employees), resolved_at TIMESTAMPTZ nullable, admin_notes TEXT nullable (internal, not visible to parents). Multi-column resolved_chk keeps resolved_by and resolved_at in lockstep — both set or both null, and RESOLVED requires both set. INDEX(student_id, incident_date DESC). INDEX(school_id, status) WHERE status != 'RESOLVED'.
  4. sis_discipline_actions — incident_id UUID FK(sis_discipline_incidents) ON DELETE CASCADE, action_type_id UUID FK(sis_discipline_action_types), assigned_by UUID (soft to hr_employees), start_date DATE nullable, end_date DATE nullable (for suspensions — multi-day range), notes TEXT nullable, parent_notified BOOLEAN DEFAULT false, parent_notified_at TIMESTAMPTZ nullable. UNIQUE(incident_id, action_type_id) — one action per type per incident. Multi-column dates_chk: when both set, end_date >= start_date.
Produces → 4 new tables. Migration: 030_sis_discipline.sql.
02 Behaviour Plan Schema — BIPs, Goals, Feedback
Schema

The M27 behaviour plan tables. These use the svc_ prefix — the first Student Services tables in the system. The svc_caseloads FK on svc_behavior_plans is nullable since caseloads ship in Cycle 11; the column is forward-compatible.

Tables (3)
  1. svc_behavior_plans — school_id UUID, student_id UUID FK(sis_students), caseload_id UUID nullable (soft to future svc_caseloads), plan_type ENUM(BIP, BSP, SAFETY_PLAN), status ENUM(DRAFT, ACTIVE, REVIEW, EXPIRED) DEFAULT 'DRAFT', created_by UUID FK(hr_employees), review_date DATE NOT NULL, review_meeting_id UUID nullable (soft to future mtg_meetings), target_behaviors TEXT[] NOT NULL, replacement_behaviors TEXT[], reinforcement_strategies TEXT[], plan_document_s3_key TEXT nullable, source_incident_id UUID nullable (soft to sis_discipline_incidents). Partial UNIQUE(student_id, plan_type) WHERE status = 'ACTIVE' — one active plan per type per student. INDEX(student_id, status).
  2. svc_behavior_plan_goals — plan_id UUID FK(svc_behavior_plans) ON DELETE CASCADE, goal_text TEXT NOT NULL, baseline_frequency TEXT, target_frequency TEXT, measurement_method TEXT, progress ENUM(NOT_STARTED, IN_PROGRESS, MET, NOT_MET) DEFAULT 'NOT_STARTED', last_assessed_at DATE nullable. INDEX(plan_id).
  3. svc_bip_teacher_feedback — plan_id UUID FK(svc_behavior_plans) ON DELETE CASCADE, teacher_id UUID FK(hr_employees), requested_by UUID FK(hr_employees) (the counsellor), requested_at TIMESTAMPTZ NOT NULL, submitted_at TIMESTAMPTZ nullable, strategies_observed TEXT[] nullable, overall_effectiveness ENUM(NOT_EFFECTIVE, SOMEWHAT_EFFECTIVE, EFFECTIVE, VERY_EFFECTIVE) nullable, classroom_observations TEXT nullable, recommended_adjustments TEXT nullable. Partial UNIQUE(plan_id, teacher_id) WHERE submitted_at IS NULL — one pending request per teacher per plan. INDEX(plan_id, submitted_at).
Produces → 3 new tables. Migration: 031_svc_behavior_plans.sql. Total Cycle 9 tables: 7.

04 Discipline NestJS Module — Incidents + Actions
Code
Services & Endpoints
  1. IncidentService — POST /discipline/incidents (teacher/staff reports; stamps reported_by from actor.employeeId; emits beh.incident.reported). GET /discipline/incidents (row-scoped: teacher sees incidents they reported + incidents for students in their classes; admin sees all; parent sees own children's; filters: status, category, student, dateRange, severity). GET /discipline/incidents/:id (row-scope 404). PATCH /discipline/incidents/:id/review (admin: OPEN → UNDER_REVIEW; locks row FOR UPDATE). PATCH /discipline/incidents/:id/resolve (admin: sets resolved_by + resolved_at; emits beh.incident.resolved). PATCH /discipline/incidents/:id/reopen (admin: RESOLVED → OPEN; clears resolved fields).
  2. ActionService — POST /discipline/incidents/:id/actions (admin assigns a consequence; if action_type has requires_parent_notification=true, emits beh.action.parent_notification_required with guardian account IDs resolved from sis_student_guardians). GET /discipline/incidents/:id/actions. PATCH /discipline/actions/:id (update dates, notes, mark parent_notified). DELETE /discipline/actions/:id (admin removes a consequence).
  3. CategoryService — GET /discipline/categories (tree with action types). POST /discipline/categories (admin). PATCH /discipline/categories/:id (admin). GET /discipline/action-types. POST /discipline/action-types (admin). PATCH /discipline/action-types/:id (admin).
Row-Level Visibility

Teachers: see incidents they reported OR incidents for students enrolled in their classes (via sis_class_teachers + sis_enrollments). Parents: see their own children's incidents (via sis_student_guardians), excluding admin_notes. Students: no direct access to discipline records. Admins/Counsellors: see all incidents in the tenant.

Produces → DisciplineModule with 3 services, ~14 endpoints + 3 Kafka events.
05 Behaviour Plan NestJS Module — BIPs + Goals + Feedback
Code
Services & Endpoints
  1. BehaviorPlanService — GET /behavior-plans (by student, by status; row-scope: teacher sees plans for students in their classes; counsellor/admin sees all). GET /behavior-plans/:id (with goals + feedback inline). POST /behavior-plans (counsellor/admin creates; validates partial UNIQUE on active plan per type; source_incident_id validated if supplied). PATCH /behavior-plans/:id (update status, strategies, review_date; status transitions use SELECT FOR UPDATE). PATCH /behavior-plans/:id/activate (DRAFT → ACTIVE). PATCH /behavior-plans/:id/expire (ACTIVE/REVIEW → EXPIRED).
  2. GoalService — GET /behavior-plans/:id/goals. POST /behavior-plans/:id/goals (add a goal). PATCH /behavior-plan-goals/:id (update progress, last_assessed_at). DELETE /behavior-plan-goals/:id.
  3. FeedbackService — POST /behavior-plans/:id/feedback-requests (counsellor requests feedback from a specific teacher; emits beh.bip.feedback_requested). GET /behavior-plans/:id/feedback (list all feedback for a plan). PATCH /bip-feedback/:id (teacher submits their feedback: strategies_observed, overall_effectiveness, observations, adjustments; sets submitted_at). GET /bip-feedback/pending (teacher's pending feedback requests).
Produces → BehaviourPlanModule with 3 services, ~12 endpoints. Total behaviour endpoints: ~26.
06 Behaviour Notification Consumer
Code
Components
  1. BehaviourNotificationConsumer — Kafka consumer on beh.incident.reported, beh.incident.resolved, beh.action.parent_notification_required, beh.bip.feedback_requested. On incident.reported: notifies school admins (same pattern as attendance/absence). On action.parent_notification_required: resolves guardian accounts via sis_student_guardians (portal_access=true) and enqueues "Your child has received a disciplinary action" notification. On bip.feedback_requested: notifies the target teacher.
  2. Auto-task integration: The 2 auto-task rules from Step 3 feed the existing Task Worker. Admin gets "Review incident" tasks. Teachers get "BIP feedback requested" tasks.
Produces → 1 Kafka consumer + auto-task rule wiring.
07 Discipline UI — Report Incident + Admin Queue
UI
Pages
  1. Report Incident /behaviour/report — student picker (searchable, filtered to caller's students for teachers), category dropdown, date + optional time, location (free text or room picker), description (required), optional witnesses field. Submit creates the incident and navigates to the detail page.
  2. Incident Queue /behaviour — "Behaviour" app tile on the home page for anyone with beh-001:read. List with filter chips: Open / Under Review / Resolved / All. Per-row: student name, category + severity pill (LOW gray / MEDIUM amber / HIGH orange / CRITICAL red), date, reported by, status pill, action count. Admin-only bulk actions: review, resolve.
  3. Incident Detail /behaviour/:id — header with student name, category, severity, status, reported by, date/time/location. Description panel. Admin: Review / Resolve / Reopen buttons. Actions section: list of assigned consequences with action type, dates, parent notification status. "Add Action" button (admin) opens a modal with action type dropdown, optional date range (for suspensions), notes. Admin notes panel (internal, not visible to parents). Activity timeline at bottom.
  4. Category Management /behaviour/admin/categories — admin-only. Categories list with severity pills + active toggle. Action types list with parent-notification toggle. Add/edit modals.
Produces → "Behaviour" app tile, incident report, admin queue, incident detail, category management.
08 Behaviour Plan UI — BIP Editor + Goals + Feedback
UI
Pages
  1. Student Behaviour Summary /students/:id/behaviour — accessible from the student profile (admin/teacher/counsellor). Incident count by severity. Recent incidents list. Active BIPs section. "Create BIP" button for counsellors/admins. Links to the BIP detail page.
  2. BIP Editor /behavior-plans/:id — header with student name, plan type pill, status pill, review date. Editable sections: target behaviours (text array editor), replacement behaviours, reinforcement strategies. Goals table: goal text, baseline, target, measurement method, progress pill, last assessed date, inline edit. "Add Goal" row. Status transition buttons: Activate (DRAFT → ACTIVE), Mark for Review (ACTIVE → REVIEW), Expire. Source incident link (if set). Feedback section: list of teacher feedback with effectiveness pill, "Request Feedback" button for counsellors.
  3. Pending Feedback /behavior-plans/feedback — teacher's view of their pending feedback requests. Each row: student name, plan type, requested by, requested date. Click opens a feedback form: strategies observed (checklist from the plan's reinforcement_strategies), effectiveness rating, classroom observations (text), recommended adjustments (text). Submit sets submitted_at.
Produces → Student behaviour summary, BIP editor with goals + feedback, teacher feedback view.

Cycle 9 Completion Criteria

Exit Checklist
  1. Tenant schema: 7 new tables (4 discipline + 3 behaviour plans). Tenant table count: ~138.
  2. Discipline API: ~14 endpoints. Report, review, assign actions, resolve. Parent notification on flagged actions.
  3. Behaviour Plan API: ~12 endpoints. Create/edit BIPs, track goals, request and submit teacher feedback.
  4. Notifications: incident reported → admin, action with parent flag → guardian, feedback request → teacher.
  5. Auto-task rules: incident.reported → admin review task, feedback.requested → teacher task.
  6. Row-level visibility: teacher scope (reported + class students), parent scope (own children, no admin_notes), admin/counsellor scope (all).
  7. Staff UI: report incident, admin queue, incident detail with actions, category/action-type management.
  8. BIP UI: editor with target/replacement behaviours, goal progress tracker, feedback request + submission.
  9. Parent UI: child's incident history + BIP summary (read-only).
  10. HANDOFF-CYCLE9.md and CLAUDE.md updated. CI green.

After Cycle 9, CampusOS tracks the full behaviour lifecycle from informal student notes (Cycle 1) through formal discipline incidents with consequences and parent notification, to structured Behaviour Intervention Plans with measurable goals and teacher feedback. The soft link from BIPs to discipline incidents creates the audit trail; the soft link from BIPs to future counselling caseloads (Cycle 11) is forward-compatible via the nullable caseload_id column.
